**Quick-command scope.** The five modules above are exactly this phase's blast radius: Plan 01 edits
`typsphinx/pdf.py` (→ `test_pdf_generation.py`) and `typsphinx/builder.py` (→ `test_builder.py`,

`test_preview_version_sync.py` is the standing milestone invariant that must never go red.
**Once Plan 02 Task 2 lands, append `tests/test_nested_master_render_gate.py`** to the quick command
(measured cost of a comparable render-gate module: ~0.6 s for 2 gates; a single real
`-b typstpdf` build of a small fixture measures ~0.35 s, so the quick command stays under ~2.5 s).

**Worktree note:** executors run in isolated git worktrees (standing mode). Provision first with
`env -u VIRTUAL_ENV -u UV_PROJECT_ENVIRONMENT uv sync --extra dev`, then run every command via `uv run`.
The one exception is *inside* test code: Sphinx builds are spawned as
`subprocess.run([sys.executable, "-m", "sphinx", ...])`, never `uv run sphinx-build` and never a
resolved `sphinx-build` binary (NixOS sandbox PATH-shadowing hazard).

**GATE-01 (standing milestone bar):** every fix ships or extends a real `typst.compile()` regression
fixture. String-agreement asserts alone never suffice. Plan 02 discharges this for PDF-02.

**typst-py availability:** `import typst` succeeds in this environment (typst-py at
`.venv/lib/python3.13/site-packages/typst/`), so Plan 02's `@pytest.mark.skipif(not TYPST_AVAILABLE)`
gates run rather than skip. A skipped render gate is **not** a pass — if the gate reports `skipped`,
treat the phase as unverified.

- **Before `/gsd-verify-work`:** Full fast suite green, **plus** one run of `uv run pytest -m slow -q`
  (the full-corpus gate). Plan 01's D-04 change makes the corpus gate materially more likely to
  surface a real failure than before, so it must be run at least once before phase close.
- **Max feedback latency:** **28 s** (full fast suite). Per-task latency is ~1.4 s.
