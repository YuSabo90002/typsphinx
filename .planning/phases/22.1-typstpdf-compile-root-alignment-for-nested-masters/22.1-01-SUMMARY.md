# Phase 22.1 Plan 01: pdf.py File-Path Compile Entry Point + finish() Attempt-All-Then-Raise Summary

**New `compile_typst_file_to_pdf` compiles a master's own on-disk `.typ` directly (no temp file), and `TypstPDFBuilder.finish()` now compiles that real path and fails loudly (aggregated `ExtensionError`) instead of silently swallowing a compile failure.**

## Performance

- **Duration:** ~25 min
- **Started:** 2026-07-21 (session start)
- **Completed:** 2026-07-21T15:35:51Z
- **Tasks:** 3
- **Files modified:** 4 (`typsphinx/pdf.py`, `typsphinx/builder.py`, `tests/test_pdf_generation.py`, `tests/test_corpus_gate.py`)

## Accomplishments
- Added `typsphinx.pdf.compile_typst_file_to_pdf(typ_path, root_dir=None) -> bytes`, which calls `typst.compile(typ_path, root=root_dir)` directly against an already-on-disk file — no temporary file is created or deleted, and `TypstCompilationError.source_location` now points at a real, actionable file path.
- `compile_typst_to_pdf` keeps byte-identical observable behavior (same temp-file-at-`root_dir` write, same `finally: os.unlink` cleanup) but now delegates its compile step to `compile_typst_file_to_pdf`; its docstring carries a `.. deprecated::` notice (removal announced for v0.7, no runtime warning — `pyproject.toml` escalates `DeprecationWarning` to a hard test failure).
- `TypstPDFBuilder.finish()` now compiles each master's own `.typ` at its real docname-derived location (`outdir/api/index.typ`, not a file at the outdir root) via `compile_typst_file_to_pdf(typ_file, root_dir=self.outdir)` — the docname-relative `#include()`/`image()` paths the translator emits now resolve correctly by construction for nested masters.
- Replaced `finish()`'s `except Exception: logger.error(...)` swallow with attempt-all-then-raise: every configured master is attempted, masters that compile still get their `.pdf` written, and one aggregated `sphinx.errors.ExtensionError` naming every failed docname + its error is raised after the loop — a build can no longer report success (exit code 0) while a master's PDF is silently absent.
- Repaired the two stale `tests/test_pdf_generation.py` cases that encoded the pre-fix internals (Pitfall 1 from `22.1-RESEARCH.md`): the mock-target patch and the swallow-and-continue assertion; added a new mocked unit test pinning the path-resolution basis (D-08b), runnable without `typst-py` installed.
- Corrected a now-inaccurate `tests/test_corpus_gate.py` comment (finish() previously "logged rather than raised"); only the prose changed — assertions, control flow, and collected test ids are unchanged.

## Decisions Made
- **D-01/D-02 (locked by CONTEXT.md):** `compile_typst_file_to_pdf` is the new primary compile entry point; `compile_typst_to_pdf` stays byte-identical and delegates, with a docstring-only deprecation (no runtime warning — the module's `filterwarnings` config escalates `DeprecationWarning`/`PendingDeprecationWarning` to hard errors across the whole suite).
- **D-04/D-05 (locked by CONTEXT.md):** `finish()` attempts every master, keeps writing `.pdf`s for masters that succeed, and raises a single `ExtensionError` after the loop naming every failure — chosen over `logger.warning` (invisible without `-W`) and fail-fast-on-first-error (would hide the second/third failure from the user).
- **Exception type discretion (RESEARCH.md recommendation, adopted):** used `sphinx.errors.ExtensionError` rather than reusing `TypstCompilationError` (shaped for one failure) or inventing a new exception class — `ExtensionError` is what `sphinx-build`'s own CLI handler already knows how to print cleanly and turn into exit code 2.
