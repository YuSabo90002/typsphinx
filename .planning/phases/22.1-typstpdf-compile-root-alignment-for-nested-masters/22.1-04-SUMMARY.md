# Phase 22.1 Plan 04: G-22.1-4 Gap Closure -- Depth-Based Template-Import Repair Summary

**Replaced `TypstWriter.translate()`'s sentinel-based `_template.typ` import computation with a depth-based `TypstWriter._compute_template_import_path(docname)` staticmethod, pinned by a 7-case parametrized unit matrix and a real-compile render gate for a master whose directory is literally named `_template`.**

## Performance

- **Duration:** ~25 min
- **Started:** 2026-07-22 (session start, base commit `c6d9b30`)
- **Completed:** 2026-07-22
- **Tasks:** 2
- **Files modified:** 5 (1 modified, 4 created)

## Accomplishments

- Added `TypstWriter._compute_template_import_path(docname)` as a `@staticmethod`: computes the number of upward `../` segments from `len(PurePosixPath(docname).parent.parts)` and appends `_template.typ` -- a pure function of the master's own nesting depth to the outdir root, with zero string-equality dependence on the reserved `_template` basename.
- Rewrote `translate()`'s `template_file = ...` expression to call the new staticmethod directly, removing the prior call that routed the literal string `"_template"` as a fake target docname through `self.visitor._compute_relative_include_path(...)` -- the exact mechanism that let a real `_template`-named directory collide with the sentinel (gap `G-22.1-4`, review finding CR-01).
- Rewrote the explanatory comment above the call site to state the CR-01 root cause and why the depth basis is immune to it by construction, replacing the old comment's justification for reusing the translator's relativizer.
- `tests/test_template_import_path.py` (new module): `TestComputeTemplateImportPath` pins all 7 cases from the plan's `<behavior>` table in one parametrized test with `ids=` labelling each row `fence-*` (already correct pre-fix, anti-regression) or `repaired-*` (previously malformed), plus an independent `test_fence_rows_match_depth_invariant` that re-derives the expected depth from each fence docname's own path structure rather than re-reading the same table.
- `TestTemplateNamedDirMasterRenderGate` (added in Task 2): drives one real `-b typst` build of the new `tests/fixtures/template_named_dir_master/` fixture (masters at `_template/index` depth 1 and `_template/sub/index` depth 2), asserts the outdir-root `_template.typ` FILE / `_template/` DIRECTORY coexistence, asserts both emitted masters carry the depth-correct `#import` line (and, structurally, that neither's import target's final path component is malformed), then compiles both masters for real via `typst.compile(master, root=outdir)` and asserts valid `%PDF` bytes at both depths.
- Verified manually with a real `sphinx-build -b typst` + `typst.compile()` run against the new fixture before wiring the assertions: depth-1 emits `#import "../_template.typ"`, depth-2 emits `#import "../../_template.typ"`, and both compiled to `%PDF-1.7` bytes -- confirming the fix and the fixture shape empirically, not just via the plan's anticipated forms.

## Decisions Made

- Implemented the depth computation via `PurePosixPath(docname).parent.parts` length rather than any string-splitting on `_template`, per the plan's explicit prohibition against reintroducing a string-equality dependence.
- Kept Task 1 and Task 2 as genuinely separate commits matching the plan's task boundaries: Task 1's test module has no `typst` import and no fixture dependency (runs in milliseconds, stays green without the compiler installed); Task 2 adds the guarded `try: import typst` block, the fixture, and the render gate class in the same module.
- `typst_documents` target names intentionally equal each docname's own basename (`"index"`), mirroring the D-07 reasoning already established in the sibling `nested_master_render_gate` fixture from Plan 22.1-02/03, so this gate isolates G-22.1-4 from the unrelated Phase 22 target-name-rename feature.
