## Summary

Reviewed the PDF-02 compile-root alignment fix (`typsphinx/pdf.py::compile_typst_file_to_pdf`, `typsphinx/builder.py::TypstPDFBuilder.finish`) and its companion `TypstWriter.translate()` fix for the nested-master `_template.typ` import path, plus the new/changed test coverage in `tests/test_nested_master_render_gate.py`, `tests/test_pdf_generation.py`, and `tests/test_corpus_gate.py` (comment-only change there).

The core `compile_typst_file_to_pdf` / `TypstPDFBuilder.finish()` change is correct and well-tested: I ran the full `test_nested_master_render_gate.py` + `test_pdf_generation.py` suites against a real `typst-py` install (30/30 passed) and confirmed `ruff`/`mypy` are clean on the changed files. The new gate tests are non-vacuous — they build the fixture for real, transcribe the actual Typst error text, and assert on the specific failure class each half claims to isolate (verified by direct re-execution, not just reading).

However, the writer.py companion fix (reusing `TypstTranslator._compute_relative_include_path("_template", docname)` to compute the template import's relative path) has a genuine, reproducible correctness bug: when a master document's docname directory tree contains a path segment that is *literally* `_template` (the same string used as the synthetic "docname" for this call), the relative-path algorithm treats it as a shared path component instead of the reserved template basename, producing a malformed import (`"..typ"` / `"../.typ"`) instead of a path back to the real `outdir/_template.typ`. I reproduced this by direct execution against `typsphinx.translator.TypstTranslator._compute_relative_include_path`. None of the phase's new tests exercise this path, since the fixture's master lives at `api/index`, not under a `_template/` directory.

I also found one overstated guarantee in `TypstPDFBuilder.finish()`'s new docstring/behavior (D-04) and one test-robustness concern in the new SC#2 tests (pinned to non-contractual `typst-py` error-message substrings). See below.

## Critical Issues

### CR-01: Template-import path collision when a master's docname contains a literal `_template` path segment

**File:** `typsphinx/writer.py:163-165` (and `typsphinx/translator.py:2928-3038`, `_compute_relative_include_path`)

**Issue:** The companion fix computes the master's `_template.typ` import path by calling
`self.visitor._compute_relative_include_path("_template", docname)` — i.e. it repurposes the
translator's real docname-to-docname relative-path calculator, passing the literal string
`"_template"` as a *fake* `target_docname`. That function's "same directory" fast path calls
`PurePosixPath(target).relative_to(current_dir)`; if the master's own immediate parent directory
component is *also* literally `_template` (e.g. a master at docname `_template/index`, or nested
`_template/sub/index`), `relative_to()` treats the fake target as literally naming the current
directory, and returns `"."` (or `"../"` for one nesting level deeper) instead of climbing to and
naming the real `outdir/_template.typ` file. Concatenated with `.typ` in `writer.py:165`, this
produces `template_file = "..typ"` (for `_template/index`) or `"../.typ"` (for `_template/sub/index`)
— references to files that do not exist, so the emitted master fails to compile with a "file not
found" (or similarly nonsensical) Typst error.

Verified directly (not just by inspection):

```
$ uv run python3 -c "
from typsphinx.translator import TypstTranslator
class Dummy: pass
fn = TypstTranslator._compute_relative_include_path
print(fn(Dummy(), '_template', '_template/index'))       # '.'      -> template_file = '..typ'
print(fn(Dummy(), '_template', '_template/sub/index'))   # '../'    -> template_file = '../.typ'
"
```

This is a real, plausible collision: `_template` (singular) is exactly the reserved basename
`_write_template_file()` (`typsphinx/builder.py:553`) always writes at the outdir root, so a project
author who names a source subdirectory `_template` (e.g. to hold custom Typst partials, mirroring
the tool's own naming) and places a master document inside it will hit this — a scenario none of
the phase's new fixtures or tests exercise (the fixture's master sits at `api/index`, never under
`_template/`).

**Fix:** Do not reuse a docname-vs-docname relative-path function with a synthetic sentinel string
that can coincide with a real path component. Compute the template import path structurally, by
depth alone, with no string-equality dependence on the literal reserved name:

```python
# writer.py
from pathlib import PurePosixPath

depth = len(PurePosixPath(docname).parent.parts)
template_file = ("../" * depth) + "_template.typ" if depth else "_template.typ"
```

This produces the same output as the current fix for every docname that does *not* collide with
`_template` (root-level masters get `"_template.typ"`; `api/index` gets `"../_template.typ"`), but
never misinterprets a `_template`-named directory component as the sentinel target. Add a regression
test with a master fixture at docname `_template/index` (or `some/_template/index`) asserting the
emitted import is the correct `../_template.typ` (or equivalent depth-correct path), not a malformed
one.

## Warnings

### WR-01: D-04's "no silent success" guarantee is incomplete — a missing `.typ` file still bypasses failure tracking

**File:** `typsphinx/builder.py:840-859, 895-897`

**Issue:** The new `finish()` docstring states: "a build can no longer 'succeed' while silently
producing no PDF for a broken master," backed by the new `failures` list + `ExtensionError` raised
after the loop. However, the pre-existing branch a few lines above the new failure-tracking logic —
`if not path.exists(typ_file): logger.warning(...); continue` — does **not** append to `failures`.
If a configured master's `.typ` was never written (e.g. a docname/target mismatch, or `write_doc`
silently skipped it for any reason), `finish()` logs a warning and moves on; the loop completes with
an empty `failures` list, no `ExtensionError` is raised, and `sphinx-build` exits 0 with that master's
PDF simply absent — exactly the "silent success" scenario the phase's new docstring/behavior claims
to have eliminated. This branch itself predates Phase 22.1, but the phase's new copy explicitly
broadens the completeness claim without closing this gap, so the documented guarantee is now
overstated relative to actual behavior.

**Fix:** Route the missing-file case through the same `failures` list so it participates in the D-04
guarantee:

```python
if not path.exists(typ_file):
    logger.warning(f"Master document not found: {typ_file}")
    failures.append((docname, f"expected .typ file not found: {typ_file}"))
    continue
```

### WR-02: SC#2 pre-fix-basis tests pin to non-contractual `typst-py` error-message substrings

**File:** `tests/test_nested_master_render_gate.py:365-381, 502-525`

**Issue:** `test_outdir_root_compile_basis_still_fails` asserts `"escape" in lowered` and
`"_template.typ" in error_text`; `test_sibling_include_fails_at_outdir_root_and_resolves_in_place`
asserts `"usage.typ" in error_text` and `"not found" in lowered`. Both assertions depend on the exact
wording of `typst-py`'s underlying Rust-compiler error strings, which are not a documented, versioned
contract of that dependency. The module docstring itself acknowledges this ("Typst's exact wording is
not a contract, but the failure CLASS is"), which is the right instinct, but the implementation still
asserts on the literal wording rather than a more structural signal (e.g. exception type, or a
looser/broader keyword set with a documented fallback). A `typst-py` version bump that merely rewords
its error message (without changing the underlying failure semantics) would turn these two tests red
for a reason unrelated to any regression in typsphinx itself, which is exactly the kind of brittleness
the `drift.yml` weekly re-resolution workflow (per CLAUDE.md) is likely to surface as noisy failures.

**Fix:** Consider widening the acceptable keyword set per assertion (e.g. `any(kw in lowered for kw in
("escape", "outside", "root"))`) or, if precise wording must be pinned for now, add a code comment
noting which `typst` package version these strings were captured against, so a future drift-triggered
failure is quickly diagnosable as "wording changed, not behavior."

### IN-02: Redundant `check_typst_available()` call on the `compile_typst_to_pdf` → `compile_typst_file_to_pdf` delegation path

**File:** `typsphinx/pdf.py:182, 138`

**Issue:** `compile_typst_to_pdf()` calls `check_typst_available()` at line 182, then (after writing the temp file) delegates to `compile_typst_file_to_pdf()`, which calls `check_typst_available()` again at line 138. Harmless (idempotent, cheap), but it's dead redundancy introduced by the refactor.

**Fix:** Drop the now-redundant call in `compile_typst_to_pdf()` since `compile_typst_file_to_pdf()` already performs the check before compiling.
