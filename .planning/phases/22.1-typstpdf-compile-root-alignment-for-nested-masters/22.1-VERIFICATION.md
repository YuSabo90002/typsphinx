### Scrutiny: Unplanned `typsphinx/writer.py` change

1. **Genuine bug of the same class, not scope creep.** `TypstWriter.translate()` hardcoded
   `template_file="_template.typ"` for every master. `_write_template_file()` always writes
   `_template.typ` at the outdir root. For a root-level master this is correct by coincidence (same
   directory); for any nested master it is wrong — the identical docname-relative-vs-outdir-root
   mismatch PDF-02 targets, just at a different call site (the template import, not `#include()`/
   `image()`). Without this fix, no nested master could compile at all — this gate (and the phase goal)
   could not be proven. Confirmed root cause, confirmed fix mechanism (reuses
   `_compute_relative_include_path()` verbatim, no new path-derivation logic), confirmed via reading the
   diff (`git show d911f12 -- typsphinx/writer.py`).
2. **Root-level masters (`index`) are provably unaffected.** Traced `_compute_relative_include_path`
   directly: for `current_docname` whose `PurePosixPath(...).parent == PurePosixPath(".")` (any
   root-level docname), the function returns `target_docname` unchanged at line 2988 — so
   `_compute_relative_include_path("_template", "index")` returns `"_template"`, and `writer.py`'s
   `+ ".typ"` produces exactly `"_template.typ"`, byte-identical to the pre-fix hardcoded string. This
   is a structural (not merely test-observed) guarantee, confirmed by reading the branch logic itself,
   not just by the passing regression suite.
3. **Does NOT violate the tested milestone invariant.** `tests/test_preview_version_sync.py` scans for
   `#import "@preview/<name>:<version>"` lines specifically. The `writer.py` diff (`git show d911f12`)
   touches only lines ~150-166 (the master-document template-file computation inside `translate()`);
   the `@preview` import declarations for *included* (non-master) documents at lines 92-100 are
   untouched. Confirmed by reading both the diff and the surrounding unchanged code. The version-sync
   test passes (2/2, both orchestrator-observed and independently re-confirmed in this session).
4. **Does contradict CONTEXT.md's stated scope and Plan 02's own verification gate.** CONTEXT.md states
   "All edits land in typsphinx/pdf.py + typsphinx/builder.py; no translator.py change" — `writer.py` is
   not `translator.py`, but it is also not `pdf.py`/`builder.py`, so the edit falls outside the locked
   scope statement. Plan 02's own `<verification>` step 6 lists `typsphinx/writer.py` among files
   expected to have **zero** diff. This is a genuine, honestly-documented Rule-1 deviation (both
   SUMMARYs disclose it prominently with rationale, root cause, and verification), not a concealed one —
   but it is still a deviation from a locked decision and an explicit plan gate, so it is routed to
   human_verification rather than silently accepted.

### Scrutiny: SC#2 error-class shift

Reasoned through directly from the code (no working-tree mutation performed):

- **If Wave 1's `pdf.py`/`builder.py` fix (D-01/D-04/D-05) were reverted** (restoring
  `except Exception: logger.error(...)` and the `NamedTemporaryFile(dir=root_dir)` compile), the gate
  would still go RED, with HIGH confidence — but via `test_typstpdf_nested_master_resolves_include_and_image`'s
  `pdf_output.exists()` assertion, not its `returncode == 0` assertion. Reasoning: `write_doc()` is
  untouched by any Wave-1 revert, so `outdir/api/index.typ` is still correctly written with its
  `#import "../_template.typ"` / `include("usage.typ")` / `image("../logo.png")` references intact. Old
  `finish()` would read that file into a string, write a temp copy at the outdir ROOT, and compile with
  `root=outdir` — the *first* reference (`../_template.typ`) would already need to resolve one directory
  above `outdir`, triggering Typst's root-escape rejection immediately. The old
  `except Exception: logger.error(...)` swallow catches this, so `sphinx-build`'s own exit code stays 0
  (the pre-fix hole this phase's D-04 specifically closes) — meaning `returncode == 0` would pass
  *spuriously* on a reverted build, but `pdf_output.exists()` would correctly fail because no `.pdf` was
  ever written. **The gate is not vacuous with respect to a Wave-1 revert; the artifact-existence
  assertion is the actual regression detector, not the exit code.**
- **`test_outdir_root_compile_basis_still_fails` (SC#2) does not itself exercise `finish()`/`pdf.py` at
  all** — it manually reproduces "a copy at the outdir root, compiled with `root=outdir`" using
  `typst.compile()` directly. Its pass/fail outcome is therefore independent of whether Wave 1's fix is
  present or reverted; it is a standing *documentary* proof (per D-08(a)'s intent: "so the gate can
  never start passing vacuously"), not a regression detector for the production code path. That
  detector role is fulfilled by test 1's artifact-existence assertion, confirmed above.
- **The `"not found" in lowered or "escape" in lowered` tolerance is not, on its own, dangerously loose**
  — the test also asserts `"_template.typ" in error_text`, which narrows it to the specific reference
  whose resolution differs by compile-root depth. However, the test **only ever proves the
  template-import call site breaks pre-fix** — it does not independently prove, in isolation, that the
  sibling `include("usage.typ")` / image `"../logo.png"` references (the literal PDF-02 canonical
  failure mode) would also break if compiled from the outdir root. That fact is provable in principle
  (the same root-cause mechanism applies to all three references) but is not directly exercised by any
  standing test in this module — it is asserted only implicitly, by the success path (test 1) proving
  they resolve correctly *after* the fix. This is the substance of the flagged caveat in
  `human_verification` item #2.
