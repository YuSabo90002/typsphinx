### Code Review Findings (`22.1-REVIEW.md`) — Triage

The code review completed after the prior verification/UAT round surfaced 1 Critical + 2 Warnings
(+ 2 Info, not reproduced here as non-blocking). None of these were part of the `G-22.1-2` gap this
re-verification pass closes, and none were adjudicated by the UAT round that accepted the `writer.py`
scope expansion (that UAT test predates this review). They are new, untriaged findings against
**accepted, shipped** production code from this phase:

- **CR-01 (Critical, `typsphinx/writer.py`/`translator.py`):** a docname literally containing a
  `_template` path segment (e.g. `_template/index`) causes the new template-import path computation to
  mis-resolve to a malformed reference (`"..typ"` / `"../.typ"`), because it repurposes
  `_compute_relative_include_path()` with the sentinel string `"_template"` as a fake target docname,
  which collides with a real path component of that exact name. **Independently reproduced in this
  verification session** (not merely re-stated from the review) — see Behavioral Spot-Checks above.
  This does **not** affect the `api/index` fixture or any ROADMAP success criterion this phase declares
  (none use a docname containing a literal `_template` segment), so it does not fail any must-have
  truth. It is, however, a real, newly-introduced (by this phase's own companion fix) correctness bug
  in shipped code, with no test coverage and no backlog/follow-up tracking as of this verification.
  Routed to `human_verification` — not something a verifier should silently wave through or silently
  block on.
- **WR-01 (Warning, `typsphinx/builder.py`):** the missing-`.typ`-file branch in `finish()` does not
  append to the `failures` list, so a configured master whose `.typ` was never written can still let
  `sphinx-build` exit 0 with that master's PDF silently absent — the exact "silent success" scenario
  D-04's docstring claims to have eliminated is not fully closed for this one pre-existing branch.
  Lower severity: this branch predates Phase 22.1, and the scenario (`write_doc()` silently skipping a
  configured master) is narrow. Routed to `human_verification` as an acceptable-risk confirmation.
- **WR-02 (Warning, `tests/test_nested_master_render_gate.py`):** SC#2 and the new `G-22.1-2` test both
  pin to specific `typst-py` error-message substrings (`"escape"`, `"not found"`) that are not a
  documented, versioned contract of that dependency; a wording-only `typst-py` bump could turn these
  tests red for reasons unrelated to any `typsphinx` regression. The module docstring already
  acknowledges wording isn't a contract but the implementation still pins to literal substrings.
  Lower severity, test-brittleness concern only. Routed to `human_verification` as an
  acceptable-risk confirmation.

**Judgment: none of CR-01/WR-01/WR-02 block the literal phase goal or any declared ROADMAP success
