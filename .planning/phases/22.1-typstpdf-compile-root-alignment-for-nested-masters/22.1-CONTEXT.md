# Phase 22.1: typstpdf Compile-Root Alignment for Nested Masters - Context

**Gathered:** 2026-07-21
**Status:** Ready for planning

<domain>
## Phase Boundary

`-b typstpdf` and `-b typst` resolve `#include()` / `image()` on the **same basis** — the master's
own directory — so a master at a nested docname (`api/index`) compiles to PDF with its includes and
images intact. **No output file moves.**

**Verified root cause (re-measured during discussion, 2026-07-21):**

- `TypstPDFBuilder.finish()` (`builder.py:890-895`) reads the master `.typ` into a **string** and
  passes it to `compile_typst_to_pdf(content, root_dir=self.outdir)`.
- `compile_typst_to_pdf()` (`pdf.py:136-149`) writes that string to a `NamedTemporaryFile` with
  `dir=root_dir` — i.e. **the outdir root** — and compiles *that* file.
- Typst resolves relative paths against the **file being compiled**, so `-b typstpdf` resolves from
  the outdir root.
- The translator emits **docname-relative** paths (`_compute_relative_include_path`,
  `translator.py:2928`; `_compute_relative_image_path`, `translator.py:3039`).

The two bases coincide **only** when the master sits at the outdir root (`index`) — which is why
every existing test and the whole corpus passes. A master at `api/index` emits `include("usage.typ")`
(sibling, relative to `api/`) which the temp copy at the outdir root resolves to `outdir/usage.typ`
→ file not found. **Nested masters are already broken today.**

**Why nobody noticed (measured):** `finish()` wraps the compile in
`except Exception: logger.error(...)` (`builder.py:903-905`). On failure the `.pdf` write at
`builder.py:897-899` is **never reached** — the artifact is silently absent — the loop continues to
the next master, and `sphinx-build` exits **0**. A completely missing deliverable currently produces
a green build. D-04 closes this.

**Out of scope:**
- Aggregating master artifacts so multiple masters don't scatter across the build tree — **dropped
  2026-07-21** (owner decision; would require moving output and re-basing relative paths).
- `typst_output_dir` (registered `__init__.py:60`, documented `docs/configuration.rst:255-269`, read
  nowhere) — tracked in `.planning/todos/pending/2026-07-21-dead-typst-output-dir-config.md`.
- The Phase 22 target-name mapping itself (done); the Phase 23 release prep.

**Milestone invariants carried forward:** zero new runtime dependencies, no `@preview` version bump,
the 3-way version-sync surface (`writer.py` / `template_engine.py` / `templates/base.typ`) untouched.
All edits land in `typsphinx/pdf.py` + `typsphinx/builder.py`; **no `translator.py` change.**

</domain>

<decisions>
## Implementation Decisions

### The fix shape (pdf.py)

- **D-01: Add a file-path entry point and compile the master `.typ` directly — the temporary file disappears entirely.**
  New `compile_typst_file_to_pdf(typ_path, root_dir)` in `pdf.py`;
  `TypstPDFBuilder.finish()` calls it with the master's own `.typ` path (the path it already computes
  via `_resolve_output_stem` + `_directory_preserving_relpath`) and `root_dir=self.outdir` unchanged.
  Because the compiled file *is* the master `.typ` at its real location, the docname-relative paths
  the translator emitted resolve correctly **by construction** — the basis divergence cannot recur.
  Rationale: this removes the root mismatch, the outdir-pollution, the name-collision risk, and the
  interrupted-build leftovers in one move, and Typst error locations start pointing at a real file
  instead of `tmpXXXX.typ`. Rejected: (a) keeping the temp copy but placing it beside the master
  (ROADMAP's stated minimal fix — smaller diff, but leaves a temp file inside `outdir` forever and
  keeps a second, divergable notion of "where the compile happens"); (b) rewriting
  `compile_typst_to_pdf()`'s own signature to take a path (cleanest end state, but a breaking change
  across 18 existing call sites in `tests/test_pdf_generation.py`).

- **D-02: Keep `compile_typst_to_pdf(typst_content, root_dir)` with byte-identical behavior, but delegate internally and mark it deprecated in its docstring.**
  It still writes its temp file to
  `root_dir` exactly as today, then calls the new file-path function — so `typst.compile()`
  invocation and error wrapping exist in exactly one place. The docstring states it is internal /
  legacy, that new code must use the file-path form, and names the limitation explicitly: *because
  the temp file lands at `root_dir`, relative paths in content authored for a master that does not
  sit at `root_dir` will not resolve.* No `DeprecationWarning` is emitted — v0.6.2 is a bug-fix
  release. All 18 existing tests stay untouched.

- **D-03: Announce removal of the content-string form for v0.7.** The Phase 23 `CHANGELOG` `[0.6.2]`
  entry carries the deprecation notice, and a backlog item is filed for the actual removal. This is a
  hand-off obligation, not extra work in this phase.

### Compile-failure behavior

- **D-04: A PDF compile failure now raises and fails the build.** The current
  `except Exception: logger.error(...)` swallow is replaced. Rationale: on failure the `.pdf` is
  never written at all, so today a user gets a **silently missing deliverable and exit code 0** —
  precisely the hole that let the nested-master break survive undetected. This is a deliberate
  behavior change and **more aggressive than Phase 22's D-06 warn-and-continue posture**: a missing
  artifact is not a degradation, it is a non-delivery. Rejected: `logger.warning` (only `-W` users
  would notice, and the corpus/CI that missed this bug does not run `-W` everywhere); leaving it
  unchanged (keeps the detection hole open).
- **D-05: Try every master, then fail once with an aggregated report.** `finish()` iterates all
  `typst_documents` entries, records each failure (master name + Typst error), keeps writing the
  `.pdf`s that *do* compile, and raises a single exception listing every failure after the loop.
  Rationale: a user with three broken masters learns all three from one build instead of three.
  Rejected: fail-fast on the first error.
- **D-05a (hand-off): D-04 is a user-visible breaking change and MUST appear in the Phase 23 CHANGELOG `[0.6.2]` entry** — builds that "succeeded" while producing no PDF will now fail.

### Regression fixture (GATE-01)

- **D-06: A dedicated new render gate**, following the established GATE-01 shape verbatim —
  fixture project under `tests/fixtures/`, test module `tests/test_*_render_gate.py`, built via
  `sys.executable -m sphinx` (**never** `uv run` — NixOS sandbox PATH hazard), asserting the emitted
  artifact plus `%PDF` magic, with the break and the confirmation of both directions written into the
  module docstring. Rejected: extending `tests/fixtures/integration_nested_toctree` (would change the
  premise of the existing nested-toctree tests); adding a new `tests/roots/` root (all real-compile
  gates live under `tests/fixtures/`, splitting the convention).
- **D-07: The fixture isolates the nested-master condition and nothing else.**
  `typst_documents = [('api/index', 'index', …)]` — **target == docname**, so the Phase 22
  target-name rename is deliberately *not* exercised here. `api/index` `#include()`s a sibling
  (`api/usage`) and references an image that lives at the project root (so the emitted `image()` is
  `../<...>` and crosses a directory boundary upward). Rationale: when this gate goes red the cause
  is unambiguously PDF-02. Rejected: combining with a `target != docname` rename (matches ROADMAP
  SC#1's example but confounds two failure modes); declaring both a root master and a nested master
  in one project (SC#4's root-master non-regression is already covered by the entire existing corpus
  and `tests/roots/test-basic`).
- **D-08: Prove the pre-fix failure two ways.** (a) Run the new gate against pre-fix code during
  implementation and transcribe the real Typst error — including the path it searched at — into the
  module docstring as the standing "Confirmed both directions" record, matching every existing gate.
  (b) **Additionally** add a unit test that asserts the path-resolution basis directly: that the file
  handed to `typst.compile()` is the master's own `.typ` (same directory as the master), not a file
  at the outdir root. Rationale: (b) catches a regression even where `typst` is not installed and
  states the invariant in one line instead of relying on a downstream compile failure.

### Cross-builder equivalence (SC#3)

- **D-09: Also compile the `-b typst` output for real.** The same fixture is built with `-b typst`,
  and the emitted `api/index.typ` is compiled via `typst.compile(path, root=outdir)` — reproducing
  exactly what a user running `typst compile` by hand does — with success asserted. Rationale: after
  D-01 the two builders compile the *same file at the same location*, so equivalence is structural;
  this test pins that structural property so a future refactor that reintroduces an intermediate copy
  is caught. Rejected: relying on the `-b typstpdf` gate alone (true today, unguarded tomorrow);
  byte-comparing the two builders' `.typ` output (Phase 22 already unified filename resolution, and
  byte equality proves nothing about path *resolution*).

### Claude's Discretion

- **Exception type for D-04/D-05** — a Sphinx `ExtensionError`, the existing `TypstCompilationError`,
  or a small aggregate wrapper. The planner picks; the requirement is a non-zero exit and a message
  naming every failed master.
- **Fixture content details** — the exact `.rst` bodies, which image format, and whether the image
  ships as a committed binary or is generated. Only the structural shape from D-07 is fixed.
- **Factoring of the two `pdf.py` functions** — whether the deprecated content version delegates via
  a private helper or calls the public file-path function directly.
- **Whether to migrate any `tests/test_pdf_generation.py` cases to the new file-path function** —
  allowed and encouraged for the cases that are really testing compilation rather than the legacy
  API, but not mandated; the 18 existing call sites must keep passing either way.
- **Documenting nested-docname masters in `docs/`** — optional; not raised as a requirement.

</decisions>

## Existing Code Insights

### Reusable Assets
- **`_resolve_output_stem()` + `_directory_preserving_relpath()`** (`builder.py`, added in Phase 22)
  — already give `finish()` the master's exact `.typ` path. D-01 needs no new path computation at
  all; it just stops throwing that path away and passes it to the compiler.
- **GATE-01 render-gate shape** (`tests/test_glob_image_render_gate.py`) — build via
  `sys.executable -m sphinx`, assert artifact + `%PDF` magic, guard on `TYPST_AVAILABLE`, document
  both directions in the docstring. Reused verbatim; only the fixture and assertions change.
- **`tests/fixtures/integration_nested_toctree/`** — an existing `chapter1/index` + siblings layout
  that models the directory shape the new fixture needs.
- **`TypstCompilationError`** (`pdf.py:15-56`) — already carries `typst_error` + `source_location`;
  a ready-made carrier for D-04/D-05's aggregated failure report.

### Established Patterns
- **Content-in / bytes-out `compile_typst_to_pdf()`** is the only compile entry point today, and it
  is the *reason* for the temp file. Phase 22's CONTEXT called it "filename-agnostic, no change
  expected" — true for Issue #117, but it is exactly the surface PDF-02 must change.
- **Every write/read/compile site must agree on one path expression.** Phase 22 established this
  invariant for filenames (`write_doc` vs `finish`); D-01 extends it to the compile step.
- **Real-compile gates live in `tests/fixtures/` + `tests/test_*_render_gate.py`**; `tests/roots/`
  holds only `test-basic`.
- **Failures in `finish()` have historically been swallowed** — D-04 breaks with that precedent
  deliberately, and unlike Phase 22's D-06 (warn) it hard-fails, because the artifact is absent
  rather than merely misnamed.

### Integration Points
- `typsphinx/pdf.py` — new file-path function + the deprecated delegating content function.
- `typsphinx/builder.py` `TypstPDFBuilder.finish()` — call the new function with the master's own
  `.typ` path; replace the swallow with aggregate-and-raise.
- New `tests/fixtures/<gate>/` project + new `tests/test_*_render_gate.py` module + a unit test for
  the path-resolution basis.
- Phase 23 hand-off: CHANGELOG entries for the D-04 breaking change and the D-03 deprecation.

<specifics>
## Specific Ideas

- The temp file is **deleted, not relocated.** After the fix `-b typstpdf` compiles the very same
  `.typ` file that `-b typst` emits, at the very same location — basis divergence becomes
  structurally impossible rather than merely corrected.
- `root=self.outdir` stays exactly as it is. Only the *compiled file's location* changes.
- The legacy content-string API survives byte-for-byte, delegates inward, is documented as
  deprecated with its root-relative limitation spelled out, and is scheduled for removal in v0.7.
- A silently missing PDF with exit code 0 is treated as unacceptable: the build fails, after every
  master has been attempted, with one aggregated message naming each failure.
- The gate fixture is a nested master with `target == docname` — one variable, one failure mode.
- Two independent proofs of the pre-fix break: the real Typst "file not found" transcribed into the
  gate's docstring, and a unit test asserting the compiled path is the master's own `.typ`.
- SC#3 is pinned by actually running `typst.compile()` on the `-b typst` output, mirroring what a
  user does by hand.

</specifics>
