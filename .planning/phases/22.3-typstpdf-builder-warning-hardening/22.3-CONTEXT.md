# Phase 22.3: typstpdf Builder Warning Hardening - Context

**Gathered:** 2026-07-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 22.1 の `/gsd-code-review` が挙げた Warning 2 件（WR-01 / WR-02）を閉じる。新機能はゼロ。

- **WR-01** — `TypstPDFBuilder.finish()` の失敗契約の非対称を解消する。ループ内の 2 つの
  `logger.warning(...) + continue` 分岐（`builder.py:885-890` の malformed `doc_tuple`、
  `builder.py:895-897` の欠損 `.typ`）は `failures` に何も積まないため、設定済みマスターの
  PDF が 1 つも出ないまま `sphinx-build` が exit 0 する。docstring（`builder.py:850-856`）の
  D-04「no silent success」主張と実装が一致していない。
- **WR-02** — `tests/test_nested_master_render_gate.py` が `typst-py` のエラーメッセージ
  リテラルに assert しているため、上流が文言を変えただけで typsphinx に回帰が無くても CI が赤くなる。

**スコープ外（明示）:** 同レビューの Critical `CR-01` は Phase 22.1 内で gap `G-22.1-4` として
既に閉じている。`typst_documents` の入力型バリデーション全般（非文字列 docname 等）も範囲外。
`typsphinx/translator.py` は一切触らない。

</domain>

<decisions>
## Implementation Decisions

### WR-01 — 欠損マスターの扱い

- **D-01: 欠損 `.typ` は behavioral に解決する — `failures` に乗せてビルドを失敗させる。**
  `builder.py:895-897` の `logger.warning(...); continue` を `failures.append(...); continue` に
  変える。docstring を弱める documentary 案（コードは変えず「コンパイル失敗については」と限定）は
  却下 — 「PDF が一つも出ないのに exit 0」という実害が残るため。新設定値による opt-in 案も却下 —
  直前の Phase 22.2 で死に設定値を削った直後に新規設定値を増やすのは方針矛盾。

- **D-02: 既存の `failures` リストに同居させ、ループ後に一括集約する。**
  22.1 D-05 の attempt-all-then-raise 契約をそのまま維持する。fail-fast（その場で raise）は
  22.1 D-05 と非対称になるため却下。`missing` と `failures` を別リストに分けて区別表示する案も
  却下 — `finish()` の後半が不必要に複雑化する。**ただし集約メッセージの末尾文言は要調整:**
  現行は `f"typst_pdf: {len(failures)} master document(s) failed to compile to PDF: ..."` と
  「compile」に限定されているので、コンパイル以前の失敗も含む形（例「failed」）へ広げる必要がある。

- **D-03: 移行緩和は置かない — clean break ＋ CHANGELOG。**
  互換シムも deprecation 期間も設けない。22.1 D-04/D-05a および 22 D-08 の前例に従い、これは
  docstring が既に約束していた振る舞いへのバグ修正として扱う。**Phase 23 への引き継ぎ義務:**
  この user-visible な振る舞い変更は `[0.6.2]` CHANGELOG エントリに明記すること（ROADMAP SC#3）。
  22.1 D-05a が既に「compile 失敗がビルド失敗になる」を同エントリに載せる前提なので、同じ項目に同居させられる。

- **D-04: 欠損時のメッセージは「docname が未知か」を判別して文言を分ける。**
  `docname not in self.env.found_docs` のとき、`typst_documents` の docname がプロジェクトに
  存在しない旨を名指しする（タイプミス／`.rst` 拡張子付きで書いた／`exclude_patterns` で除外、が
  圧倒的多数の原因）。それ以外の原因なら現行のパス提示にフォールバックする。
  **根拠（実測）:** `get_outdated_docs()` は毎回 `env.found_docs` 全件を返す（`builder.py:269-279`）
  ので、増分ビルドで書き漏れる経路は存在しない。よって欠損は実質「設定された docname が
  `found_docs` に無い」ケースに絞られる。`difflib` による候補提示（"did you mean ...?"）は
  warning ハードニングというフェーズ目的に対して機能追加寄りなので却下。

### WR-01 — malformed `doc_tuple` の扱い

- **D-05: malformed 分岐も欠損 `.typ` と同じ扱いにする — `failures` に乗せる。**
  ROADMAP SC#2 が求める「2 つのスキップ分岐を非対称のまま放置しない」を最も直接に満たす。
  空エントリは常に設定ミスであり、意図した PDF が 1 つ出ない点で欠損 `.typ` と同質。

- **D-06: ガードの検知範囲は現行の `if not doc_tuple:` のまま広げない。**
  WR-01 が指摘したのは「スキップが `failures` に入らない」ことであって、検知できていない入力が
  あることではない。変更は `logger.warning + continue` → `failures.append + continue` の
  乗せ換えに留め、フェーズの診断差分を最小に保つ。
  **既知の未対応（今回は入れない）:** `(123, ...)` のような非文字列 docname は `path.join` に
  到達して `try` の**外側**で `TypeError` を投げ、集約 `ExtensionError` に乗らず生の traceback で
  落ちる。これは別種の欠陥なので `<deferred>` に記録する。

- **D-07: malformed エントリの識別子はエントリの値そのもの（`repr`）を使う。**
  `failures: List[Tuple[str, str]]` の第 1 要素はこれまで常に docname だが、空エントリには
  docname が無い。`failures.append((repr(doc_tuple), "..."))` として集約メッセージに
  `(): malformed typst_documents entry` の形で出す。現行 warning のメッセージと見た目が揃い、
  実装も 1 行で済む。`enumerate` を導入してインデックス表記にする案は、得られる一意性に対して
  ループ構造の変更が割に合わないため却下。

### WR-02 — テストの上流エラー文言からの切り離し

- **D-08: 失敗クラスの証明を ablation（欠けているものを置いたら通る）に乗せ換える。**
  「文言が何と書いてあるか」ではなく「何を直すと通るか」でクラスを特定する。上流が文言を変えても
  赤くならず、かつ `pytest.raises(Exception)` だけの tolerant assertion（レビュアーが却下済み）に
  逆戻りしない。フィルタとしての判別力を保ったまま WR-02 を閉じられる唯一の案。
  **前提の明示:** ablation は「通る」側の確認が証明の本体である。赤だけでは赤の理由を何も
  示せないため、グリーン半分を省略してはならない。フィクスチャは極小なのでコンパイル追加実行の
  実行時間コストは無視できる。

- **D-09: include 半分を完全グリーンの ablation にし、template 半分はその差分として位置づける。**
  `test_sibling_include_fails_at_outdir_root_and_resolves_in_place` 側で、`usage.typ` /
  `logo.png` を outdir root に置けば root コピーが通ることを示し、「失敗原因は sibling include /
  upward image の解決だった」を文言なしで確定させる。これが確定すれば、
  `test_outdir_root_compile_basis_still_fails` のバイト同一コピーの失敗は「template を中立化しても
  消えない別クラス」として差分で位置づけられる。**却下:** template 半分にも独立した ablation を
  組む案（outdir の親に `_template.typ` を置き `root=parent(outdir)` でコンパイルして脱出エラーを
  消す）— 成立はするがフィクスチャ外にファイルを置く段取りが入る。2 つのテストをどう分割・統合するかの
  最終形は research/plan の裁量。

- **D-10: typst のエラー文言への assert はゼロにする。**
  実測 6 箇所（`"escape"`, `"_template.typ"`, `"usage.typ"`, `"not found"`,
  `"_template.typ" not in`, `"api/usage.typ" not in`）を全廃する。エラー文字列は失敗時の診断
  メッセージ（`f"...: {error_text!r}"`）にのみ埋め込み、人間が読む情報としては残すが CI の
  赤黒判定には一切影響させない。分類語だけ落としてパス名 assert を残す部分対応は却下 — WR-02 を
  半分しか閉じない。typst-py の版 pin ＋文言維持も却下 — 問題を先送りするだけで、`drift.yml`
  （毎週の最新依存再解決）と衝突する。

### 検証

- **D-11: 新しく落ちるようになる振る舞いは専用フィクスチャで固定する。**
  22.1 D-06 / 22.2 D-09 と同じ GATE-01 形式に従い、`tests/fixtures/` 配下に最小フィクスチャを
  新設する（`typst_documents` に存在しない docname と空エントリを含む conf）。実際に
  `-b typstpdf` を走らせ、非ゼロ終了と `ExtensionError` を確認する。既存フィクスチャは汚さない。
  ユニットテストで conf をモンキーパッチする軽量案は、実 `sphinx-build` 経路を通らない分
  回帰検知力が落ちるため却下。SC#5 のとおり、このテストは pre-fix コードに対して**失敗**しなければ
  ならない。

### Claude's Discretion

- 2 つの render-gate テストを最終的にどう分割／統合するか（D-09 の後半）は research/plan の裁量。
- 集約 `ExtensionError` 本文の正確な文面（D-02 で「compile」限定を広げる必要があることのみ確定）。
- 欠損 docname メッセージの正確な文面（D-04 で判別ロジックの有無のみ確定）。

  — 本フェーズの source of record そのもの。backlog 999.5 として登録され、2026-07-22 に

  D-01 で behavioral に確定した。フェーズ完了時にこの todo は close されるべき。

<specifics>
## Specific Ideas

- **「通す確認」の位置づけ（オーナー確認事項、2026-07-22）** — ablation を選んだ以上、
  グリーン半分は省略できない。「失敗した」だけでは失敗の理由を何も示せず、それは
  Phase 22.1 のレビュアーが明示的に却下した tolerant assertion に逆戻りする。
  「X を足したら通った」があって初めて「失敗原因は X の不在だった」と言える。

- **スコープの線引き（オーナー指摘、2026-07-22）** — WR-01 / SC#2 が問うているのは
  「2 つの `continue` が `failures` に乗らない非対称」であって、`typst_documents` の入力型
  バリデーションではない。議論中にこの線を越えかけたが、D-06 で現行ガードの範囲に戻した。

</specifics>
