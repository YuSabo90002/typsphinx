# Phase 22.3 Plan 01: finish() Fails Loudly for Missing-.typ and Malformed Entries Summary

**`TypstPDFBuilder.finish()` now appends missing-`.typ` and malformed-`typst_documents`-entry failures to the same aggregate `ExtensionError` as compile errors, closing the silent-success gap WR-01 exists to remove, with a `found_docs`-based message split that keeps `tests/test_target_name_render_gate.py:122`'s byte-frozen assertion intact.**

## Accomplishments

- Added four new unit tests to `TestPDFErrorHandling` in `tests/test_pdf_generation.py`, all proven RED against the unmodified builder (Task 1) — half of ROADMAP SC#5's pre-fix must-fail proof.
- Extended `TypstPDFBuilder.finish()` (`typsphinx/builder.py`) so both the malformed-`doc_tuple` guard and the missing-`.typ` guard append to the existing `failures` list instead of only warning-and-continuing (Task 2). The missing-`.typ` guard now discriminates on `self.env.found_docs` membership, producing a new diagnostic for an unknown docname while preserving the old fallback text byte-for-byte for a known docname whose `.typ` simply wasn't generated.
- Widened the aggregate `ExtensionError` wording past "compile" and extended the `finish()` docstring to name all three failure kinds explicitly.
- Filed the D-06 non-str-docname `TypeError` deferral as a new pending todo, with the verified call chain and scope rationale (Task 3).
- Ran the full non-slow suite: 519 passed, 45 failed (pre-existing, environmental — see "Issues Encountered"), 23 deselected. Zero regressions attributable to this plan's changes.

## Decisions Made

- Kept the malformed-entry detection condition exactly `if not doc_tuple:` per D-06 — did not widen it to catch the non-str-docname case, which is a distinct, deliberately-deferred defect class.
- Preserved the fallback message `f"Master document not found: {typ_file}"` byte-identical (single-line, unwrapped) specifically so `grep -c` and `tests/test_target_name_render_gate.py:122`'s negative-substring assertion both keep working against the exact pre-existing text.
- Reformatted the new unknown-docname message onto a single logical line (`f"Master document {docname!r} is not a known Sphinx document"` + continuation) rather than the illustrative multi-line split in RESEARCH.md's example diff, because the plan's own acceptance criterion (`grep -c 'is not a known Sphinx document' typsphinx/builder.py` returns 1) requires the full phrase to appear on one physical line — a naive line-wrap that split "Sphinx" and "document" across lines would have made that grep return 0. Verified with `black --check` that the single-line form is still within the project's 88-column limit after formatting.

> `sphinx-build -b typstpdf` now fails the build for **any** configured master that produces no PDF — not only a Typst compile error, but also a master whose `.typ` file was never generated (typically a `typst_documents` docname that is not a real Sphinx document) and a malformed `typst_documents` entry. Builds that previously "succeeded" while silently producing no PDF for such a master now exit non-zero.
