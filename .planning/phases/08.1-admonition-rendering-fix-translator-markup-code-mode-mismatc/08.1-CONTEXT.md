# Phase 08.1: admonition rendering fix — translator markup/code-mode mismatch - Context

**Gathered:** 2026-07-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the admonition markup/code-mode mismatch in `typsphinx/translator.py` so that
admonitions (`.. note::`, `.. warning::`, etc.) render **typeset prose** in the compiled
PDF instead of literal, unevaluated Typst source (`par({text("...")})`).

**Root cause:** `_visit_admonition` opens the gentle-clues call in Typst **markup mode**
(`info[`, `warning[`, …) but the body content (paragraphs, text) is emitted by
`visit_paragraph`/`visit_Text` in the translator's "unified code mode" as
`par({...})` / `text(...)` function-call forms. Inside `[...]` markup content those
code-mode calls are **not re-entered as code** — they print as literal characters.
This is orthogonal to `@preview` package versions; it reproduces with any
gentle-clues/mitex/codly version (pre-existing since 2025-10-13, first visible only after
Phase 7's `kai` fix let `docs-pdf` compile).

**This phase deliberately widens slightly beyond a pure bug fix** (per user's explicit
choices below): it also preserves inline markup in admonition titles and adds the
currently-unimplemented admonition types. The ROADMAP goal for this inserted urgent phase
was TBD; this discussion defines its scope.

**Out of scope:** translator state-management refactor, changes to non-admonition nodes
beyond what the shared paragraph/text code-mode interaction requires, configurable
`@preview` versions (FWD-03).

</domain>

<decisions>
## Implementation Decisions

### Fix approach (markup ↔ code-mode bridge) — Claude's discretion
- **D-01:** The user did **not** select this area, delegating the exact approach to
  research/planning. **Recommended direction:** keep the translator's *unified code mode*
  intact end-to-end rather than mixing modes. Prefer a **content-block form** where the
  admonition body stays in code mode (e.g. wrapping the emitted body so `par({...})`,
  lists, code blocks, and nested admonitions all evaluate), instead of the alternative
  `info[#par({...})]` per-element `#`-prefix hack.
  - **Rationale:** the whole translator already emits code-mode calls (`par({...})`,
    `text(...)`); a single code-mode-consistent invocation naturally handles arbitrary
    nested block content (D-05) without per-child prefixing.
  - **Left to research:** the exact gentle-clues invocation syntax under the bumped
    versions (gentle-clues 1.3.1, typst 0.15). Verify how `info`/`warning`/`tip` accept a
    code-mode body AND a content-type `title:` argument together (see D-02). Confirm against
    the real gentle-clues API — do not assume the syntax.

### Title inline markup
- **D-02:** **Preserve inline markup in admonition titles.** Replace the current
  `title: "{node.astext()}"` (which flattens emphasis/`code`/etc. to a plain string) with a
  content-mode title that keeps inline markup. This requires walking the `title` node's
  children through the normal inline visitors rather than calling `.astext()`.
  - **Left to research:** whether gentle-clues' `title:` parameter accepts content
    (`title: [...]`) — it is expected to, but must be verified — and how that composes with
    the code-mode body invocation chosen in D-01.

### Regression-proofing / acceptance gate
- **D-03:** **Strengthen unit assertions AND add a real `docs-pdf` compile gate.** The
  existing `tests/test_admonitions.py` asserts are too loose (`"info[" in output`,
  `"This is a note." in output`) — they passed even while the body rendered as literal
  source, which is exactly how this bug slipped through. New tests must assert the
  **code-mode structure** (e.g. no literal `par({` appears directly inside the admonition
  body markup / the body is emitted in the chosen code-mode form).
- **D-04:** Acceptance gate additionally requires a real compile: `tox -e docs-pdf` exits 0
  and a **text extraction from the compiled PDF** confirms no literal Typst source
  (`par({`, `text(`, `raw(`) leaks into rendered admonition prose. This mirrors the fact
  that the bug was only ever visible in a real render, not in the loose in-process asserts.

### Coverage / scope of the fix
- **D-05:** **Guarantee the fix for all nested admonition content** — multiple paragraphs,
  lists inside admonitions, code/literal blocks inside admonitions, and nested admonitions
  — not just single paragraphs. Regression tests should cover these nested cases.
- **D-06:** **Also implement the currently-missing admonition types** in this phase
  (user explicitly chose scope-expansion over deferral): generic `.. admonition::`
  (`visit_admonition`/`depart_admonition`), plus `hint`, `error`, `danger`, `attention`.
  Existing handlers (`note`→info, `warning`, `tip`, `important`→warning+title,
  `caution`→warning, `seealso`→info+"See Also") stay, mapped consistently to gentle-clues
  clue types. **Left to planning:** the clue-type mapping for the new types (e.g.
  `danger`/`error`→warning-family, `hint`→tip/info) — pick sensible gentle-clues analogs.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **Unified code-mode emission** (`par({...})`, `text(...)`, list/code-block visitors): the
  fix reuses this — the goal is to make the admonition body *stay in* this mode, not to
  invent a parallel markup-mode path.
- **`_visit_admonition(node, clue_type, custom_title=...)` helper**: single choke point —
  every admonition type routes through it, so the mode fix + title fix land in one place.
  New types (D-06) plug into the same helper.

### Established Patterns
- Everything downstream of a block container is emitted in code mode; the only markup-mode
  island is the admonition `[...]` bracket. The mismatch is precisely at that boundary.
- `in_list_item` / `list_item_needs_separator` state already special-cases paragraph
  wrapping — the admonition body fix must stay consistent with that list-item handling
  (admonitions can appear inside list items and vice versa; D-05 nesting).

### Integration Points
- gentle-clues (`@preview`) is the render target for admonitions; the invocation syntax is
  the contract. Verify body + `title:` content-mode composition against the pinned
  gentle-clues 1.3.1 API.

</code_context>
