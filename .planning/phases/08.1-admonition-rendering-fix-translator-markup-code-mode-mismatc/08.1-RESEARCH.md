## Summary

The bug is a genuine markup-mode/code-mode boundary defect, not a package-version issue.
`_visit_admonition` opens the gentle-clues call with a Typst **markup-mode trailing content
block** (`info[`), but every visitor downstream (`visit_paragraph`, `visit_Text`,
`visit_literal`, etc.) unconditionally emits **code-mode function-call syntax**
(`par({...})`, `text(...)`, `raw(...)`) with no `#` prefix. Inside `[...]` that syntax is
literal text, not evaluated code — exactly the symptom in the deferred-items writeup.

I obtained the actual pinned `gentle-clues` **1.3.1** source from this machine's local Typst
package cache (`~/.cache/typst/packages/preview/gentle-clues/1.3.1/lib/{clues,predefined}.typ`)
and empirically compiled multiple test snippets with the project's own pinned `typst-py`
0.15.0 (verified installed), then extracted text from the resulting PDFs with `pypdf` to
confirm actual rendered output. This is the same real code that ships in `typst_documents`
today — not a guess from training data or a web search.

**Verified finding:** `clue`'s signature is `clue(content, title: none, icon: none,
accent-color: navy, ...)` and every predefined clue (`info`, `warning`, `tip`, `danger`,
`error`, etc.) is a pure `..args` passthrough to `clue`. `content` is a genuine **positional
content-typed parameter** — it can be supplied as a **code-mode expression**
(`info({ par({...}) })`), not only via the markup-bracket sugar (`info[...]`). `title:` also
accepts an arbitrary content value (verified empirically — no runtime type assertion exists
on `title` in `clue()`; only color parameters are asserted). This means the whole admonition
body — and the title — can be produced entirely by the translator's existing "unified code
mode" machinery, with **zero markup-mode toggling, zero `#`-prefix hacks needed anywhere**.

**Primary recommendation:** Change `_visit_admonition`/`_depart_admonition` to emit
`{clue_type}({` ... `})` (a code-mode content-block argument) instead of `{clue_type}[` ...
`]` (a markup-mode trailing block), and defer the `title:` argument to the closing call by
buffering the title node's rendered content — reusing the exact buffer-swap pattern already
established in this file for definition-list terms (`visit_term`/`depart_term`). This is a
**verified, compiled, PDF-text-extracted, working solution** for nested paragraphs, lists,
code blocks, nested admonitions, and inline-markup titles (see Code Examples).

### Pattern 1: Code-mode content-block admonition body (verified working)

**What:** Replace the markup-bracket trailing-content-block call (`info[...]`) with a
genuine code-mode function call whose sole positional argument is a `{ ... }` content
block. Typst code blocks auto-join every content-producing statement inside them (this is
the exact same mechanism the document-level `#{ ... }` wrapper and `par({...})` already
rely on elsewhere in this file — no new Typst semantics are introduced).

**When to use:** Every admonition visitor (`_visit_admonition`/`_depart_admonition`), for
all clue types, with or without a title.

**Verified example** (compiled with the project's pinned `typst` 0.15.0 + cached
`gentle-clues` 1.3.1, PDF text extracted with `pypdf` — see raw output below):

```typst
// Source: empirically compiled in this research session against
// ~/.cache/typst/packages/preview/gentle-clues/1.3.1/lib/{clues,predefined}.typ
#import "@preview/gentle-clues:1.3.1": *

#{
  info({
    par({text("Before list.")})
    list({
      par({text("Item one.")})
    }, {
      par({text("Item two.")})
    })
    codly(number-format: none)
    ```python
    x = 1
    ```
  })
}
```

`pypdf` extraction of the compiled PDF: `'Info\nBefore list.\n• Item one.\n• Item two.\nx = 1 python'`
— fully typeset prose, list bullets rendered, code block rendered. **Zero literal Typst
source leaked.**

### Pattern 2: Deferred title-content via buffer swap (reuses existing `visit_term` idiom)

**What:** `_visit_admonition` cannot know the title's *rendered* Typst fragment at the
moment it fires (`visit_note` runs before docutils descends into the note's children,
where the title node actually lives). The fix must **defer** title emission using the
buffer-swap pattern **already implemented** in this file for definition-list terms
(`visit_term`/`depart_term`, `typsphinx/translator.py:1047-1082`) — do not invent a new
mechanism.

**When to use:** `visit_title`/`depart_title`, gated on `isinstance(node.parent,
nodes.Admonition)` (confirmed: `docutils.nodes.Admonition` is the shared mixin base for
`note`, `warning`, `tip`, `important`, `caution`, `hint`, `error`, `danger`, `attention`,
and the generic `admonition` — verified via `cls.__mro__` introspection in this session).

**Example (concrete recommendation, not yet in the codebase):**
```python
def visit_title(self, node: nodes.title) -> None:
    if isinstance(node.parent, nodes.Admonition):
        self._saved_body_for_admonition_title = self.body
        self.body = []
        self._in_admonition_title = True
        return
    # ... existing heading()-emission path unchanged ...

def depart_title(self, node: nodes.title) -> None:
    if getattr(self, "_in_admonition_title", False):
        self._pending_admonition_title = "".join(self.body)
        self.body = self._saved_body_for_admonition_title
        self._in_admonition_title = False
        return
    # ... existing heading()-close path unchanged ...
```
```python
def _visit_admonition(self, node, clue_type, custom_title=None):
    self._pending_admonition_title = None  # cleared per-admonition
    self.add_text(f"{clue_type}({{")        # open code-mode content-block (NOT "[")
    # custom_title (static Python str, e.g. "Important"/"See Also") still handled as before —
    # emitted at depart time alongside any buffered dynamic title.

def _depart_admonition(self, custom_title: str = None) -> None:
    self.add_text("}")
    title_expr = None
    if self._pending_admonition_title:
        title_expr = "{" + self._pending_admonition_title + "}"
    elif custom_title:
        title_expr = f'"{custom_title}"'
    if title_expr:
        self.add_text(f", title: {title_expr}")
    self.add_text(")\n\n")
```

**Verified compile** (title with inline emphasis, composed with the content-block body):
```typst
#{
  clue({par({text("Generic admonition body.")})}, title: [My Custom *Title*])
}
```
`pypdf` extraction: `'My Custom Title\nGeneric admonition body.'` — title renders as
typeset content (bold markup consumed correctly by Typst even though `pypdf` can't show
boldness in extracted text; word "Title" is present exactly once, i.e. the current
double-emission bug, reproduced below, is gone).

### CONCRETE BUG FOUND (not previously documented): current code double-emits titles

Running the **existing, unmodified** `_visit_admonition` against docutils' own traversal
(reproduced in this session with a debug test mirroring `test_admonition_with_title_in_content`)
produces:
```
info(title: "Custom Title")[heading(level: 0, text("Custom Title"))

par({text("Content here.")})

]
```
The extracted title text (`title: "Custom Title"`) is a **string flatten** taken by
`_visit_admonition` scanning `node.children` directly, but docutils' own traversal *also*
subsequently visits the same `title` child normally (since it's never skipped), calling the
**default** `visit_title`, which emits a second, broken `heading(level: 0, ...)` occurrence
**inside the still-markup-mode body** — i.e., today's code has a **title duplication bug**
in addition to the code-mode mismatch, for the one existing test that constructs a title
child. Pattern 2 above (buffer-swap + skip default heading path when parent is an
Admonition) fixes both issues simultaneously: the title is captured exactly once, and it
is captured via the buffer-swap rather than `node.astext()`, which is what enables D-02's
inline-markup preservation.

### Anti-Patterns to Avoid
- **`#`-prefix hack per visitor (the alternative D-01 explicitly asks to avoid):** Extending
  the existing `_in_markup_mode` flag (already used by `visit_Text`/`visit_reference`/
  `visit_emphasis`/`visit_strong` for the *unrelated* reference-with-target case, lines
  ~1509-1965) to `visit_paragraph`, `visit_literal_block`, and every nested-admonition
  visitor would require touching every current and future body-content visitor and is
  fragile against future node types. The content-block form fixes the boundary once, at
  the single admonition choke point.
- **Mutating `node.children` to remove the title node:** Tempting alternative to the
  buffer-swap (drop the title child so the normal traversal never reaches it), but risks
  mutating a doctree that may be reused/pickled elsewhere in the Sphinx build (the
  `typst`/`typstpdf` builders share the same resolved doctree object per docname with
  other builders in a multi-builder run). The buffer-swap approach touches only the
  translator's own transient state, not the tree.
- **`node.astext()` for titles:** Flattens all inline markup (the exact defect D-02 must
  fix) — never use `.astext()` for content that must preserve emphasis/literal/etc.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Buffering a sub-tree's rendered output for later reuse | A new ad-hoc "capture mode" flag/mechanism | The existing `saved_body` / buffer-swap pattern (`visit_term`/`depart_term`, `visit_definition`/`depart_definition`, `typsphinx/translator.py:1047-1122`) | Already implemented, already tested indirectly via definition-list tests; reusing it keeps the state-management surface area from growing |
| PDF text extraction for the D-04 gate | A regex/heuristic scrape of the raw `.typ` source (would only prove the *translator* didn't emit `par(`, not that the *compiled PDF* renders correctly — CONTEXT.md explicitly requires the real-compile gate because that's the only place the bug was ever visible) | `pypdf.PdfReader(...).pages[i].extract_text()` | Verified in this session: correctly distinguishes rendered prose from literal leaked source |
| Deciding which gentle-clues function to call per admonition type | A parallel custom clue definition via `#clue(...)` for every Sphinx type | The predefined functions already shipped in `gentle-clues` 1.3.1 (`info`, `warning`, `tip`, `danger`, `error`; `clue` itself for the generic type) | `predefined.typ` already defines exactly these with theme-matched icon/color; recreating them duplicates the package's own job |

**Key insight:** The whole fix is contained to a single choke point
(`_visit_admonition`/`_depart_admonition`) plus one conditional branch in
`visit_title`/`depart_title`. No new Translator state categories, buffering mechanisms, or
gentle-clues customization are needed — everything required already exists either in this
file (buffer-swap) or in the pinned package (`content`-typed positional arg + content-typed
`title:`).

## Common Pitfalls

### Pitfall 1: Assuming `title:` only accepts a string
**What goes wrong:** Continuing to build `title: "{node.astext()}"` because the gentle-clues
source *comment* annotates `title` as `-> string | none`.
**Why it happens:** That doc-comment is descriptive, not type-enforced — Typst is
dynamically typed and `clue()` never asserts on `title`'s type (only on the four color
parameters). Internally, `title` is passed straight into `text(font: ..., title)`, and
Typst's built-in `text()` accepts a `content` body just as readily as a `str`.
**How to avoid:** Verified empirically in this session — `title: [My Custom *Title*]` and
`title: {par({text("...")})}`-style content both compiled and rendered correctly.
**Warning signs:** A `mismatched types` Typst compile error mentioning `title` would falsify
this — none occurred across all four verification compiles in this session.

### Pitfall 2: Forgetting that `_visit_admonition` fires *before* its own children
**What goes wrong:** Trying to compute the final `title: ...` argument text inside
`_visit_admonition` itself, before the title's inline content has actually been rendered by
the normal traversal (which happens strictly after `visit_note`/`visit_warning`/etc.
returns).
**Why it happens:** Docutils' visitor contract is `visit_X(node)` → **then** descend into
`node.children` → `depart_X(node)`. The title, if present, is a *child*, not something
`_visit_admonition` can render synchronously.
**How to avoid:** Defer — buffer the title, close the call (append `title: ...`) only in
`_depart_admonition`, exactly like the reasoning in Pattern 2.
**Warning signs:** Any implementation that tries to build the whole `info(title: ..., {`
string in one `add_text()` call inside `visit_note` is guaranteed wrong for dynamic (inline
markup) titles.

### Pitfall 3: The existing test suite would NOT have caught the double-title-emission bug
**What goes wrong:** `tests/test_admonitions.py::test_admonition_with_title_in_content`
currently passes even though the real (unmodified) output contains a broken, duplicate
`heading(level: 0, ...)` fragment inside the still-markup-mode body (reproduced above).
**Why it happens:** The assertion is a loose substring check (`"Custom Title" in output`),
which is true whether the title appears once (correct) or twice (bug).
**How to avoid:** Per D-03, new tests must assert **exact structural absence** of stray
`heading(`/duplicate title fragments inside the body, not just substring presence. See
Validation Architecture below for the concrete assertion shape.
**Warning signs:** Any test using `"X" in output` where `X` is short/generic text that could
appear from either the correct or the broken code path.

### Pitfall 4: `visit_literal_block`'s captioned-code path is a *second*, unrelated
markup-mode island
**What goes wrong:** While auditing `visit_literal_block` for D-05 nested-code-block
coverage, this session found `figure(caption: [...])[` (line ~910) — a trailing
markup-bracket content block, structurally identical in shape to the admonition bug.
**Why it happens:** It is a different node path (only triggered by
`literal-block-wrapper` containers, i.e., `:caption:` on a code block) and is **out of
scope** for this phase per CONTEXT.md's explicit "Broader translator markup/code-mode
audit... not in scope" deferral.
**How to avoid:** Do not fix it in this phase. Flag it as a candidate for the deferred
broader audit (already logged in CONTEXT.md's Deferred Ideas) — mentioning here only so the
planner does not accidentally assume D-05's "code blocks inside admonitions" coverage
implies this caption path is also fixed. **Verified**: a plain (non-captioned) code block
nested inside the new content-block admonition form compiles and renders correctly (see
Pattern 1's example, which includes a bare ` ```python ` fence) — only the *captioned*
figure-wrapping path has the separate, out-of-scope issue.

## Open Questions

   - What's unclear: Whether the planner should replace/add a fixture using the real `nodes.admonition` type for the generic-admonition D-06 test.
   - Recommendation: Add a `nodes.admonition()`-based test for D-06 coverage; keep or fix the existing `nodes.note()`-with-title test only if there's a reason a `note` might carry a title in practice (there is not, per standard rST semantics) — otherwise consider it a test-fixture inaccuracy to note during planning, not something this research should silently "fix" by rewriting existing tests.

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Typst source injection via unescaped rST content flowing into `add_text()` as raw interpolated strings | Tampering (of generated document, not a runtime security boundary — worst case is malformed/broken PDF output, not code execution, since Typst is a sandboxed typesetting language, not a general-purpose script host with filesystem/network access by default) | Continue routing all dynamic text through the existing `visit_Text`/`visit_literal` escape logic; the title buffer-swap fix must **not** bypass these by reintroducing a raw `.astext()` interpolation (this is exactly what D-02 requires removing) |

No new threat surface is introduced by this phase; if anything, replacing
`title: "{node.astext()}"` (an f-string interpolation of arbitrary rST text with no
escaping applied at all today) with content walked through the existing escaping visitors
is a **security improvement** — the current code has an unescaped-string-interpolation gap
for admonition titles that this fix incidentally closes.
