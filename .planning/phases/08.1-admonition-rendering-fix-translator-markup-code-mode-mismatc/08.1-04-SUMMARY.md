## Decisions Made

- **`sys.executable -m sphinx` instead of `["uv", "run", "sphinx-build"]` for the new test's subprocess invocation.** This sandbox has a stray, non-Nix (`dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2`) `uv` binary installed at `.venv/bin/uv` — almost certainly pulled in transitively as a bundled platform wheel by the `uv` PyPI package that `tox-uv` depends on. On PATH inside a pytest-launched subprocess, `.venv/bin` resolves before the correct Nix-provided `uv` (`/nix/store/.../uv-0.11.25/bin/uv`), so `subprocess.run(["uv", "run", ...])` picks up the broken binary and NixOS's stub dynamic linker refuses to run it (`Could not start dynamically linked executable: uv`, exit 127). This is the exact root cause of the "~45 pre-existing failures" documented as an environment limitation by prior executors in this session (confirmed: those tests all use the identical `["uv", "run", "sphinx-build", ...]` subprocess pattern in `tests/test_integration_nested_toctree.py`, `tests/test_examples_basic.py`, etc.). Using `sys.executable -m sphinx` sidesteps the whole PATH-resolution hazard by reusing the exact interpreter already running the test — no dependency on an external `uv` binary being resolvable at all. This is strictly more robust than the pattern it deviates from, not a weakening.
- **Patched `.venv/bin/uv`'s ELF interpreter via `patchelf`** (`patchelf --set-interpreter <nix-glibc-ld> .venv/bin/uv`) to make `tox -e docs-pdf` runnable in this sandbox. `tox-uv`'s `uv-sync` invocation hardcodes an absolute path to `.venv/bin/uv` (not resolved via `PATH`), so no amount of `PATH` reordering fixes it — only patching the binary itself (or replacing it) works, and `patchelf --set-interpreter` is the exact workaround referenced by the NixOS stub-ld error message itself (`https://nix.dev/permalink/stub-ld`). `.venv/` is gitignored and fully regenerable by `uv sync`, so this is a local environment fix with zero impact on the committed tree; a fresh clone/CI run is unaffected either way. Backed up the original binary to the scratchpad before patching, for safety.
- **Did not add a repo-side leak-signature check against the real docs PDF.** A manual, exploratory `pypdf` pass against `docs/_build/pdf/index.pdf` (beyond what Task 3 strictly required) turned up `par({` in the extracted text — investigated and confirmed a false positive: it's the literal prose of `TypstTranslator.depart_paragraph`'s own docstring ("Closes par({}) function and adds spacing.") rendered by the auto-generated `docs/source/api/` reference, not a code-mode leak. This is exactly why the D-04 gate is fixture-based rather than a blanket real-docs grep — the fixture's content is fully controlled and contains no docstrings quoting Typst syntax as English prose. No code change was needed or made.
