# Phase 22.2: Code Review Report

**Reviewed:** 2026-07-22
**Depth:** standard
**Files Reviewed:** 20
**Status:** issues_found

## Summary

The core routing fix (BUG-A: writer/builder agreement on when `_template.typ`
exists) is sound and well-tested — `test_package_template_routing.py` and
`test_package_only_config_gate.py` both drive real `sphinx-build` subprocesses
and assert on emitted source text, and `test_examples_charged_ieee_gate.py`
additionally proves the two bundled example projects build to real, valid
PDFs. The parameter-precedence fixes (BUG-B/E/F, D-05/D-07/D-08) in
`template_engine.py` are individually correct and covered by genuine
config→output assertions (not merely registration checks), which is what
CONF-03 demanded.

However, the specific mechanism used to close BUG-F (hoisting the four
essential `@preview` imports + codly-init calls unconditionally to the top of
`render()`) was not reconciled against the pre-existing "inline the default
template" fallback branch that the same function still contains. That branch
also emits the full contents of `base.typ` — which carries its own copies of
those same four imports and the same codly-init/`#codly()` calls — so calling
`TemplateEngine.render()` the way its own docstring says is legitimate (no
`template_file`, no `typst_package`) now emits every one of those lines
twice. I reproduced this directly (see CR-01) with the exact construction
several pre-existing tests in `test_template_engine.py` already use; those
tests still pass only because they assert substring presence, never absence
of duplicates. `writer.py`'s current routing happens to never call `render()`
this way in practice, so the shipped builders are not affected today — but
the regression is real, freshly introduced by this phase's own `render()`
change, and sits one caller-refactor away from reaching production output.

I also found one piece of dead code this phase's own stated purpose should
have removed (`_format_authors_with_details()`, superseded by the new
`map_parameters()`/`_format_typst_value()` pipeline but left in place and
still unit-tested in isolation), a stale default-mapping claim in
`docs/configuration.rst` that predates this phase but sits in a section this
phase's CONF-01 work was meant to make trustworthy, an undocumented
`typst_authors` config value (the very value BUG-C's fix makes work), and a
duplicated (and now-divergent) copy of the package/template routing decision
between `writer.py` and `builder.py`.

## Critical Issues

### CR-01: `TemplateEngine.render()` duplicates essential imports and codly-init when falling back to the inline-template path

**File:** `typsphinx/template_engine.py:335-361`
**Issue:**

Plan 22.2-04/05's BUG-F fix hoisted the four essential `@preview` imports and
the two codly show-rules out of the `if template_file:` block so they are now
emitted **unconditionally**, at the top of every `render()` call:

```python
output_parts.append("// Essential package imports")
output_parts.append('#import "@preview/codly:1.3.0": *')
output_parts.append('#import "@preview/codly-languages:0.1.10": *')
output_parts.append('#import "@preview/mitex:0.2.7": mi, mitex')
output_parts.append('#import "@preview/gentle-clues:1.3.1": *')
output_parts.append("")
output_parts.append("#show: codly-init.with()")
output_parts.append("#codly(languages: codly-languages)")
output_parts.append("")

if template_file:
    ...
else:
    # Load template inline (old behavior)
    if not self.typst_package:
        template = self.load_template()
        output_parts.append(template)
        output_parts.append("")
```

The `else` branch (reached when `template_file` is `None` and no
`typst_package` is configured — exactly the documented "inline, old
behavior" case per `render()`'s own docstring: *"If None, template is
inlined (old behavior)"*) appends the **entire contents of
`typsphinx/templates/base.typ`**, which itself contains the identical four
`#import "@preview/..."` lines and the identical
`#show: codly-init.with()` / `#codly(languages: codly-languages)` calls
(`typsphinx/templates/base.typ:8-9,14,19,22,25`). The result is every one of
those six lines emitted **twice** in the final document.

Reproduced directly against the current tree:

```
$ uv run python3 -c "
from typsphinx.template_engine import TemplateEngine
engine = TemplateEngine()
result = engine.render({'title': 'T', 'authors': ()}, 'BODY')
print(result.count('@preview/codly:'))
print(result.count('#show: codly-init.with()'))
"
2
2
```

This is not a hypothetical corner: `tests/test_template_engine.py`'s
pre-existing `TestRenderMethod`/`TestTypstTemplateFunctionDictFormat` tests
(e.g. lines ~559, 600, 620, 638, 656, 669, 685, 697, 748, 767, 785-786, 886)
construct plain `TemplateEngine()` instances with no package and call
`engine.render(params, body)` with no `template_file` — i.e. they already
exercise the exact duplicating code path on every test run — and only pass
because they assert substrings are *present*, never that essential-import
lines occur exactly once. The two regression tests this phase *did* add for
the hoist (`TestEssentialImportHoist` in the same file) cover the
package-only case and the template-file case, but not the plain-inline case,
so the gap went undetected.

Today this is masked in the real build pipeline because `writer.py` never
calls `render()` with `template_file=None` unless `typst_package` is also
set (the one case where the `else` branch's own `if not self.typst_package`
guard skips the double-inline). But `TemplateEngine.render()` is a
general-purpose method whose own docstring promises the inline path is
still supported "old behavior" — that promise is now false, and the defect
is a direct, mechanical consequence of this phase's BUG-F fix, not
pre-existing.

**Fix:** Don't emit the hoisted essential-imports block when about to fall
through to the inline-template branch (which already supplies them via
`base.typ`) — only hoist for the two paths that don't inline the full
template:

```python
will_inline_default_template = not template_file and not self.typst_package
if not will_inline_default_template:
    output_parts.append("// Essential package imports")
    output_parts.append('#import "@preview/codly:1.3.0": *')
    output_parts.append('#import "@preview/codly-languages:0.1.10": *')
    output_parts.append('#import "@preview/mitex:0.2.7": mi, mitex')
    output_parts.append('#import "@preview/gentle-clues:1.3.1": *')
    output_parts.append("")
    output_parts.append("#show: codly-init.with()")
    output_parts.append("#codly(languages: codly-languages)")
    output_parts.append("")
```

and add a regression test asserting `render(params, body)` with no
`template_file` and no `typst_package` emits each essential import exactly
once (mirroring the existing `TestEssentialImportHoist` tests' shape).

## Warnings

### WR-01: Dead `_format_authors_with_details()` retained, still tested in isolation

**File:** `typsphinx/template_engine.py:459-489`
**Issue:** This phase's own intent describes BUG-C as "`typst_authors` /
`typst_author_params` silently ignored (dead `_format_authors_with_details()`)"
and fixes it by routing `typst_authors` through `map_parameters()` →
`_format_typst_value()`'s native list/dict recursion instead (see
`template_engine.py:212-217`). That fix is correct. But
`_format_authors_with_details()` itself was left in the class, unreferenced
by any production code path (`grep -rn "_format_authors_with_details"` shows
zero callers in `typsphinx/`). Two tests in
`tests/test_template_engine.py` (`test_typst_authors_single_author_with_details`,
`test_typst_authors_multiple_authors_with_details`, lines ~812-856) still
call this dead method directly and assert on its output — they will pass
forever regardless of whether the real `render()`/`map_parameters()`
pipeline continues to serialize authors correctly, giving false coverage
confidence for exactly the bug class this phase set out to fix.
**Fix:** Delete `_format_authors_with_details()` and the two tests that
exercise it in isolation (the newer
`test_typst_authors_through_pipeline_produces_native_array_of_dicts` and
`test_typst_authors_unset_or_empty_yields_no_authors_key_on_package_path`
already cover the real, wired-up behavior end-to-end).

### WR-02: `docs/configuration.rst` documents the wrong default parameter mapping for `release`

**File:** `docs/configuration.rst:96-100`
**Issue:** The "Default Mapping" list states:

```
- ``project`` → template parameter ``title``
- ``author`` → template parameter ``authors``
- ``release`` → template parameter ``version``
```

but `typsphinx/template_engine.py`'s actual
`DEFAULT_PARAMETER_MAPPING = {"project": "title", "author": "authors",
"release": "date"}` maps `release` to `date`, not `version`. A user
following this reference and writing a custom template with a `version:`
parameter (as the doc's own `typst_template_mapping` example on
line 106-110 correctly implies is needed to *rename* `date` to `version`)
would be confused about what the *default* actually supplies. This predates
this phase's diff, but it sits inside the configuration reference this
phase's CONF-01 sweep specifically re-validated and partially edited
(the `typst_output_dir` removal is two sections below it), so it's a
visible gap in the same document.
**Fix:** Change `release`` → template parameter ``version``` to
`release`` → template parameter ``date``` (or fix the code if `version` was
actually the intended default — but the code is what ships).

### WR-03: `typst_authors` is a working, phase-relevant config value with no entry in the configuration reference

**File:** `docs/configuration.rst` (entire file); `tests/test_documentation_configuration.py:36-47`
**Issue:** `typst_authors` is registered in `typsphinx/__init__.py:57`, is
exactly the config value this phase's BUG-C fix makes actually work, and is
demonstrated in both `docs/source/examples/advanced.rst:42-48` and
`examples/charged-ieee/approach1/conf.py:37-44` — yet
`docs/configuration.rst`'s "Configuration Reference" never documents it (no
`typst_authors` heading anywhere in the file), and
`test_configuration_documents_all_config_values()`'s `required_configs` list
(`tests/test_documentation_configuration.py:36-47`) doesn't include it
either, so this gap is invisible to CI. A user reading only the reference
(rather than the advanced examples) has no way to discover this option.
**Fix:** Add a `typst_authors` subsection to `docs/configuration.rst`
(alongside `typst_template_function`) and add `"typst_authors"` to
`required_configs` in `test_configuration_documents_all_config_values()`.

### WR-04: Package/template routing suppression logic is duplicated (and only half-applied) between `writer.py` and `builder.py`

**File:** `typsphinx/builder.py:557-566`; `typsphinx/writer.py:179-190`
**Issue:** `writer.py:179` computes
`package_for_engine = None if raw_template_path else typst_package` and
passes that (correctly suppressed) value into the `TemplateEngine` used for
per-document rendering. `builder.py`'s `_write_template_file()` does **not**
apply the same suppression — it constructs its own `TemplateEngine` with
`typst_package=typst_package` (the raw, unsuppressed value) even in the
"both configured" case (`builder.py:562`). This is currently harmless
because that builder-side engine instance only ever calls
`get_template_content()`, which doesn't read `self.typst_package`. But the
whole point of this phase (per the phase intent) is that writer.py and
builder.py must not independently re-derive and disagree about
package/template routing — that class of two-places-diverge bug is
literally BUG-A. Today the divergence is inert; the next person who adds a
`render()` or `map_parameters()` call to `_write_template_file()` (e.g. to
support per-template default parameters) will silently reintroduce a
BUG-A-shaped defect because the suppression logic isn't centralized.
**Fix:** Extract the suppression rule (`None if raw_template_path else
typst_package`) into a single shared helper (e.g. a `staticmethod` on
`TypstWriter` or a free function both `builder.py` and `writer.py` import)
so there is exactly one place this decision is made, matching the
`@preview` version-sync convention already established elsewhere in this
codebase.

### IN-01: `examples/advanced/conf.py`'s custom-template comment contradicts the line immediately below it

**File:** `examples/advanced/conf.py:65-67`
**Issue:**

```python
# Custom template (optional)
# Uncomment to use a custom Typst template
typst_template = "_templates/custom.typ"
```

The comment tells the reader to "uncomment" the line to enable a custom
template, but the line is already active (not commented out) — so the
example ships with the custom template on by default, contradicting its own
instructions. This predates this phase's diff (not touched by it) and is
low-impact, but it sits in a file this phase edited (to remove the
`typst_output_dir` block) and is worth a one-line fix while in the
neighborhood.
**Fix:** Either comment out the `typst_template = ...` line to match the
comment, or update the comment to say the custom template is enabled by
default and how to disable it.
