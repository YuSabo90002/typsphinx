# Phase 22.2 Plan 04: Package-vs-Template Routing Repair Summary

**Fixed the fatal BUG-A defect where a Typst-Universe-package-alone project emitted a master document importing a shared `_template.typ` file the builder deliberately never writes, and made the package+template-both-configured combination explicitly unsupported (one warning, custom template wins) instead of silently broken.**

## Performance

- **Duration:** ~12 min
- **Started:** 2026-07-22T06:28:59+09:00 (base commit `8338cde`)
- **Completed:** 2026-07-22T06:36:29+09:00
- **Tasks:** 2
- **Files modified:** 2 (`typsphinx/writer.py`, `typsphinx/builder.py`), 1 file created (`tests/test_package_template_routing.py`)

## Accomplishments

- `TypstWriter.translate()` now computes the routing decision from the raw `typst_template`/`typst_package` config values at the top of the master-document branch: when a package is configured ALONE (no custom template), `template_file=None` is passed to `TemplateEngine.render()`, so the master emits its package import and show-rule but zero reference to a shared template file the builder never writes. Every other combination (default template, custom template alone, both configured) computes the import path exactly as before via the existing depth-based helper -- byte-identical for the two cases the corpus gate exercises.
- When a custom template IS configured (with or without a package also configured), `None` is now passed as the `typst_package` argument to the `TemplateEngine` constructor in `writer.py`, so the package import line is genuinely suppressed rather than merely deprioritised -- making "the package setting is ignored" literally true.
- `TypstBuilder._write_template_file()`'s early return narrows from "a package is configured" to "a package is configured AND no custom template is configured." When both are configured, execution now falls through and the custom template file IS written to the output directory (previously it was silently skipped, leaving the master's `#import` dangling).
- Immediately before that narrowed early return, a single `logger.warning()` (through the existing module-level Sphinx logger, no new logger created) fires when both `typst_package` and `typst_template` are configured, naming both config values verbatim, stating the combination is unsupported, and stating that `typst_template` is honoured. Because `_write_template_file()` runs exactly once per build (single call site in `prepare_writing()`), this warning fires once per build, not once per master document, matching D-03's acceptance criterion.
- `typst_template` plus `typst_template_function` alone (no package) -- the combination the bundled `charged-ieee/approach2` example and `docs/source/examples/advanced.rst` document -- is untouched: no warning path was added for it, verified by a dedicated non-regression test rather than by inspection.
- No handling that inspects, matches on, or rewrites Typst compiler error text was added anywhere in this diff (D-06): a grep for `stderr`/`stdout`/error-string inspection patterns over `writer.py` and `builder.py` returns 0 matches both before (base commit `8338cde`) and after this plan's changes.

## Files Created/Modified

- `typsphinx/writer.py` - `translate()` now reads `raw_template_path`/`typst_package` once, derives `package_for_engine` (None whenever a custom template is configured) for the `TemplateEngine` constructor, and derives `template_file` (None only for the package-alone case) for the `render()` call
- `typsphinx/builder.py` - `_write_template_file()` gains a both-set warning immediately before its early return, and the early return itself narrows to the package-alone predicate so a both-set build still writes the custom template file
- `tests/test_package_template_routing.py` - new module with three named tests, one per routing case (package alone / both configured / template alone), each building a tmp-path Sphinx project via `sys.executable -m sphinx -b typst` as a subprocess and asserting only on the emitted `.typ` SOURCE text and build output (no Typst compilation -- that is plan `22.2-05`'s job)

## Decisions Made

- Followed the plan's assumption-delta decision verbatim: `typst_template` is the primary route, `typst_package` a strictly weaker alternative that applies only in the template's absence. No new configuration surface was added.
- Kept the both-set warning exclusively in `builder.py`'s once-per-build method rather than duplicating it in `writer.py`'s per-document method, per the plan's explicit constraint that the warning must fire once per build regardless of master-document count.
- Left the `typst_package` argument passed to the `TemplateEngine` constructor inside `builder.py`'s `_write_template_file()` unchanged (not gated to `None` on the custom-template path) -- that call site only reaches `get_template_content()`/`load_template()`, neither of which reads `self.typst_package`, so gating it there would be a no-op; the plan's Task 1 instruction to pass `None` applied specifically to `writer.py`'s constructor call (which DOES reach `map_parameters()`/`render()`, where `self.typst_package` matters).
