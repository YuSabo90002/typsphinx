## WR-02 の代替契約

**提示した実測データ:** 文言 assert は todo が記載する 4 リテラルではなく実測 6 箇所。
分類語系（`"escape"`, `"not found"`）は Typst の言い回しそのもので最も脆く、パス名系
（`"_template.typ"`, `"usage.typ"` とその否定 2 件）は「Typst がエラーで問題のパスを示す」
という比較的安定した性質に乗っており、かつ G-22.1-2 の要求そのもの。

### Q1: 失敗クラスの証明を何に乗せ換えるか？

| Option | Description | Selected |
|--------|-------------|----------|
| ablation（欠けているものを置いて通す） | 文言ではなく「何を直すと通るか」でクラスを特定。tolerant にならず、上流が文言を変えても赤くならない。コスト：コンパイル回数が増える | ✓ |
| 分類語だけ落とし、パス名は残す | todo が「現実的な緩和」として挙げた案。変更最小で判別力を確実に保つが、WR-02 を半分しか閉じない | |
| ファイルシステム事実で代替 | `raises(Exception)` ＋ Path 検査で差異を示す。完全に文言非依存だが「そのファイルが原因で落ちた」因果は示せず、却下された tolerant 側に寄る | |
| typst-py を pin して文言は残す | 依存を上限 pin し文言 assert を 1 ヘルパーに集約。判別力は完全維持だが問題を先送りするだけで drift.yml と衝突 | |

**User's choice:** ablation（欠けているものを置いて通す）
**Notes:** → CONTEXT.md D-08。

### Q2（オーナーからの確認）: 「こういうのってやっぱ通すの確認しないといけないのかな」

**回答した内容:** ablation を選んだ以上、グリーン半分が証明の本体。「失敗した」だけでは
*なぜ*失敗したかを何も示せず、文言 assert を外した後に残るのは `pytest.raises(Exception)` だけ
——これはレビュアーが却下した tolerant assertion そのもの。「X を足したら通った」があって初めて
「失敗原因は X の不在だった」と言える。フィクスチャは極小なのでコンパイル追加実行の実行時間コストは
実質ゼロで、増えるのはテストコードの手間のみ。「通る」の水準には完全グリーンと「その失敗だけ消える」
の 2 段階があり、include 半分は完全グリーンが素直に作れるが template 半分（脱出エラー）は
「ファイルを置く」では消せない、と整理した。
**Notes:** → CONTEXT.md `<specifics>` に記録。

### Q3: template 半分にも ablation を徹底するか？

| Option | Description | Selected |
|--------|-------------|----------|
| include 半だけ ablation、template 半は統合 | include 半を完全グリーン化すれば「template を中立化したコピーは include が原因で落ちている」が証明され、バイト同一コピーの失敗は「中立化しても消えない別クラス」として差分で位置づけられる。root を上げる小細工が不要 | ✓ |
| template 半も root を上げて ablation | outdir の親に `_template.typ` を置き `root=parent(outdir)` でコンパイルし脱出エラーが消えることを直接示す。両半分が独立に自己完結するが、フィクスチャ外にファイルを置く段取りが入る | |
| 実装方針はプランナーに任せる | 契約だけロックし、テスト分割は research/plan で決める | |

**User's choice:** include 半だけ ablation、template 半は統合
**Notes:** → CONTEXT.md D-09。2 つのテストの最終的な分割・統合形は research/plan の裁量として残した。

### Q4: エラー文言への参照をどこまで残すか？

| Option | Description | Selected |
|--------|-------------|----------|

**Notes:** → CONTEXT.md D-10。

## 欠損 .typ の診断品質

**提示した実測データ:** `get_outdated_docs()` は毎回 `env.found_docs` 全件を返す
（`builder.py:269-279`）ため、増分ビルドで書き漏れる経路は存在しない。よって `.typ` 欠損は
実質「`typst_documents` の docname が `found_docs` に無い」ケースに絞られる。

### Q1: 欠損時のメッセージはどこまで診断するか？

| Option | Description | Selected |
|--------|-------------|----------|
| docname が未知かを判別して別文言 | `docname not in self.env.found_docs` なら「typst_documents の docname 'x' はこのプロジェクトに存在しない（タイプミス／`.rst` 拡張子付き／exclude）」と名指す。他の原因なら現行のパス提示にフォールバック | ✓ |
| パスを示すだけのまま | 現行文言を維持し failures に乗せ換えるだけ。WR-01 の文面に厳密で差分最小だが、ユーザーはパスから自力で原因推定 | |
| 候補提示までやる | `difflib` で found_docs から近い名前を提案。親切だが warning ハードニングという目的に対して機能追加寄り | |

**User's choice:** docname が未知かを判別して別文言
**Notes:** → CONTEXT.md D-04。候補提示（difflib）は `<deferred>` へ。

### Q2: 新しく落ちるようになる振る舞いをテストでどう固定するか？

| Option | Description | Selected |
|--------|-------------|----------|
| 専用フィクスチャを新設 | 22.1 D-06 / 22.2 D-09 と同じ GATE-01 形式で、存在しない docname と空エントリを含む最小フィクスチャを置き、`-b typstpdf` が非ゼロ終了し ExtensionError を出すことをテスト | ✓ |
| 既存テストにユニットで追加 | conf をモンキーパッチした軽量ユニットテスト。実 `sphinx-build` 経路を通らない分、回帰検知力が下がる | |
| プランナーに任せる | 「pre-fix で失敗するテストで固定する」という要件だけロック | |

**User's choice:** 専用フィクスチャを新設
**Notes:** → CONTEXT.md D-11。SC#5 のとおり pre-fix コードに対して失敗することが必須。
