#### WR-01 — 欠損マスターの扱い

- **D-01: 欠損 `.typ` は behavioral に解決する — `failures` に乗せてビルドを失敗させる。**
  `builder.py` の `logger.warning(...); continue` を `failures.append(...); continue` に
  変える。docstring を弱める documentary 案（コードは変えず「コンパイル失敗については」と限定）は
  却下 — 「PDF が一つも出ないのに exit 0」という実害が残るため。新設定値による opt-in 案も却下 —
  直前の Phase 22.2 で死に設定値を削った直後に新規設定値を増やすのは方針矛盾。

- **D-02: 既存の `failures` リストに同居させ、ループ後に一括集約する。**
  22.1 D-05 の attempt-all-then-raise 契約をそのまま維持する。fail-fast（その場で raise）は
  22.1 D-05 と非対称になるため却下。`missing` と `failures` を別リストに分けて区別表示する案も
  却下 — `finish()` の後半が不必要に複雑化する。**ただし集約メッセージの末尾文言は要調整:**
  現行は `f"typstpdf: {len(failures)} master document(s) failed to compile to PDF: ..."` と
  「compile」に限定されているので、コンパイル以前の失敗も含む形（例「failed」）へ広げる必要がある。
  *(Research note: the CONTEXT.md text quoted this as `f"typst_pdf: ..."` with an underscore; the
  real code at `builder.py:946` reads `"typstpdf:"` — no underscore. Use the real string as the
  edit basis.)*

- **D-03: 移行緩和は置かない — clean break ＋ CHANGELOG。**
  互換シムも deprecation 期間も設けない。22.1 D-04/D-05a および 22 D-08 の前例に従い、これは
  docstring が既に約束していた振る舞いへのバグ修正として扱う。**Phase 23 への引き継ぎ義務:**
  この user-visible な振る舞い変更は `[0.6.2]` CHANGELOG エントリに明記すること（ROADMAP SC#3）。
  22.1 D-05a が既に「compile 失敗がビルド失敗になる」を同エントリに載せる前提なので、同じ項目に同居させられる。

- **D-04: 欠損時のメッセージは「docname が未知か」を判別して文言を分ける。**
  `docname not in self.env.found_docs` のとき、`typst_documents` の docname がプロジェクトに
  存在しない旨を名指しする（タイプミス／`.rst` 拡張子付きで書いた／`exclude_patterns` で除外、が
  圧倒的多数の原因）。それ以外の原因なら現行のパス提示にフォールバックする。
  **根拠（実測）:** `get_outdated_docs()` は毎回 `env.found_docs` 全件を返す（`builder.py:269-279`）
  ので、増分ビルドで書き漏れる経路は存在しない。よって欠損は実質「設定された docname が
  `found_docs` に無い」ケースに絞られる。`difflib` による候補提示（"did you mean ...?"）は
  warning ハードニングというフェーズ目的に対して機能追加寄りなので却下。

#### WR-01 — malformed `doc_tuple` の扱い

- **D-05: malformed 分岐も欠損 `.typ` と同じ扱いにする — `failures` に乗せる。**
  ROADMAP SC#2 が求める「2 つのスキップ分岐を非対称のまま放置しない」を最も直接に満たす。
  空エントリは常に設定ミスであり、意図した PDF が 1 つ出ない点で欠損 `.typ` と同質。

- **D-06: ガードの検知範囲は現行の `if not doc_tuple:` のまま広げない。**
  WR-01 が指摘したのは「スキップが `failures` に入らない」ことであって、検知できていない入力が
  あることではない。変更は `logger.warning + continue` → `failures.append + continue` の
  乗せ換えに留め、フェーズの診断差分を最小に保つ。
  **既知の未対応（今回は入れない）:** `(123, ...)` のような非文字列 docname は `path.join` に
  到達して `try` の**外側**で `TypeError` を投げ、集約 `ExtensionError` に乗らず生の traceback で
  落ちる。これは別種の欠陥なので `<deferred>` に記録する。
  *(Research note: empirically reproduced in this session — see Common Pitfalls #4. The real crash
  site is `builder.py:919` calling `_directory_preserving_relpath`, which raises at `builder.py:264`
  — `path.dirname(docname)` on a non-str. Confirmed OUTSIDE the try block, which starts at
  `builder.py:926`.)*

- **D-07: malformed エントリの識別子はエントリの値そのもの（`repr`）を使う。**
  `failures: List[Tuple[str, str]]` の第 1 要素はこれまで常に docname だが、空エントリには
  docname が無い。`failures.append((repr(doc_tuple), "..."))` として集約メッセージに
  `(): malformed typst_documents entry` の形で出す。現行 warning のメッセージと見た目が揃い、
  実装も 1 行で済む。`enumerate` を導入してインデックス表記にする案は、得られる一意性に対して
  ループ構造の変更が割に合わないため却下。

#### WR-02 — テストの上流エラー文言からの切り離し

- **D-08: 失敗クラスの証明を ablation（欠けているものを置いたら通る）に乗せ換える。**
  「文言が何と書いてあるか」ではなく「何を直すと通るか」でクラスを特定する。上流が文言を変えても
  赤くならず、かつ `pytest.raises(Exception)` だけの tolerant assertion（レビュアーが却下済み）に
  逆戻りしない。フィルタとしての判別力を保ったまま WR-02 を閉じられる唯一の案。
  **前提の明示:** ablation は「通る」側の確認が証明の本体である。赤だけでは赤の理由を何も
  示せないため、グリーン半分を省略してはならない。フィクスチャは極小なのでコンパイル追加実行の
  実行時間コストは無視できる。
  *(Research note: empirically strengthened in this session — `typst.compile()`'s escape-error and
  missing-file-error BOTH raise the exact same `typst.TypstError(RuntimeError)` with no
  distinguishing subtype or attribute (see Common Pitfalls #2). Exception-type-based
  discrimination is not just undesirable, it is IMPOSSIBLE here. The ablation is the only
  remaining discriminator once message-text is banned.)*

- **D-09: include 半分を完全グリーンの ablation にし、template 半分はその差分として位置づける。**
  `test_sibling_include_fails_at_outdir_root_and_resolves_in_place` 側で、`usage.typ` /
  `logo.png` を outdir root に置けば root コピーが通ることを示し、「失敗原因は sibling include /
  upward image の解決だった」を文言なしで確定させる。これが確定すれば、
  `test_outdir_root_compile_basis_still_fails` のバイト同一コピーの失敗は「template を中立化しても
  消えない別クラス」として差分で位置づけられる。**却下:** template 半分にも独立した ablation を
  組む案（outdir の親に `_template.typ` を置き `root=parent(outdir)` でコンパイルして脱出エラーを
  消す）— 成立はするがフィクスチャ外にファイルを置く段取りが入る。2 つのテストをどう分割・統合するかの
  最終形は research/plan の裁量。
  *(Research note: empirically verified — see Common Pitfalls #3. Placing `usage.typ` + `logo.png`
  at the outdir root is NECESSARY but NOT SUFFICIENT. The `image("../logo.png")` reference text
  itself must ALSO be rewritten to `image("logo.png")`, because Typst's root-boundary enforcement
  rejects ANY `".."` segment that resolves above the compile root — and once the compiled file
  itself sits AT the root, `".."` always points outside it, regardless of what files exist there.
  This is an addition to D-09's literal wording, not a contradiction of it.)*

- **D-10: typst のエラー文言への assert はゼロにする。**
  実測 6 箇所（`"escape"`, `"_template.typ"`, `"usage.typ"`, `"not found"`,
  `"_template.typ" not in`, `"api/usage.typ" not in`）を全廃する。エラー文字列は失敗時の診断
  メッセージ（`f"...: {error_text!r}"`）にのみ埋め込み、人間が読む情報としては残すが CI の
  赤黒判定には一切影響させない。分類語だけ落としてパス名 assert を残す部分対応は却下 — WR-02 を
  半分しか閉じない。typst-py の版 pin ＋文言維持も却下 — 問題を先送りするだけで、`drift.yml`
  （毎週の最新依存再解決）と衝突する。
  *(Research note: the 6 literals were re-located and confirmed at their real current line numbers:
  `test_nested_master_render_gate.py:366, 376, 504, 509, 514, 520` — see Common Pitfalls #1 for the
  full table. D-10's ban is scoped to typst-py's OWN diagnostic text; asserting on typsphinx's own
  authored strings — e.g. the new `ExtensionError` message, or the emitted `.typ` source text — is
  unaffected and remains normal practice throughout this test suite.)*

## Summary

Both warnings are narrower and more mechanically tractable than the WR-01/WR-02 prose alone
suggests, but this session's direct reading of `typsphinx/builder.py` found that **every line
number CONTEXT.md cites for the two `finish()` skip branches has drifted** from the real file —
by roughly 20-27 lines, most likely from edits made during Phase 22.2 earlier in the same file.
The actual coordinates (re-verified by `grep -n` and a fresh `Read`, not computed) are:
`finish()` docstring **868-886** (not 850-856), the malformed-`doc_tuple` guard **912-916** (not
885-890), and the missing-`.typ` guard **922-924** (not 895-897). The *shape* of both branches —
`logger.warning(...); continue`, nothing appended to `failures` — is exactly as CONTEXT.md
describes; only the line numbers moved. The plan must cite the corrected coordinates.

WR-01's fix is a small, surgical change: both skip branches gain a `failures.append((identifier,
message))` call before their existing `continue`, reusing the `List[Tuple[str, str]]` mechanism
Phase 22.1 already built (`builder.py:902, 943-948`). D-04's discrimination logic
(`docname not in self.env.found_docs` → a "not a known Sphinx document" message, else the current
path-based message) was live-verified in this session: `env.found_docs` is populated at
`SphinxTestApp` construction time (before `.build()` is even called), and a `TypstPDFBuilder`
constructed directly and driven through `finish()` — exactly the shape every existing unit test in
`tests/test_pdf_generation.py` already uses — reproduces the present-day silent-success bug live
(`returncode`-equivalent success, zero PDFs, `"ghost" not in found_docs` confirmed True). No
existing test in the suite constructs a `typst_documents` entry that hits either skip branch during
a call to `finish()` that is expected to succeed, so the blast radius of turning both branches into
failures is empirically zero against the current suite (560 passed / 23 slow-deselected baseline
recorded this session).

WR-02's fix is conceptually simple (replace 6 literal-text assertions with structural/type/artifact
assertions) but the underlying reason D-08 chose "ablation over exception type" is stronger than
CONTEXT.md states: this session confirmed by direct execution that Typst's root-escape error and
its missing-file error are BOTH raised as the exact same `typst.TypstError(RuntimeError)` class,
with no discriminating attribute — so type-based discrimination is not merely undesirable, it is
structurally impossible. The ablation is the only remaining lever. A concrete, non-obvious gap in
D-09's stated design was also found and closed by direct experiment: placing `usage.typ` and
`logo.png` at the outdir root is necessary but **not sufficient** to make the neutralized copy
compile — the `image("../logo.png")` reference text itself must ALSO be rewritten to
`image("logo.png")`, because Typst's root-boundary check rejects any `".."` segment once the
compiled file sits at the root itself. Three separate real `typst.compile()` runs in this session
prove this precisely (Common Pitfalls #2/#3).

**Primary recommendation:** For WR-01, add `failures.append(...)` at both skip sites using the
corrected line numbers, add the `found_docs`-based message split at the missing-`.typ` site, use
`repr(doc_tuple)` as the malformed entry's identifier, broaden the aggregate message from "failed
to compile to PDF" to "failed", and pin the new behavior with one new GATE-01 fixture (real
`sphinx-build -b typstpdf`, non-zero exit, `ExtensionError` in stderr, assert on typsphinx's OWN
message text — never typst-py's). For WR-02, keep the two existing render-gate tests as two
functions (per D-09's explicit choice), strip the 6 literal `typst-py` text assertions, and extend
`test_sibling_include_fails_at_outdir_root_and_resolves_in_place` with the full ablation
(neutralize BOTH the template import AND the image path, place `usage.typ` + `logo.png` at the
outdir root, assert a real PDF results) while keeping `test_outdir_root_compile_basis_still_fails`
as a structural/differential RED-only proof (no literal text, but a precondition assertion on
typsphinx's OWN emitted source, which is unaffected by D-10).

## Architectural Responsibility Map

This project has no browser/frontend/API/CDN tiers — it is a Sphinx builder extension (a Python
library invoked as a `sphinx-build` process) plus its pytest test suite. The standard web-tier
table is adapted to this project's actual architecture:

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Master-failure detection & aggregation (WR-01) | Sphinx Builder (`TypstPDFBuilder.finish()`, in-process Python) | — | This is pure build-orchestration logic inside the builder; it owns the `typst_documents` loop, the `failures` list, and the final `raise ExtensionError` |
| Docname validity check (`found_docs` membership, D-04) | Sphinx Builder (reads `self.env.found_docs`, populated by Sphinx core during the read phase) | Sphinx Environment (`BuildEnvironment`, upstream-owned) | `env.found_docs` is Sphinx's own environment state; the builder only reads it, never writes it — a boundary that must not be crossed (e.g., the builder must not try to "fix" `found_docs`) |
| Real Typst compilation (`.typ` → PDF bytes) | External Process Boundary (`typst-py` / Rust binary via PyO3, `typsphinx/pdf.py`) | Sphinx Builder (catches `Exception`, wraps as `TypstCompilationError`/aggregates) | The builder's `try/except` around `compile_typst_file_to_pdf()` is the ONLY place typsphinx's code and the external Typst compiler's error surface meet; WR-02's whole premise is that this boundary's error TEXT is not typsphinx's to contract |
| Real-compile regression proof (WR-02 ablation) | Test Suite (pytest, `tests/test_nested_master_render_gate.py`) | External Process Boundary (drives real `typst.compile()` / `sphinx-build` subprocess) | The gate's job is to prove a structural property (which reference class breaks, which fix restores it) using the SAME external compiler as production, never a mock — this is why the ablation must be a REAL compile, not a stubbed one |
| New GATE-01 fixture for WR-01 (D-11) | Test Suite (new `tests/fixtures/<name>/` + `tests/test_<name>_gate.py`) | Sphinx Builder (drives the real `finish()` failure path via subprocess `sys.executable -m sphinx`) | Must reproduce the exact CLI invocation shape (never `uv run sphinx-build`, per the NixOS-sandbox PATH-shadowing note already documented in three sibling gate modules) |

### Pattern 1: GATE-01 real-compile fixture (established convention, reuse verbatim for D-11)

Surveyed all three sibling fixtures (`nested_master_render_gate/`, `template_named_dir_master/`,
`package_only_config_gate/`) and their driving test modules. All three share this exact shape:

```python
# Source: tests/test_package_only_config_gate.py / test_nested_master_render_gate.py
```

### Pattern 2: Ablation-based failure-class proof (WR-02, D-08/D-09) — recommended final shape

Recommended split (honoring D-09's explicit choice to keep two test functions, informed by this
session's three real `typst.compile()` experiments):

**`test_sibling_include_fails_at_outdir_root_and_resolves_in_place`** (include/image class — full

2. Structural preconditions on the emitted `.typ` — these check typsphinx's OWN emitted source
   text, not typst-py's diagnostic text, so they are unaffected by D-10 (existing Step 2, keep
   verbatim).
3. Neutralize the template import ONLY (existing Step 3, keep verbatim: `'#import
   "../_template.typ"'` → `'#import "_template.typ"'`).
4. **RED** (existing, strip literal-text asserts): `pytest.raises(Exception)` on the
   template-only-neutralized copy at the outdir root. Replace the 4 literal asserts
   (`"usage.typ" in error_text`, `"not found" in lowered`, `"_template.typ" not in error_text`,

   `not (temp_build_dir_typst / "usage.typ").exists()` BEFORE the compile call — this proves, from
   the filesystem (not the compiler's text), that a failure at this exact reference is the

5. **NEW — full ablation** (closes the D-09 gap found in this session): additionally rewrite
   `'image("../logo.png")'` → `'image("logo.png")'` in a SECOND, further-neutralized copy; copy

   outdir root; `typst.compile()` this fully-neutralized copy; assert it succeeds and the returned
   bytes start with `b"%PDF"`. This is the genuine "what fixes it" proof D-08 requires — proven by
   this session's Experiment C (see Common Pitfalls #3).
6. Keep the existing Step 5 (compile the ORIGINAL, unmodified master at its REAL location,
   `outdir/api/index.typ`) as an additional, complementary green proof — it demonstrates the fix
   (compile-in-place) resolves the identical references with ZERO text rewriting, which the new
   ablation (step 5 above) does NOT by itself demonstrate (the ablation proves "what class of
   reference broke", not "the shipped fix resolves it").

**`test_outdir_root_compile_basis_still_fails`** (template-reference class — RED-only differential,
D-09's explicit choice): keep the raw, byte-identical copy at the outdir root; replace the 2 literal
asserts (`"escape" in lowered`, `"_template.typ" in error_text`) with `pytest.raises(Exception)`
only, relying on the ALREADY-PRESENT structural precondition (`assert '#import
"../_template.typ"' in typ_source_text`, which checks typsphinx's own emitted source, not typst-py's
error) to establish which reference is under test. Its status as "the differential" is established
by comparison with the sibling test above: once that test's ablation proves the template import
alone (when neutralized) lets compilation proceed to a DIFFERENT, independently-explicable failure
(missing `usage.typ`, proven via filesystem, not text), this test's failure — which occurs
precisely when that SAME import is left unmodified — is attributable to it by elimination, without
either test needing to read typst-py's message.

### Anti-Patterns to Avoid
- **Discriminating failure class by exception subtype:** `typst.TypstError` is a single flat class
  (`RuntimeError` subclass, no subtypes, no error-code attribute) for every Typst compile failure —
  confirmed empirically. Do not attempt `except SomeSpecificTypstError` style discrimination; it
  does not exist in this API.
- **Widening the malformed-`doc_tuple` guard "while we're in there":** D-06 explicitly keeps the
  guard's detection surface (`if not doc_tuple:`) unchanged. Do not add non-str-docname detection
  in this phase — it is a separate, deferred defect (Common Pitfalls #4).
- **Asserting on Sphinx's own CLI traceback formatting** (`"Extension error!"`, the "Versions" /
  "Loaded Extensions" banner text): this is Sphinx-core's own uncontracted CLI presentation, a
  different but analogous coupling risk to the one WR-02 is fixing. Prefer asserting on
  `result.returncode != 0` plus a substring of typsphinx's OWN `ExtensionError` message.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Aggregating multiple failure reasons across a loop | A new failure-tracking structure/class | The existing `failures: List[Tuple[str, str]]` (`builder.py:902`) | Already built in Phase 22.1 (D-05); WR-01's two new failure kinds are `append()`-compatible with it as-is — no new data structure needed |
| Determining whether a docname is a real Sphinx document | Scanning `srcdir` for a matching `.rst`/`.txt` file, or re-deriving suffix rules | `self.env.found_docs` (a `set[str]`, populated by Sphinx core) | Sphinx already computes and maintains this; re-deriving it independently risks drifting from Sphinx's own `exclude_patterns`/suffix/`source_suffix` handling |
| Proving a specific compile-failure class without reading error text | Regex-matching error message shapes, or pinning `typst-py`'s version to freeze wording | The ablation pattern (D-08): fix the suspected cause, show a DIFFERENT-or-no failure results | This is the whole point of WR-02; anything that reads typst-py's message text for CI pass/fail reintroduces the exact coupling being removed |

**Key insight:** Both warnings are fixed by *reusing* mechanisms Phase 22.1 already built
(`failures` list, attempt-all-then-raise, GATE-01 fixture convention) rather than introducing new
abstractions — consistent with D-01's explicit rejection of a new opt-in config value and D-06's
explicit rejection of widening the malformed-entry guard.

## Common Pitfalls

### Pitfall 1: CONTEXT.md's cited line numbers for both `finish()` skip branches have drifted

**What goes wrong:** Editing at the coordinates CONTEXT.md cites (`885-890` for malformed,
`895-897` for missing-`.typ`, `850-856` for the docstring) edits the WRONG code — those lines are
currently the `if not typst_documents:` whole-project-empty early return (`894-898`) and its
preceding blank/comment lines, not the per-entry skip branches.

**Why it happens:** Line numbers drift whenever earlier code in the same file changes (most likely
Phase 22.2's edits to config-handling code earlier in `builder.py`). CONTEXT.md's coordinates were
accurate at authoring time but the tree has moved since.

**How to avoid:** Use these RE-VERIFIED coordinates (confirmed via `grep -n` AND a fresh `Read` in
this session, cross-checked against each other):
- `finish()` def: line **867**
- `finish()` docstring (containing D-04's "no silent success" prose): lines **868-886**
- Malformed `doc_tuple` guard (`if not doc_tuple: logger.warning(...); continue`): lines **912-916**
- `docname = doc_tuple[0]` through `typ_file = path.join(...)`: lines **917-920**
- Missing-`.typ` guard (`if not path.exists(typ_file): logger.warning(...); continue`): lines
  **922-924**
- `try:` block (compile, write, log): lines **926-941**
- `except Exception as e:` (existing compile-failure path, unchanged): lines **939-941**
- Aggregate `if failures: raise ExtensionError(...)`: lines **943-948**

**Warning signs:** If a diff touches a line containing `"No documents defined in typst_documents"`
when the intent was to fix the malformed-entry or missing-`.typ` branch, the coordinates were not
re-verified before editing.

### Pitfall 2: `typst.TypstError` has no discriminating subtype between failure classes

**What goes wrong:** An implementation might try to replace the banned literal-text assertions with
`except SomeSpecificError` / `isinstance(exc, ...)` style discrimination between the
root-escape-class failure (template import) and the missing-file-class failure (sibling include).

**Why it happens:** It's a reasonable first instinct — "if we can't check the message, check the
type" — but this API doesn't support it.

**How to avoid:** Confirmed empirically in this session (`typst` 0.15.0):
```python
import typst
# escape-class:
typst.compile(str(escaping_file), root=str(root))    # raises typst.TypstError
# missing-file-class:
typst.compile(str(missing_ref_file), root=str(root))  # raises typst.TypstError  -- SAME class
```
Both raise `typst.TypstError`, MRO `(TypstError, RuntimeError, Exception, BaseException, object)`,
with no error-code attribute, no distinguishing `args` shape — `dir(typst.TypstError)` shows only
the standard `Exception`/`BaseException` surface. This is precisely why D-08's ablation approach is
not just the safer choice but the ONLY choice once literal-text assertions are banned (D-10).

**Warning signs:** A plan or PLAN.md task that proposes "catch the specific error type" for this
gate should be flagged for revision toward the ablation pattern instead.

### Pitfall 3: Placing `usage.typ` + `logo.png` at the outdir root is necessary but NOT sufficient for the D-09 ablation

**What goes wrong:** D-09's literal wording ("usage.typ / logo.png を outdir root に置けば root
コピーが通ることを示し") could be read as "just copy the two files there and it will compile." It
will not — the compile still fails, now on the image reference.

**Why it happens:** The emitted `image("../logo.png")` reference is written relative to the
master's real directory (`api/`), where `".."` legitimately climbs to the outdir root (which is
INSIDE the compile root). Once the compiled file itself is relocated to the outdir root, the exact
same `".."` text now tries to climb to outdir's PARENT — which is OUTSIDE the compile root — and
Typst's root-boundary enforcement rejects it with the SAME `"would escape the project root"` class
of error the template import triggers.

**How to avoid — verified empirically in this session, three real `typst.compile()` runs:**

```python
# Experiment A (baseline): template import neutralized only, nothing copied to root
# -> FAILS: 'file not found (searched at <outdir>/usage.typ)'  (matches the CURRENT test)

# Experiment B: usage.typ copied to root too (logo.png already there from the -b typst build)
# -> STILL FAILS: 'path "../logo.png" would escape the project root'
#    (proves files-alone is NOT sufficient)

# Experiment C: usage.typ copied to root AND image("../logo.png") rewritten to image("logo.png")
# -> SUCCEEDS: real PDF bytes, b'%PDF-1.7...'
#    (proves the minimal SUFFICIENT ablation requires the path-text rewrite too)
```
The full ablation needs: (1) template import rewritten, (2) image path text rewritten
(`"../logo.png"` → `"logo.png"`), (3) `usage.typ` copied to the outdir root (`logo.png` is already
there, copied by the normal `-b typst` build's asset-copy step). Only after all three does the
compile succeed.

**Warning signs:** If the new ablation test's GREEN half doesn't rewrite the `image(...)` line, it
will fail with an "escape" error that looks superficially similar to the template-class failure —
easy to mis-diagnose as "the ablation didn't work" rather than "the ablation is incomplete."

### Pitfall 4: A non-string docname in `typst_documents` raises a raw `TypeError` OUTSIDE the try block (confirmed, deferred by D-06 — do not fix in this phase)

**What goes wrong:** `typst_documents = [(123, "manual.typ", "T", "A")]` crashes `finish()` with an
uncaught `TypeError`, bypassing the `ExtensionError` aggregation entirely.

**Why it happens (confirmed by direct execution in this session):**
```
Traceback (most recent call last):
  File ".../typsphinx/builder.py", line 919, in finish
    relative_path = self._directory_preserving_relpath(docname, stem)
  File ".../typsphinx/builder.py", line 264, in _directory_preserving_relpath
    directory = path.dirname(docname)
TypeError: expected str, bytes or os.PathLike object, not int
```
`_resolve_output_stem(docname)` itself does NOT raise for a non-str docname (the entry-matching
loop's `entry[0] == docname` comparison and the eventual string-typed `target`/`stem` handling never
touch `docname`'s type directly) — the crash is specifically in `_directory_preserving_relpath`
(`builder.py:239-267`), at the `path.dirname(docname)` call (`builder.py:264`), called from
`finish()` at `builder.py:919`, which is BEFORE the `try:` block starts at `builder.py:926`.

**How to avoid:** Do not fix this in Phase 22.3 — D-06 explicitly defers it as a distinct defect
class (input-type validation), out of scope for "warning hardening." Record it as a follow-up todo
if not already tracked (CONTEXT.md's `<deferred>` section already does this).

**Warning signs:** None needed for THIS phase — this is a "confirm it's really out of scope, don't
accidentally fix it as a drive-by" pitfall, not a "must fix" one.

### Pitfall 5: `tests/test_target_name_render_gate.py:122` asserts on a substring that WOULD change if the missing-`.typ` message wording changes

**What goes wrong:** `test_target_name_render_gate.py:122` asserts
`"Master document not found" not in result.stderr` as a regression guard against a DIFFERENT bug
(write-path/read-back-path stem disagreement, Issue #117-era). It currently passes because that
fixture's build always succeeds (the master's `.typ` is always found). This is not literal-text
coupling to typst-py (D-10 doesn't apply — this is typsphinx's OWN message), but it IS a test that
embeds the CURRENT wording of the missing-`.typ` warning.

**Why it happens:** The message this test guards against is exactly the fallback-path string D-04
says stays unchanged for the "other cause" branch — so this specific assertion is expected to
survive the phase unmodified. But if a future edit changes that fallback wording too (not just adds
the found_docs-based alternate message), this assertion would silently stop testing anything
meaningful (it would never see the string again, in success OR failure).

**How to avoid:** When implementing D-04, keep the "other cause" fallback message's text as
`"Master document not found: {typ_file}"` verbatim (matches CONTEXT.md's own framing: "それ以外の
原因なら現行のパス提示にフォールバックする" — "fall back to the CURRENT path-based presentation").
Only the found_docs-known-missing branch gets NEW wording; the fallback string must not change,
precisely so this pre-existing negative assertion keeps meaning what it currently means.

**Warning signs:** If `test_target_name_render_gate.py` needs an edit as part of this phase's diff,
that is a signal the fallback message wording changed when D-04 said it should not.

## Security Domain

`security_enforcement` is enabled in `.planning/config.json` (ASVS level 1), but — matching the
precedent set by every prior v0.6.2 phase's RESEARCH.md — this project is a local Sphinx
extension / CLI build tool with no network service, no authentication surface, and no user-facing
HTTP input; the classic OWASP-ASVS web categories (V2/V3/V4) do not apply. This phase specifically
touches only: (a) build-time control flow inside `finish()` (which failures get reported and how),
and (b) test assertions (which do not ship to users at all). Neither introduces a new input surface
or a new trust boundary — `conf.py` remains fully-trusted, arbitrary-code-execution Python before
typsphinx ever runs, unchanged by this phase.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | No | No auth surface — local build tool |
| V3 Session Management | No | No sessions — single-process CLI build |
| V4 Access Control | No | No multi-user access boundary |
| V5 Input Validation | Marginal-yes | `typst_documents` entries are developer-authored `conf.py` values; this phase makes MORE of their malformed shapes surface loudly (as a build failure) rather than silently — a net IMPROVEMENT to input-validation posture, not a new input path. The one gap this phase deliberately does NOT close (non-str docname → raw `TypeError`) is explicitly deferred, not silently ignored (Pitfall #4, D-06) |
| V6 Cryptography | No | No cryptographic operations in this phase |
| V7 Error Handling and Logging | Yes | This phase's entire purpose is tightening error-handling behavior in `finish()` — converting silent `logger.warning` to an aggregated, build-failing `ExtensionError`. Standard control: fail loudly on a broken build artifact rather than degrading silently, which this phase directly implements |

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| A CI pipeline silently produces zero PDFs while reporting success (the WR-01 bug itself) | Availability / Denial-of-service-by-omission (a downstream consumer of the PDF artifact never gets it, with no signal) | Addressed by this phase directly: `failures.append(...)` + aggregated `ExtensionError` converts a silent no-op into a loud, non-zero-exit build failure |
| A test suite silently stops testing a real property because an unrelated upstream dependency changed its error wording (the WR-02 bug itself) | Tampering (of test-suite integrity, not data) / a false sense of security from a green CI that no longer proves anything | Addressed by this phase directly: the ablation pattern proves the property structurally, independent of upstream wording |

No new threats are introduced by this phase; both threat patterns addressed above are the SAME
underlying class already named in this phase's own WR-01/WR-02 requirement descriptions — this
phase's outcome is the mitigation, not a new risk.
