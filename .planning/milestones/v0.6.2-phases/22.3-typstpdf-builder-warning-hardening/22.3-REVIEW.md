# Phase 22.3: Code Review Report

**Reviewed:** 2026-07-22T00:00:00Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

`TypstPDFBuilder.finish()`'s widened failure handling (WR-01) is internally
correct: I traced every `continue` in the loop and confirmed both the
malformed-entry branch and the missing-`.typ` branch now append to
`failures` before continuing, the `found_docs` discrimination uses the
right attribute/membership semantics, the aggregate message renders
strictly in `typst_documents` iteration order, and no early return or
uncaught path short-circuits the attempt-all-then-raise loop for the cases
the loop itself can reach. The new unit tests in `test_pdf_generation.py`
correctly avoid depending on typst-py being reachable when a test's
predicted code path never reaches `compile_typst_file_to_pdf()`.

However, standard-depth review of the actual control flow this fixture
drives (not just the file that was patched) surfaced one BLOCKER: a
pre-existing, unguarded `doc_tuple[0]` in `typsphinx/writer.py`'s
`_is_master_document()` raises an uncaught `IndexError` for a malformed
(empty-tuple) `typst_documents` entry as soon as the build writes ANY
document whose docname does not appear earlier in `typst_documents` than
the malformed entry. I reproduced this live against a two-document project
and confirmed the entire `sphinx-build` process aborts with a raw traceback
during the *write* phase -- before `TypstPDFBuilder.finish()` (the very
code WR-01 hardened) ever runs, so **no PDF is produced for any master,
including an otherwise-valid one**. The new `missing_and_malformed_master_gate`
fixture is a single-document project with the valid master listed *first*
in `typst_documents`, which incidentally short-circuits `_is_master_document`'s
loop before it ever reaches the malformed `()` entry -- so both new gate
tests pass, but only by accident of fixture shape, not because the
attempt-all-then-raise contract actually holds for a realistic
(multi-document) project with a malformed `typst_documents` entry.

I also found one residual, incomplete piece of the WR-02 cleanup:
`test_nested_master_render_gate.py`'s pre-existing `PASS`-path test still
asserts a negative on typst-py's own diagnostic wording (`"file not found"
not in result.stderr`), the exact defect class the rest of that file's
WR-02 rewrite eliminated everywhere else.

## Critical Issues

### CR-01: Malformed `typst_documents` entry crashes the write phase (uncaught `IndexError`) for any realistic multi-document project, defeating WR-01's attempt-all-then-raise contract

**File:** `typsphinx/writer.py:55-57` (root cause), interacting with `typsphinx/builder.py:906-959` (`TypstPDFBuilder.finish()`) and the new fixture `tests/fixtures/missing_and_malformed_master_gate/conf.py` / `tests/test_missing_and_malformed_master_gate.py`

**Issue:** `TypstWriter._is_master_document()` is:

```python
for doc_tuple in typst_documents:
    if doc_tuple[0] == docname:
        return True
return False
```

This has no guard against a falsy/empty `typst_documents` entry, unlike
every other consumer of `typst_documents` in `typsphinx/builder.py`
(`_compute_master_included_docnames` guards with `if entry`,
`_resolve_output_stem` guards with `if entry and len(entry) >= 2`, and
`TypstPDFBuilder.finish()` itself now guards with `if not doc_tuple:` --
exactly the WR-01/D-07 defense this phase added). `_is_master_document()`
is called from `TypstWriter.translate()`, which runs during `write_doc()`
for **every** document in the project, not just masters -- i.e. it runs
long before `finish()`'s new, carefully-hardened failure handling ever
gets a chance to execute.

Given `typst_documents = [("index", ...), ("ghost", ...), ()]` (the
gate's fixture), `_is_master_document("index")` matches on the *first*
loop iteration and returns before ever touching the malformed `()` entry
-- pure luck of ordering. But for *any* docname that is not itself listed
before the malformed entry (i.e. any second real page, a `genindex`, a
`search` page, or simply "ghost" checked before `index` if the list were
reordered), the loop reaches `doc_tuple[0]` on `()` and raises
`IndexError: tuple index out of range`, which Sphinx's CLI wraps as a
fatal, uncaught build error. I reproduced this live:

```
$ sphinx-build -b typstpdf <src-with-2-docs-and-a-malformed-entry> <out>
...
writing output... [other]
Traceback (most recent call last):
  File ".../typsphinx/writer.py", line 56, in _is_master_document
    if doc_tuple[0] == docname:
IndexError: tuple index out of range
$ echo $?
2
```

The build does exit non-zero (`2`), but only `index.typ` was written --
`index.pdf` was **never produced**, because the crash happens during the
write phase, before `TypstPDFBuilder.finish()` (where WR-01's
attempt-all-then-raise / "the good master still gets its PDF" guarantee
lives) is ever reached. This directly contradicts the phase's own stated
contract ("Every configured master is attempted, even if an earlier one
fails... masters that compile successfully still get their .pdf
written", `typsphinx/builder.py:877-886`) for any project with more than
one document -- which is the overwhelmingly common case, not an edge
case.

The new `missing_and_malformed_master_gate` fixture cannot detect this: it
has exactly one document ("index"), and that document happens to be
listed *before* both bad entries in `typst_documents`, so
`_is_master_document` never reaches the malformed entry. Both new gate
tests (`test_mixed_bad_entries_fail_build_but_good_master_still_compiles`,
`test_aggregate_message_discriminates_unknown_docname_from_missing_file`)
therefore pass today, but they do not actually prove the "good master
survives a bad sibling" claim for any project realistic enough to have a
second page.

**Fix:** Guard `_is_master_document()` the same way
`_resolve_output_stem()` already does, e.g.:

```python
for doc_tuple in typst_documents:
    if doc_tuple and doc_tuple[0] == docname:
        return True
return False
```

Then extend `tests/fixtures/missing_and_malformed_master_gate/` with a
second, non-master document (e.g. a `chapter1.rst` referenced from
`index.rst`'s toctree, not listed in `typst_documents`) so the gate
actually exercises `_is_master_document()` against the malformed entry --
without this, re-introducing the crash in a future refactor would not
turn any test in this suite red.

## Warnings

### WR-01: Residual assertion on typst-py's own diagnostic wording in a PASS-path test, incomplete WR-02 cleanup

**File:** `tests/test_nested_master_render_gate.py:265-268`

**Issue:** The module docstring and every RED-half test in this file were
rewritten (this phase, per the WR-02/D-10 rationale documented at the top
of the file) specifically to stop asserting on typst-py's own diagnostic
text, replacing those assertions with structural/filesystem preconditions
and an ablation. One assertion was missed:

```python
assert "file not found" not in result.stderr, (
    "typst.compile() reported a missing file -- the nested master's "
    f"sibling include or image did not resolve:\nstderr: {result.stderr}"
)
```

`"file not found"` is typst-py's own uncontracted diagnostic wording (the
same phrase this file's own historical-record comments quote and
explicitly disclaim reliance on, e.g. line 125: *"an upstream rewording of
that text is expected NOT to break this test"*). This assertion is a
negative check (`not in`) in the test that is supposed to demonstrate the
fix WORKS (`result.returncode == 0` already established), so today it is
low-impact: if typst-py reworded this diagnostic, the assertion would
simply always pass (never catch anything) rather than ever going red for
the wrong reason. But that "always passes, never red" failure mode is
precisely the class of defect WR-02 exists to eliminate elsewhere in this
same file, and this line was left behind.

**Fix:** Drop the `"file not found" not in result.stderr` assertion (the
`returncode == 0` check already proves the build succeeded and no such
diagnostic was raised); if a defense-in-depth stderr check is wanted,
assert on a typsphinx-authored string instead (e.g. absence of the
builder's own `"Typst compilation failed"` / `"master document(s)
failed"` fragments, which are already used safely elsewhere in this
file and in `test_missing_and_malformed_master_gate.py`).

## Info

### IN-01: Missing precondition symmetry for the second (non-ASCII) docname

**File:** `tests/test_pdf_generation.py:348-384` (`test_builder_appends_failure_for_unknown_docname`)

**Issue:** The test asserts a precondition that `"ghost" not in
temp_sphinx_app.env.found_docs` before exercising the unknown-docname
branch for both `"ghost"` and `"日本語ドキュメント"`, but only checks the
precondition for `"ghost"`. If `temp_sphinx_app`'s fixture project ever
grew a document literally named `日本語ドキュメント`, this test would
silently start exercising the *other* (`found_docs`) branch for that
entry without any assertion failing to say so.

**Fix:** Add the symmetric precondition:
```python
assert "日本語ドキュメント" not in temp_sphinx_app.env.found_docs, (
    "precondition: '日本語ドキュメント' is not a real Sphinx document in "
    "temp_sphinx_app's srcdir"
)
```
