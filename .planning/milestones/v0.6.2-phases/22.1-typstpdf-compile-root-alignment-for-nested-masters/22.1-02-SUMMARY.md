All 3 pass, none skipped (`typst-py` 0.15.0 is installed).
- **Found and fixed a companion nested-master bug** (same failure class as PDF-02, different call site): `TypstWriter.translate()` hardcoded `template_file="_template.typ"` for every master document -- a bare reference correct only when the master sits at the outdir root, since `_write_template_file()` always writes `_template.typ` there regardless of where a master lives. A master at a nested docname could never resolve its own template import, and this gate's first real run failed on exactly that, before ever reaching the include/image bug PDF-02's canonical refs describe. Fixed by reusing the translator's existing `_compute_relative_include_path()` -- the same docname-relative pattern already established and tested for `#include()`/`image()` -- treating the template file as a top-level `"_template"` docname.
- Confirmed the "Confirmed both directions" record captures the REAL, run-captured Typst error text (not a paraphrase): `path "../_template.typ" would escape the project root` -- Typst's own root-boundary enforcement catching the wrong-directory compile, which is a more decisive proof of the outdir-root-vs-outdir/api divergence than a plain missing-file error would have been.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `TypstWriter.translate()` template import was hardcoded to the outdir root, breaking every nested master**
- **Found during:** Task 2, first real run of `test_typstpdf_nested_master_resolves_include_and_image` against the Task-1 fixture (pre-writer.py-fix code)
- **Issue:** `writer.py:152` called `template_engine.render(params, body, template_file="_template.typ")` unconditionally. `_write_template_file()` (`builder.py:553`) always writes `_template.typ` at the outdir *root*. For a root-level master this is a same-directory reference and resolves correctly; for a master at a nested docname (`api/index`), the bare `"_template.typ"` reference resolves relative to the master's own directory (`api/`), i.e. `api/_template.typ` -- a file that was never written. The build failed with `TypstError: file not found (searched at <outdir>/api/_template.typ)` before ever reaching the include/image bug PDF-02's canonical refs document.
- **Fix:** Compute the template import path via the translator's existing `_compute_relative_include_path()` method, treating `_template.typ` as a top-level `"_template"` docname: `template_file = self.visitor._compute_relative_include_path("_template", docname) + ".typ"`. For the nested fixture this correctly emits `#import "../_template.typ": project`; for a root-level master (`current_dir == "."`) the existing "root directory" branch returns the bare `"_template.typ"` unchanged, so root-level masters are byte-identical to before.
- **Files modified:** `typsphinx/writer.py`
- **Verification:** All 3 new gate tests pass; full fast suite (excluding the 5 pre-existing NixOS-sandbox-environmental files) 466/466 passing; `tests/test_template_assets.py`, `tests/test_template_engine.py`, `tests/test_template_codly.py`, `tests/test_template_mitex.py` (root-level-master template paths) all green -- no regression. `uv run mypy typsphinx/writer.py` clean. Slow corpus gate (`uv run pytest -m slow -q`) 22 passed, 1 skipped (pre-existing network-dependent skip, unrelated).
- **Committed in:** `d911f12` (Task 2 commit)

**2. [Rule 1 - Bug] D-08(a) pre-fix-basis test assertions adapted to the real, run-captured Typst error**
- **Found during:** Task 2, after applying deviation #1 above
- **Issue:** The plan's Task 2 action text anticipated the pre-fix-basis reproduction (copy `api/index.typ` to the outdir root, compile with `root=outdir`) would fail with a plain "file not found" naming the sibling `usage.typ`. After deviation #1's fix, the emitted master's *first* line is now `#import "../_template.typ": project`, so the reproduction fails on that reference first, with a *different* real Typst error class: `path "../_template.typ" would escape the project root` (a root-boundary rejection, not a missing-file error) -- since a copy sitting at the outdir root would need that reference to resolve one directory ABOVE the compile root.
- **Fix:** Rewrote the test's assertions to be tolerant of either real error signature (`"not found"` or `"escape"`) and to assert on the actual discriminating detail present (`"_template.typ"` named in the error), rather than forcing the assertion to match a error shape that no longer occurs. Transcribed the REAL captured error text verbatim into the module docstring's "Confirmed both directions" record, with prose explaining why this is still a direct, real, bidirectional proof of the same outdir-root-vs-outdir/api divergence.
