## User Constraints (from CONTEXT.md)

### Locked Decisions

**The fix shape (pdf.py)**

- **D-01: Add a file-path entry point and compile the master `.typ` directly — the temporary file
  disappears entirely.** New `compile_typst_file_to_pdf(typ_path, root_dir)` in `pdf.py`;
  `TypstPDFBuilder.finish()` calls it with the master's own `.typ` path (the path it already computes
  via `_resolve_output_stem` + `_directory_preserving_relpath`) and `root_dir=self.outdir` unchanged.
  Because the compiled file *is* the master `.typ` at its real location, the docname-relative paths
  the translator emitted resolve correctly **by construction** — the basis divergence cannot recur.
  Rejected: (a) keeping the temp copy but placing it beside the master; (b) rewriting
  `compile_typst_to_pdf()`'s own signature to take a path (breaking change across 18 call sites).

- **D-02: Keep `compile_typst_to_pdf(typst_content, root_dir)` with byte-identical behavior, but
  delegate internally and mark it deprecated in its docstring.** It still writes its temp file to
  `root_dir` exactly as today, then calls the new file-path function. No `DeprecationWarning` is
  emitted — v0.6.2 is a bug-fix release. All 18 existing tests stay untouched.

- **D-03: Announce removal of the content-string form for v0.7.** The Phase 23 `CHANGELOG` `[0.6.2]`
  entry carries the deprecation notice, and a backlog item is filed for the actual removal. Hand-off
  obligation, not extra work in this phase.

**Compile-failure behavior**

- **D-04: A PDF compile failure now raises and fails the build.** The current
  `except Exception: logger.error(...)` swallow is replaced. A missing artifact is a non-delivery,
  not a degradation — more aggressive than Phase 22's D-06 warn-and-continue posture.
- **D-05: Try every master, then fail once with an aggregated report.** `finish()` iterates all
  `typst_documents` entries, records each failure (master name + Typst error), keeps writing the
  `.pdf`s that *do* compile, and raises a single exception listing every failure after the loop.
  Rejected: fail-fast on the first error.
- **D-05a (hand-off): D-04 is a user-visible breaking change and MUST appear in the Phase 23
  CHANGELOG `[0.6.2]` entry.**

**Regression fixture (GATE-01)**

  `sys.executable -m sphinx` (**never** `uv run`), asserting the emitted artifact plus `%PDF` magic,

- **D-07: The fixture isolates the nested-master condition and nothing else.**
  `typst_documents = [('api/index', 'index', …)]` — **target == docname**. `api/index`

## Summary

This phase fixes a real, already-reproducible bug: `TypstPDFBuilder.finish()` (`typsphinx/builder.py:867-905`)
reads a master's `.typ` file into a string and hands it to `compile_typst_to_pdf(content, root_dir=self.outdir)`
(`typsphinx/pdf.py:110-168`), which writes that string to a `tempfile.NamedTemporaryFile(dir=root_dir, ...)` —
i.e. **at the outdir root** — and compiles that temp file. Typst resolves every relative path used inside
`#include()` / `image()` / `read()` **relative to the file containing the call**, not relative to `--root`
(confirmed against `typst/typst` upstream issue tracker, not just this project's own prior measurement). The
translator (`typsphinx/translator.py:2928` `_compute_relative_include_path`, `:3039`
`_compute_relative_image_path`) emits those relative paths **docname-relative**. The two bases coincide only
when the master sits at the outdir root (`index`) — every existing test and the full Sphinx corpus happen to
use root-level masters, which is why this has never been caught. A master at `api/index` already fails today
with a genuine Typst "file not found" fatal that is silently swallowed by `finish()`'s bare
`except Exception: logger.error(...)`, so `sphinx-build` exits 0 with a missing PDF.

The fix (D-01) is structural, not a patch: stop making a temp copy at all. Add
`compile_typst_file_to_pdf(typ_path, root_dir)` to `pdf.py` that calls `typst.compile(typ_path, root=root_dir)`
directly against the file `write_doc()` already wrote at its real, docname-derived location. `finish()` already
computes that exact path via `_resolve_output_stem` + `_directory_preserving_relpath` (Phase 22 machinery) —
it currently throws that path away by reading the file into a string first. Passing the path straight through
makes the divergence structurally impossible, not merely corrected for today's fixture. `root_dir` stays
`self.outdir` unchanged; Typst's own root-boundary enforcement (`typst/typst` — a resolved path may not escape
`root`) means this is safe as long as every master's on-disk location stays inside `outdir`, which it always
does by construction (all writes go through `path.join(self.outdir, ...)`).

Two compounding behavior changes are locked by CONTEXT.md and must land alongside the path fix: (D-04/D-05) a
PDF compile failure now raises `sphinx-build` to a non-zero exit after attempting every master, replacing the
current silent swallow — verified locally that any exception escaping `Builder.finish()` propagates through
`Sphinx.app.build()` to `sphinx.cmd.build.build_main()`, which returns exit code 2; and (D-06–D-09) a new
GATE-01-shaped render gate proves both the break (pre-fix) and the fix (post-fix), plus a fast unit test that
pins the path-resolution invariant without requiring `typst` to be installed.

**Primary recommendation:** Implement `compile_typst_file_to_pdf(typ_path, root_dir)` in `pdf.py` as the new
compile entry point (delegate the old string-based function into it, byte-identical behavior, docstring-only
deprecation), call it from `TypstPDFBuilder.finish()` with the master's already-computed real `.typ` path,
replace the `except Exception: logger.error` swallow with a per-master try/except that appends to a failure
list and raises `sphinx.errors.ExtensionError` (or an equivalent aggregate) once after the loop, and add one
new `tests/fixtures/<name>/` + `tests/test_*_render_gate.py` pair modeled directly on
`tests/test_target_name_render_gate.py` / `tests/test_glob_image_render_gate.py`.

### Pattern 1: File-path compile entry point (replaces the temp-copy indirection)

**What:** A new `pdf.py` function that compiles a `.typ` file already on disk, at its real location, instead
of writing a throwaway string-derived copy elsewhere.
**When to use:** Every new call site. The old `compile_typst_to_pdf(content, root_dir)` stays only for the 18
existing tests and any third-party caller that has not migrated.
**Example (locked shape per D-01/D-02 — not a full implementation, structural skeleton for the planner):**

```python
# Source: typsphinx/pdf.py, extending the existing check_typst_available()/TypstCompilationError machinery
def compile_typst_file_to_pdf(typ_path: str, root_dir: str | None = None) -> bytes:
    """
    Compile a Typst FILE (already on disk) to PDF bytes.

    Unlike compile_typst_to_pdf(), this takes a path directly -- no temporary
    file is created. The caller is responsible for `typ_path` already being
    at its real, intended location so that relative #include()/image() paths
    inside it resolve correctly (Typst resolves relative paths against the
    file containing the call, not against `root_dir` -- see RESEARCH.md).

    Args:
        typ_path: Path to an existing .typ file to compile.
        root_dir: Typst project root (bounds relative/absolute path escape).

    Returns:
        PDF content as bytes.

    Raises:
        ImportError: If typst package not available.
        TypstCompilationError: If compilation fails. source_location is the
            REAL typ_path (not a temp filename), so error locations are
            actionable for users.
    """
    check_typst_available()
    import typst

    try:
        pdf_bytes = typst.compile(typ_path, root=root_dir)
        return pdf_bytes
    except Exception as typst_error:
        error_msg = _parse_typst_error(typst_error)
        logger.error(f"Typst compilation failed at {typ_path}: {error_msg}")
        raise TypstCompilationError(
            message=error_msg, typst_error=typst_error, source_location=typ_path
        ) from typst_error


def compile_typst_to_pdf(typst_content: str, root_dir: str | None = None) -> bytes:
    """
    Compile Typst content (string) to PDF bytes.

    .. deprecated::
        Internal/legacy entry point, scheduled for removal in v0.7 (see
        CHANGELOG [0.6.2]). New code MUST use compile_typst_file_to_pdf().
        Because the temporary file this function creates lands at
        `root_dir`, relative paths in content authored for a master that
        does not sit at `root_dir` will NOT resolve -- this is exactly the
        nested-master bug PDF-02 fixes for the file-path entry point.
    """
    check_typst_available()
    temp_file = None
    try:
        with tempfile.NamedTemporaryFile(
            mode="w", suffix=".typ", dir=root_dir, delete=False, encoding="utf-8"
        ) as f:
            f.write(typst_content)
            temp_file = f.name
        return compile_typst_file_to_pdf(temp_file, root_dir=root_dir)
    finally:
        if temp_file and os.path.exists(temp_file):
            try:
                os.unlink(temp_file)
            except Exception:
                pass
```

### Pattern 2: Aggregate-and-raise in `finish()` (D-04/D-05)

**What:** Try every master, collect failures, raise once after the loop.
**When to use:** `TypstPDFBuilder.finish()`'s `typst_documents` loop (`builder.py:867-905`).
**Example (structural skeleton — exact exception type is Claude's Discretion per CONTEXT.md):**

```python
# Source: typsphinx/builder.py, TypstPDFBuilder.finish()
from sphinx.errors import ExtensionError
from typsphinx.pdf import compile_typst_file_to_pdf  # + existing compile_typst_to_pdf import

def finish(self) -> None:
    super().finish()
    typst_documents = getattr(self.config, "typst_documents", [])
    if not typst_documents:
        logger.warning("No documents defined in typst_documents. Nothing to compile.")
        return

    failures: list[tuple[str, str]] = []  # (docname, error message)
    for doc_tuple in typst_documents:
        if not doc_tuple:
            logger.warning(f"Skipping malformed typst_documents entry: {doc_tuple!r}")
            continue
        docname = doc_tuple[0]
        stem = self._resolve_output_stem(docname)
        relative_path = self._directory_preserving_relpath(docname, stem)
        typ_file = path.join(self.outdir, relative_path + ".typ")

        if not path.exists(typ_file):
            logger.warning(f"Master document not found: {typ_file}")
            continue

        try:
            # typ_file is ALREADY the master's real, on-disk location --
            # no read-to-string, no temp copy. D-08(b)'s unit test pins
            # exactly this: the path handed to compile is typ_file itself.
            pdf_bytes = compile_typst_file_to_pdf(typ_file, root_dir=self.outdir)
            pdf_file = path.join(self.outdir, relative_path + ".pdf")
            with open(pdf_file, "wb") as f:
                f.write(pdf_bytes)
            logger.info(f"Generated PDF: {pdf_file}")
        except Exception as e:
            logger.error(f"Failed to compile {typ_file}: {e}")
            failures.append((docname, str(e)))

    if failures:
        summary = "; ".join(f"{docname}: {err}" for docname, err in failures)
        raise ExtensionError(
            f"typstpdf: {len(failures)} master document(s) failed to compile "
            f"to PDF: {summary}"
        )
```

### Anti-Patterns to Avoid

- **Re-deriving the output path independently in `finish()` vs. `write_doc()`.** Both MUST call
  `_resolve_output_stem` + `_directory_preserving_relpath` with identical arguments — any divergence
  reintroduces a Phase-22-style name mismatch (canonical refs: "the path `finish()` compiles must be the
  identical expression this method writes — they must not drift").
- **Changing `root_dir`/`root=` away from `self.outdir`.** CONTEXT.md is explicit: "`root=self.outdir` stays
  exactly as it is. Only the *compiled file's location* changes." Moving `root` would touch the output-layout
  question that was deliberately dropped out of scope (master-output aggregation).
- **Touching `translator.py`.** The docname-relative emission is the reference basis this phase makes the
  compiler MATCH, not something to change. Canonical refs: "No `translator.py` edit is expected."
- **Emitting a `DeprecationWarning` from the legacy `compile_typst_to_pdf()`.** D-02/D-03 are explicit:
  docstring-only deprecation in v0.6.2; this also matters mechanically — `pyproject.toml`'s
  `filterwarnings = ["error::DeprecationWarning", "error::PendingDeprecationWarning"]` would turn a stray
  `DeprecationWarning` into a hard pytest failure across the entire suite.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Aggregating and reporting multiple build failures | A custom multi-error collector/formatter class | `sphinx.errors.ExtensionError(message)` with a pre-formatted message string | Sphinx's own CLI (`sphinx.cmd.build.build_main` / `handle_exception`) already knows how to print a `SphinxError` subclass cleanly and set the right exit code (2) — reinventing that plumbing duplicates behavior Sphinx already owns. `[VERIFIED: local sphinx.errors / sphinx.cmd.build source inspection, Sphinx 9.1.0]` |
| Detecting "did the build actually fail" in the new render gate | Parsing stdout/stderr text for a specific fatal string as the PRIMARY signal | `result.returncode` (now meaningful post-D-04) as primary signal, stderr substring assertions as secondary corroboration | Before this phase, `returncode` was NOT a reliable pass/fail signal (existing gates like `test_target_name_render_gate.py` and `test_glob_image_render_gate.py` predate D-04 and correctly avoid trusting it alone). After D-04, `returncode` becomes trustworthy for THIS gate specifically — but existing gates should not be retrofitted to rely on it (out of scope; D-06 only requires the NEW gate). |

**Key insight:** This phase is the rare case where the "don't hand-roll" list is short because the actual fix
is a *removal* (delete the temp-file indirection) rather than an addition — the fewer moving parts survive
the fix, the less there is to hand-roll.

## Common Pitfalls

### Pitfall 1: Two existing tests encode the OLD (swallow / temp-file) behavior and will not merely "stay green" — they assert the behavior D-04/D-05/D-01 explicitly changes

**What goes wrong:** `tests/test_pdf_generation.py::TestPDFErrorHandling::test_builder_error_handling_continues_on_failure`

that a raise occurs, so as written it would pass trivially with either old or new behavior, BUT its name and
docstring ("Test that builder continues processing other files after one fails") describe swallow-and-continue
semantics that D-04 explicitly ends (the build now fails after the loop, even though the loop-continues part
of D-05 stays true). Separately, `TestTypstPDFBuilder::test_builder_finish_generates_pdf` (line 83) does
`with patch("typsphinx.builder.compile_typst_to_pdf") as mock_compile:` — after D-01, `finish()` calls
`compile_typst_file_to_pdf`, not `compile_typst_to_pdf`, so this mock patch target silently stops intercepting
anything and the test would either fail (real compile runs, `mock_compile.called` is False) or pass vacuously
depending on Sphinx's env.
**Why it happens:** Both tests were written against `finish()`'s CURRENT internals (mock target name,
swallow-only assertion), not its post-fix internals.
**How to avoid:** The planner must include an explicit task to (a) update `test_builder_finish_generates_pdf`'s
patch target to `typsphinx.builder.compile_typst_file_to_pdf`, and (b) either rewrite
`test_builder_error_handling_continues_on_failure` to assert the NEW two-part contract (valid master's PDF is
still written; a raised exception names both docnames) or split it into two tests, one per half of D-05.
**Warning signs:** `mock_compile.called` assertions failing silently-green (mock never intercepted anything)
or `pytest.raises` missing where a raise is now expected.

### Pitfall 2: Typst's own root-boundary enforcement can independently reject a legitimate cross-directory reference if `root` is set too narrowly — but this project's usage is safe by construction

**What goes wrong:** Typst enforces that no resolved path (relative or absolute) may escape the `root`
directory — attempting to do so raises a "file not found" / access-denied class error distinct from a genuine
missing file. `[CITED: github.com/typst/typst/issues/1185, github.com/typst/typst/discussions/4195 —
community/upstream-issue sourced, MEDIUM confidence]`
**Why it happens:** If a future change ever set `root_dir` to something narrower than `self.outdir` (e.g. the
master's own subdirectory) while an `image()` call still needs to reach a sibling directory under `outdir`,
the reference would break with a DIFFERENT class of error than today's bug, easy to mis-diagnose as "the fix
didn't work."
**How to avoid:** D-01 explicitly keeps `root_dir=self.outdir` (an ancestor of every possible master
location, since all builder writes go through `path.join(self.outdir, ...)`) — do not narrow it. The new
fixture's `image("../logo.png")` reference (D-07: image lives at project root, master lives in `api/`) is a
direct, deliberate test of this exact boundary staying open with the LOCKED `root=self.outdir`.
**Warning signs:** A Typst error mentioning "access" or a path outside the expected `outdir` tree rather than
a straightforward missing-file message.

### Pitfall 3: `test_corpus_gate.py`'s slow gate does NOT assert `returncode == 0` today — it will keep passing/failing on the SAME signal after D-04, not a new one

**What goes wrong (avoided, but worth confirming for the planner):** It would be easy to assume D-04 requires
updating `tests/test_corpus_gate.py::TestCorpusRenderGate::test_corpus_compiles_with_no_fatal_error` (it only
asserts `pdf_path.exists()` / non-empty / `%PDF` magic, deliberately NOT `result.returncode`, per its own
in-file comment: "GATE-02 pass/fail signal (TypstPDFBuilder.finish() logs a fatal ... as an ERROR rather than
raising it)"). Since D-04 does not change WHETHER a given master's `typst.compile()` call succeeds or fails —
only what happens to the process after a failure — `pdf_path.exists()` remains the correct assertion
regardless of D-04, and this test needs no functional change (its stale in-file comment about "logs ... rather
than raising" becomes inaccurate prose and is a candidate for a one-line comment update, not a logic change).
**Why it happens:** Reading D-04's headline ("build now fails") without checking what the existing corpus gate
actually asserts could lead to an unnecessary or wrong edit.
**How to avoid:** Confirm during planning that this file only needs (at most) a comment-accuracy touch-up, not
a behavioral change to its assertions.
**Warning signs:** A plan task that tries to add `assert result.returncode == 0` to the corpus gate without
also verifying the corpus wiring never triggers a real compile failure (it currently doesn't — corpus is
root-level-master only).

### Unit test pattern for D-08(b) — path-resolution basis, no `typst` install required

```python
# Source: new pattern for this phase; mirrors the mocking idiom already used at
# tests/test_pdf_generation.py:102-104 (patch("typsphinx.builder.compile_typst_to_pdf"))
def test_finish_compiles_master_at_its_own_directory_not_outdir_root(
    temp_sphinx_app, tmp_path
):
    """D-08(b): the path handed to the compile function is the master's own
    .typ (inside its docname's subdirectory), never a file at outdir root --
    this is the structural invariant the fix depends on, checkable even when
    typst-py is not importable."""
    from typsphinx.builder import TypstPDFBuilder

    builder = TypstPDFBuilder(temp_sphinx_app, temp_sphinx_app.env)
    builder.outdir = str(tmp_path)
    builder.config.typst_documents = [("api/index", "index", "T", "A")]

    nested_typ = tmp_path / "api" / "index.typ"
    nested_typ.parent.mkdir(parents=True)
    nested_typ.write_text("= T\n")

    with patch.object(TypstPDFBuilder.__bases__[0], "finish"):
        with patch("typsphinx.builder.compile_typst_file_to_pdf") as mock_compile:
            mock_compile.return_value = b"%PDF-1.4 mock"
            builder.finish()

    called_path = mock_compile.call_args[0][0]
    assert called_path == str(nested_typ)          # NOT tmp_path / "tmpXXXX.typ"
    assert path.dirname(called_path) == str(tmp_path / "api")   # NOT outdir root
```
