## Summary

**CR-01 verification: the fix is correct and complete.** I traced the new

`tests/test_template_import_path.py` (`TEMPLATE_IMPORT_PATH_CASES`,
including the three previously-broken `_template`-directory docnames:
`_template/index`, `_template/sub/index`, `_template/a/b/index`), confirmed

reserved `_template` basename, and ran the full test module plus the
real-compile render gates locally (`test_template_import_path.py`,

While re-reading `typsphinx/builder.py` in full, I found one new,
previously-unreported robustness gap: `_resolve_output_stem`'s
directory-traversal guard does not actually neutralize a bare `".."` (or
unguarded `"."`) `typst_documents` target, even though its own log message
claims a substitution occurred (WR-03 below). I reproduced this directly
against the running code. It is not currently exploitable as an outdir
escape — every call site appends the output suffix by string
concatenation rather than `path.join`, which incidentally folds the
trailing `".."`/`"."` into a longer, harmless filename component — but the
guard's stated invariant does not hold for this exact input, and the log
message is user-facing incorrect.

The two prior WARNINGs (`WR-01` silent missing PDF, `WR-02` typst-error
substring coupling) and two prior INFO items (`IN-01` stale CHANGELOG
forward-reference, `IN-02` redundant `check_typst_available()` call) were
deliberately deferred/left as-is; I re-verified each still applies against
current source and restate them below, unescalated, per the review brief.
One new INFO item (unused local variable) is added.

## Warnings

### WR-03: `_resolve_output_stem`'s traversal guard does not neutralize a bare `".."` (or `"."`) target

**File:** `typsphinx/builder.py:190-219`
**Issue:** The D-06/D-07 path guard in `TypstBuilder._resolve_output_stem`
is meant to reduce any path-bearing or traversal-bearing `typst_documents`
target to a safe basename before it reaches `path.join(self.outdir, ...)`.
It correctly handles multi-segment traversal such as `"../manual"`
(reduces to `"manual"`, per
`tests/test_builder_output_stem.py::test_resolve_output_stem_guards_parent_traversal`),
but fails on the single-segment edge cases:

- A target of exactly `".."` trips the guard (`".." in segments`), but the
  "safe" fallback is computed as `path.basename("..")`, which in Python
  returns `".."` unchanged (there is no separator to split on). The code
  then logs `"a path is not supported ... using '..' instead"` — a message
  that claims a substitution happened when the value is byte-identical to
  the rejected input — and returns `".."` as the resolved stem.
- A target of exactly `"."` does not trip the guard at all (only `".."` is
  checked against `segments`), so it silently passes through unmodified.

Reproduced directly against the current code:

```
$ uv run python3 -c "
from typsphinx.builder import TypstBuilder
class FakeConfig:
    typst_documents = [('sub/index', '..', 'T', 'A')]
b = TypstBuilder.__new__(TypstBuilder)
b.config = FakeConfig()
stem = b._resolve_output_stem('sub/index')
print('stem =', repr(stem))
print('relative_path =', repr(b._directory_preserving_relpath('sub/index', stem)))
"
a path is not supported in a typst_documents target name: '..' -- using '..' instead
stem = '..'
relative_path = 'sub/..'
```

This is **not currently exploitable as an actual outdir escape**: every
call site (`write_doc`, `TypstPDFBuilder.write_doc`,
`TypstPDFBuilder.finish()`) appends the output suffix by plain string
concatenation (`relative_path + self.out_suffix`), which merges the
trailing `".."`/`"."` into a longer, harmless single path component (e.g.
`"sub/.." + ".typ"` becomes the literal string `"sub/...typ"` — a filename
inside `sub/`, not a `path.join`-resolved parent-directory reference). But
this safety is incidental to how the suffix happens to be concatenated,
not a property the guard itself guarantees — a future refactor that joins
the suffix via `path.join`/`Path./` instead of string concatenation would
turn this into a real write outside the master's intended docname
directory. At minimum, the misleading "using X instead" log message is
user-facing incorrect information today.

**Fix:** After computing `fallback = path.basename(...)`, explicitly treat
`"."` and `".."` as guarded/degenerate results (not just empty-after-strip),
so both the multi-segment and single-segment traversal forms are caught
identically:

```python
segments = stem.replace("\\", "/").split("/")
is_guarded = (
    any(sep in stem for sep in separators)
    or ".." in segments
    or "." in segments  # was previously unguarded
    or path.isabs(stem)
    or is_drive_qualified
)
...
if is_guarded:
    fallback_source = stem[2:] if is_drive_qualified else stem
    fallback = path.basename(fallback_source.replace("\\", "/"))
    if not fallback.strip() or fallback in (".", ".."):
        logger.warning(
            "empty typst_documents target name for docname "
            f"{docname!r} after removing an unsupported path -- "
            f"falling back to {docname!r}"
        )
        return docname
    ...
```

### WR-01: Missing `.typ` file is silently skipped, not surfaced as a failure (deferred, unescalated)

**File:** `typsphinx/builder.py:895-897`
**Issue:** In `TypstPDFBuilder.finish()`, when a configured master's `.typ`
file does not exist on disk, the loop only `logger.warning`s and
`continue`s — it is not appended to `failures`, so the build still exits 0
and reports success even though no PDF was produced for that master. This
diverges from the D-04 contract ("a build can no longer 'succeed' while
silently producing no PDF for a broken master") for this specific failure
mode (missing file, as opposed to a raised compile exception).
**Fix:** Still tracked as a deferred backlog item — see
`.planning/todos/pending/2026-07-22-wr01-silent-missing-pdf-wr02-typst-error-substring-coupling.md`.
Restated here unescalated per the review brief, not as a new finding.

### WR-02: Test assertions couple to Typst's error-message substrings (deferred, unescalated)

**File:** `tests/test_nested_master_render_gate.py:365-381, 504-525`
**Issue:** Several render-gate assertions match on lowercase substrings of
the real Typst compiler's error text (e.g. `"escape"`, `"not found"`) to
distinguish failure classes. This couples test stability to Typst's exact
error wording, which is not a documented, versioned contract of the
`typst-py` dependency and could change across Typst versions, potentially
producing noisy `drift.yml` failures unrelated to any real regression.
**Fix:** Still tracked as a deferred backlog item in the same pending todo
file referenced above. Restated here unescalated per the review brief, not
as a new finding.

### IN-02: Redundant `check_typst_available()` call on the `compile_typst_to_pdf` → `compile_typst_file_to_pdf` delegation path (still open)

**File:** `typsphinx/pdf.py:182, 138`
**Issue:** `compile_typst_to_pdf()` calls `check_typst_available()` at line
182, then (after writing the temp file) delegates to
`compile_typst_file_to_pdf()`, which calls `check_typst_available()` again
at line 138. Harmless (idempotent, cheap), but it is dead redundancy left
by the refactor.
**Fix:** Drop the now-redundant call in `compile_typst_to_pdf()` since
`compile_typst_file_to_pdf()` already performs the check before compiling.
