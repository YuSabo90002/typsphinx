## pdf.py の修正の形

### Q1 — 修正の形

| Option | Description | Selected |
|--------|-------------|----------|
| ファイルパス版を新設 | `compile_typst_file_to_pdf(typ_path, root_dir)` を新設し master の `.typ` を直接コンパイル。一時ファイルが消滅し、ルート不整合・残骸・命名衝突が根本から消える。既存 content 版は互換のため維持し内部委譲 | ✓ |
| temp を master と同階層に作る | ROADMAP が挙げた最小差分案。`temp_dir` 相当の引数を足し、builder が master の `.typ` のあるディレクトリを渡す（root は outdir のまま） | |
| content 版のシグネチャを置換 | `compile_typst_to_pdf()` 自体をファイルパス受け取りに作り替え、content 版は廃止。既存 18 テストの呼び出しを全書き換えする破壊的変更 | |

**User's choice:** ファイルパス版を新設（推奨）
**Notes:** →D-01。一時ファイルは「移動」ではなく「廃止」。

### Q2 — content 版の扱い

| Option | Description | Selected |

| 維持し、非推奨を明記 | 上に加えて docstring に「内部専用・新規コードはファイルパス版を使うこと」を記載。DeprecationWarning は出さない（v0.6.2 はバグ修正リリース） | ✓ |

**Notes:** →D-02。委譲込み。挙動は byte 単位で不変、既存 18 テストは無改修。

### Q3 — コンパイル失敗時の振る舞い

事前確認: ユーザーから「この PDF コンパイル失敗って PDF 自体の生成がされなくなる系だよな」との確認あり。
回答 — そのとおり。`compile` で例外が出ると `.pdf` の書き込みには到達せず、`except Exception` で握られて
次の master に進み、`sphinx-build` は exit 0。つまり「成果物が丸ごと欠落しているのにビルドは緑」。

| Option | Description | Selected |
|--------|-------------|----------|
| Sphinx の warning に格上げ | `logger.error` → `logger.warning`。通常ビルドは通り続けるが `-W` 利用者と CI は硬失敗する。Phase 22 の D-06 と同じ方針 | |
| 変更しない | PDF-02 のスコープはパス解決の整合のみとし、失敗時振る舞いは deferred に回す | |
| 例外を投げてビルドを落とす | コンパイル失敗を硬エラー化。最も安全だが、`-W` を使っていない既存利用者のビルドを壊す破壊的変更 | ✓ |

**User's choice:** 例外を投げてビルドを落とす
**Notes:** →D-04。Phase 22 の warn 方針より積極的。理由は「欠落は degradation ではなく非納品」。
Phase 23 CHANGELOG への破壊的変更告知が必須（D-05a）。

### Q4 — 複数 master のうち 1 つが失敗したとき

| Option | Description | Selected |
|--------|-------------|----------|
| 全部試してからまとめて落とす | 失敗を記録しつつ残りもコンパイルし、ループ後に失敗一覧を含む例外を 1 回投げる。成功した `.pdf` は手元に残る | ✓ |
| 最初の失敗で即時に落とす | fail-fast。実装は最小だが master が 3 つ壊れていると 3 回ビルドを回すことになる | |

**User's choice:** 全部試してからまとめて落とす（推奨）
**Notes:** →D-05。

## outdir 内一時ファイルの扱い

前置き: ファイルパス版新設（D-01）の時点で builder 経由の一時ファイルは消滅し、命名衝突・中断時の
残骸・ルート不整合はすべて解消。残る論点は「非推奨になった content 版が依然として `root_dir` 直下に
temp を作る」点のみ。

| Option | Description | Selected |
|--------|-------------|----------|
| 振る舞いはそのまま、docstring に明記 | 「`root_dir` 直下に一時ファイルを作るため、ルート以外に置かれた master の相対パスは解決できない」と制限を明記するだけ | |
| content 版の temp もシステム temp へ退避 | OS の temp に作る（root= は渡す）。outdir を汚さないが、非推奨 API の黙った振る舞い変更になる | |
| content 版を v0.7 で削除予告 | 上の明記に加え、Phase 23 の CHANGELOG で「将来のメジャーで削除予定」と告知し、backlog に削除タスクを立てる | ✓ |

**User's choice:** content 版を v0.7 で削除予告
**Notes:** →D-03。

## SC#3「両ビルダー一致」の検証

前置き: 修正後は `-b typstpdf` が `-b typst` の生成した `.typ` そのものをコンパイルするため、一致は
構造上自明になる。

| Option | Description | Selected |
|--------|-------------|----------|
| -b typst 側も実コンパイルする | 同じフィクスチャを `-b typst` でもビルドし、出力された `api/index.typ` を `typst.compile(path, root=outdir)` で直接コンパイルして成功を assert。ユーザーが手で typst compile する経路を再現し、将来の基盤ずれを検知できる | ✓ |
| -b typstpdf のコンパイル成功で十分 | 一致は実装上自明なので SC#3 は論証で閉じ、テストは一本に絞る | |
| 両ビルダーの .typ をバイト比較 | 直接比較する。ただし Phase 22 でファイル名解決は既に共通化済みで、パス解決の成否は証明できない | |

**User's choice:** -b typst 側も実コンパイルする（推奨）
**Notes:** →D-09。
