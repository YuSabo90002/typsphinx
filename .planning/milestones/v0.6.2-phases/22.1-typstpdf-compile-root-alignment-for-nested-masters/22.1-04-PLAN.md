## The gap being closed

`G-22.1-4` (severity: blocker; `22.1-UAT.md` test 4; source finding CR-01 in `22.1-REVIEW.md`).

Its diagnosis is **already complete and empirically verified three independent ways** — do not
re-diagnose, do not re-derive the fix, do not run exploratory investigations to "confirm" the
premise. Verified facts you may rely on:

`typsphinx/writer.py` computes the template import by calling the translator's docname-to-docname
relativization helper with the literal string `"_template"` as a **fake target docname**. When the
master's own directory portion is literally named `_template`, that sentinel collides with a real
path component, the helper's same-directory / common-parent logic resolves it as a path-to-itself,
and writer.py concatenates `".typ"` onto a stem-less result. Observed emitter output, captured from
a real `sphinx-build -b typst` run during diagnosis:

| docname | emitted import today | status |
|---------|----------------------|--------|
| `_template/index` | `#import "..typ"` | BROKEN |
| `_template/sub/index` | `#import "../.typ"` | BROKEN |
| `_template/a/b/index` | (depth-3 form of the same artifact) | BROKEN |
| `index` | `#import "_template.typ"` | correct |
| `api/index` | `#import "../_template.typ"` | correct |
| `a/b/index` | `#import "../../_template.typ"` | correct |
| `a/_template/index` | `#import "../../_template.typ"` | correct |

Both broken forms were compiled for real during diagnosis and fail with a missing-file error naming
the malformed path. With the depth-based computation applied, both compile to valid `%PDF-` bytes.

The remediation was **executed against the same case matrix before this plan was written**: computing
the depth from the master's own directory to the outdir root produces the correct result for all
three broken cases while remaining byte-identical on every currently-correct case. It is a strict
repair, not a behaviour change.

## Why the fix goes in writer.py and not in the translator helper

The helper is correct for the job it was written for: relativizing one real docname against another
(the Issue #5 nested-toctree `#include()` path basis, covered by `tests/test_nested_toctree_paths.py`
and the nested-toctree integration tests). The defect is the **caller's** decision to smuggle a
non-docname sentinel through it. Teaching the helper to special-case a reserved name would put a
tested, unrelated behaviour at risk to fix a bug that does not live there. See this plan's
prohibitions.

## Why a depth calculation is the right shape

`_write_template_file()` in `typsphinx/builder.py` **always** writes `_template.typ` at the outdir
root, unconditionally, regardless of where any master lives. So the reference a master needs is
purely "climb from my own directory to the outdir root, then name the file" — a function of nesting
depth alone, with no dependence on string equality against the reserved basename. That is why the
repair is immune to the collision by construction rather than by special-casing it.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| source-tree directory names -> emitted Typst `#import` path | A project author's directory naming (a docname) crosses into a path string embedded in generated Typst source. | build `outdir` -> `typst.compile(master, root=outdir)` | The gate compiles files inside a pytest `tmp_path`-scoped build tree; `root` bounds Typst's path resolution to that tree, so an upward reference that climbed past it would be rejected rather than reading outside. |
