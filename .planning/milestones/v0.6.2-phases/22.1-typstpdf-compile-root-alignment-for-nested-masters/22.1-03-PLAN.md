## Explicit decision: what happens to `test_outdir_root_compile_basis_still_fails`

Three options were considered; the choice is recorded here so it is not left implicit (the phase
constraints require this to be stated).

- **Rejected — leave it exactly as-is.** Its assertion accepts *either* failure class. With a second
  test now pinned to the include/image class, leaving the disjunction in place means a future drift
  (say, the template import disappears and both tests start hitting the same class) would be
  invisible: both tests would still pass. That is the same blind spot the reviewer just rejected,
  merely relocated.
- **Rejected — delete or fold it into the new test.** Its failure is real, non-vacuous, and its
  verbatim error is already transcribed in the module docstring. It proves a genuinely distinct
  thing: that the *template* reference is also basis-sensitive. Deleting it loses standing coverage.
- **CHOSEN — keep the test, keep its name, tighten its assertion, and relabel it in prose as the
  template-reference half of the proof.**
  - **Name is kept** because `22.1-VERIFICATION.md`, `22.1-UAT.md` and `22.1-02-SUMMARY.md` all cite
    it by name; renaming would orphan those references for no evidentiary gain.
  - **Assertion is tightened** to the single class it actually exercises (the `../_template.typ`
    root escape), so the two tests partition the failure space instead of overlapping it. After this
    change, drift in either direction turns exactly one test red with a message that says which
    class moved.
  - **Docstring is relabelled** so a future reader is not misled into thinking it covers the
    include/image references PDF-02 names.
</decision_the_existing_test>
