**Primary recommendation:** README.md は D-01〜D-14 の記述修正をそのまま実施しつつ、(1) github.io リンク 7 本には `/en/` プレフィックスを追加して 404 を解消する（github.io 内でのパス修正であり RTD 移行ではないためスコープ内）、(2) `typst_toctree_defaults` と `typst_elements` の非反映キーはソース修正が要る **実装バグ** なので本フェーズでは触れず、`docs/configuration.rst`/`docs/source/user_guide/configuration.rst` の内容乖離とあわせて新規 pending todo として起票する。README.md は単一ファイルへの集中編集なので 1 プラン・1 waveオーナーで完結させ、CLAUDE.md / pyproject.toml コメントは独立ファイルなので並列可能。

### ★新規discovery #1: `typst_elements` の `papersize`/`fontsize`/`lang` キーが出力に一切反映されない（実装バグ、実ビルドで再現）

README:137-141（Custom Templates 例）と Configuration Options の `typst_elements`: Template parameters (paper size, fonts, etc.) という説明は、`typst_elements` に入れたキーが `project()` テンプレート関数へ渡ると読める。しかし実装を追うと:

## Task F: プラン整形ガイダンス

### プラン/wave 分割の推奨

触るファイルは実質 4 種類（`README.md`、`CLAUDE.md`、`pyproject.toml`（コメントのみ）、新規 pending todo ファイル 2〜4 本）で、いずれも小さくファイル数も少ない。

- **`README.md` は単一ファイルへの集中編集** であり、D-01〜D-06、D-09〜D-14 のすべてがこの1ファイルに集約される。複数プランが同一 wave で `README.md` を編集すると git の観点でのコンフリクトリスクが高い（行番号が編集のたびにズレるため、diff の競合が起きやすい）。**推奨: README.md の全修正は 1 プランが一括して担当する。** 分割するとしても、同一ファイルを触るプランは同一 wave に並べず、直列（wave をまたぐ）にすること。
- `CLAUDE.md`（D-07/D-08）と `pyproject.toml:122-123`（D-08 のコメントのみ）は README.md と disjoint なファイルなので、README.md 担当プランと同一 wave で並列実行してよい。
- 新規 pending todo ファイル（D-15, D-16、加えて discovery #1/#3/#5 由来の追加 todo 候補）は `.planning/todos/pending/` に新規作成するのみで既存ファイルと競合しないため、どの wave でも並列実行可能。
- プロジェクトの標準実行モードは worktree 分離（CLAUDE.md §Worktree-isolated execution、`workflow.use_worktrees: true`）であり、本フェーズがプラン数・wave数とも少なくても sequential 実行への切り替えは行わないこと（オーナー決定により恒常適用）。
