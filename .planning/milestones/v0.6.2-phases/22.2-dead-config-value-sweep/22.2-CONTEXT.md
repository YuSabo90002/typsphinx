# Phase 22.2: Dead Config-Value Sweep - Context

**Gathered:** 2026-07-22
**Status:** Ready for planning

<domain>
## Phase Boundary

Two user-facing config values are registered and documented but never affect output. Delete
`typst_output_dir` outright, and repair the `typst_package` (Typst Universe) path end-to-end so a
project configured with `typst_package` **alone** builds under `-b typstpdf` and compiles with zero
Typst errors. Land a config→output regression fixture so a registration-only assertion can no longer
keep a dead feature green.

**The escape being closed:** the existing tests assert only that a value is *registered*
(`tests/test_config_other_options.py:141-179`) or that its name *appears in the docs*
(`tests/test_documentation_configuration.py:46`). Both stay green while the feature is dead.

**Defect catalogue as re-measured during this discussion** (the roadmap's BUG-A..BUG-D, plus one new
finding and one correction):

- **BUG-A (fatal, confirmed):** `writer.py:218` always passes a computed `template_file`, so
  `#import "<...>_template.typ": <func>` is always emitted — but `builder._write_template_file()`
  early-returns whenever `typst_package` is set (`builder.py:533-535`), so the file is never written.
  Fires with `typst_package` **alone**.
- **BUG-B (confirmed):** `map_parameters()` (`template_engine.py:186-191`) unconditionally fills
  `title`/`authors`/`date`. `charged-ieee`'s `ieee()` takes no `date` → `unexpected argument: date`
  once BUG-A is worked around.
- **BUG-C (confirmed):** `_format_authors_with_details()` (`template_engine.py:423`) has **zero
  callers**. `typst_authors` / `typst_author_params` never reach the output; a bare
  `authors: ("Name",)` string tuple is emitted instead of the documented dicts.
- **BUG-D (docs, confirmed):** `docs/source/examples/advanced.rst:59` modern-cv example is doubly
  wrong — `"name": "modern-cv"` → `unresolved import` (the entry function is `resume`), and its
  `params` are `author`-dict fields, not `resume` arguments.
- **BUG-E (NEW — found in this discussion, not in the roadmap):** precedence inversion.
  `render()` (`template_engine.py:336-338`) applies `typst_template_params` **first** and the
  auto-derived params **second**, so a value the user explicitly wrote in
  `typst_template_function["params"]` is silently clobbered by auto-derived Sphinx metadata. The
  in-code comment ("Standard params (can override if needed)") documents the behavior but the
  semantics are backwards.
- **BUG-F (NEW — the 5th missing piece of BUG-A's fix):** `render()`'s four `@preview` imports
  (`codly`, `codly-languages`, `mitex`, `gentle-clues`) and `codly-init` live **only inside the
  `template_file` branch** (`template_engine.py:310-317`). Fixing BUG-A by simply not importing
  `_template.typ` would leave the package-only path with no `codly`/`mitex`/`gentle-clues` at all, so
  every code block, math block, and admonition in the body would fail on undefined symbols.

**ROADMAP CORRECTION (SC#3 is stale).** ROADMAP SC#3 states that the `advanced.rst:126-133`
important-note "currently blames combining `typst_package` with `typst_template`, not the observed
cause". **This is no longer true.** Commit `1ae8c52` (`docs: drop redundant typst_package from
custom-template-wrapping examples`) already replaced that note; the current text names the real
mechanism (`_template.typ` never written → `file not found`) and states *use `typst_package` **or**
`typst_template` — not both*. **SC#3's remaining substance is the `advanced.rst:59` modern-cv example
only.** Downstream agents must not "fix" a note that is already correct.

**Out of scope:**
- Any `@preview` version bump or change to the 3-way version-sync surface (`writer.py` /
  `template_engine.py` / `templates/base.typ`).
- A full audit of every registered `typst_*` config for config→output coverage (see Deferred Ideas).
- Backlog 999.5 (silent missing-PDF skip + typst-error substring coupling) — untouched here, and
  D-08 below explicitly avoids adding new typst-error-string coupling.

**Milestone invariants carried forward:** zero new runtime dependencies, no `@preview` bump, the
3-way version-sync surface untouched.

</domain>

<decisions>
## Implementation Decisions

### BUG-A — package-only output shape

- **D-01: Fix BUG-A with a branch in `writer.py`, not in `builder.py`.**
  When `typst_package` is set and `typst_template` is not, call `template_engine.render()` with
  `template_file=None`, so the emitted master contains the package import + `#show: func.with(...)` +
  body and no `_template.typ` reference at all. `builder._write_template_file()`'s early return is a
  *correct* intent ("a package needs no separate template file") and stays. Rationale: the diff stays
  local to the `writer.py:218` region and does not re-enter the depth-computation surface Phase 22.1
  just repaired. Rejected: (a) having the builder emit a re-export shim `_template.typ`
  (`#import "@preview/...": ieee` + `#let project = ieee`) — unifies the import path but leaves a
  contentless intermediate file in `outdir` and keeps the package path dependent on
  `_compute_template_import_path()`; (b) demoting package-alone to unsupported — contradicts CONF-02.

- **D-02: The four `@preview` imports and `codly-init` are emitted for every master, unconditionally.**
  Closes BUG-F. `render()` emits `codly`, `codly-languages`, `mitex`, `gentle-clues` plus
  `#show: codly-init.with()` and `#codly(languages: codly-languages)` at the head of master output
  regardless of whether `template_file` is set — the same treatment included documents already get at
  `writer.py:139-150`. Rationale: consistency with the included-document path, and code blocks / math
  / admonitions must not break just because the user chose a Typst Universe package. Note for the
  planner: verify that the pre-existing `template_file`-set path does not end up emitting these twice
  (they are currently inside that branch), and that duplicate emission, if any, is removed rather
  than tolerated.

- **D-03: `typst_package` + `typst_template` together is unsupported, and the code now says so — warning + `typst_template` wins.**
  Emit a Sphinx warning when both are set, and narrow `builder._write_template_file()`'s early-return
  condition from "`typst_package` is set" to "`typst_package` is set **and** `typst_template` is
  unset", so the user's wrapper is actually written. This matches the documented intent in
  `docs/source/examples/advanced.rst:122-133` and in `examples/charged-ieee/approach2/conf.py`'s
  comment. Rationale: today the combination passes silently and emits a broken `.typ`; after this,
  behavior converges on the working `approach2` route. Rejected: letting `typst_package` win (throws
  away a template the user wrote), and docs-only (leaves the one silent-breakage path alive).

- **D-04: `typst_template` + `typst_template_function` together stays a first-class supported combination.**
  Owner-confirmed semantics: `render()` emits `#import "{template_file}": {template_func}`, i.e. the
  named function defined **in the user's own template file** is what gets imported and executed. That
  is the intended reading — "it is perfectly fine that the function defined by `typst_template` is
  the one that runs". `examples/charged-ieee/approach2/conf.py` already ships exactly this shape
  (`typst_template = "_templates/_template.typ"` + `typst_template_function = "project"`) and works
  today. No change; do not add a warning for this pair. This also defines where D-03 lands: after
  `typst_package` is ignored, the `typst_template_function` name resolves against the user's wrapper,
  joining the same path as `approach2`.

### BUG-B — parameter injection into package functions

- **D-05: On the `typst_package` path, stop the default injection of `title` / `authors` / `date`.**
  `map_parameters()`'s "fill if missing" fallback (`template_engine.py:186-191`) does not apply when
  `typst_package` is set. Only values explicitly mapped through `typst_template_mapping`, plus
  `typst_template_function["params"]`, are passed to the package function. Users match the package's
  real signature by writing the mapping. The existing template path keeps its current behavior
  untouched. Rationale: a Typst function signature cannot be introspected from Python, so the only
  sound rule is "pass what the user asked for". Rejected: (a) a new `typst_package_params` whitelist
  config — adds config surface in the very phase that removes dead config; (b) injecting defaults
  only when the mapping is left at its default — does not distinguish package from template and needs
  separate proof that default-template users are unaffected.

- **D-06: When a package function still receives an argument it does not accept, surface the raw Typst error unchanged.**
  It rides the Phase 22.1 `ExtensionError` attempt-all-then-raise path. `unexpected argument: date` is
  already self-explanatory. Rationale: adding a typsphinx-side hint would require matching on the
  typst error string, which is exactly the coupling backlog 999.5 already flags as a defect.

### BUG-C / BUG-E — author configs and parameter precedence

- **D-07: Wire `typst_authors` into `map_parameters()`; delete `typst_author_params` entirely.**
  `typst_authors` becomes live — when set, `authors` is emitted as the documented dict tuple via
  `_format_authors_with_details()` — and is proven by a real compile (`charged-ieee`'s `ieee()` takes
  authors as `(name:, department:, organization:, email:)` dicts). `typst_author_params` is removed
  from every surface (registration `__init__.py:58`, the `TemplateEngine` ctor arg, the two
  `getattr` call sites in `writer.py:177` / `builder.py:546`, the dead legacy branch in
  `_format_authors_with_details()`, and its backward-compat unit test
  `tests/test_template_engine.py:717`) and folded into CONF-01's scope. Evidence gathered during
  discussion: `typst_authors` appears in `docs/source/user_guide/configuration.rst:179,235`,
  `docs/source/examples/advanced.rst:42`, `docs/source/api/index.rst:67`,
  `examples/charged-ieee/approach1/conf.py:37` (labelled "Recommended") and `README.md:32`, while
  `typst_author_params` appears in **zero** user-facing docs or examples — only in `CLAUDE.md:67`'s
  inventory list and in code. Since neither has ever affected output, its "backward compatibility"
  never existed, so removal breaks nothing real.

- **D-08: Invert the parameter precedence so `typst_template_function["params"]` wins.**
  In `render()` (`template_engine.py:336-338`), apply the auto-derived params first and
  `typst_template_params` last, making what the user wrote explicitly authoritative and auto-derived
  Sphinx metadata the fallback. Rationale: owner ruling — the explicit config should win; today it
  loses silently. This applies to the template path as well as the package path, so the planner must
  include regression coverage proving existing default-template behavior is unchanged for configs
  that do not set overlapping keys. Rejected: warning on every key collision (noisy for people
  intentionally overriding), and deferring to backlog (leaves the template path silently clobbering).

### CONF-02 / CONF-03 — fixture scope

- **D-09: The CONF-02 real-compile gate is a NEW minimal fixture under `tests/fixtures/`, using `charged-ieee` alone.**
  A minimal Sphinx project with `typst_package = "@preview/charged-ieee:0.1.4"` and no
  `typst_template`, built with `-b typstpdf` through a real `typst.compile()` (GATE-01 form), with
  BUG-A / BUG-B / BUG-C / BUG-E each pinned by a named assertion, and a proof that the fixture
  **fails** against the pre-fix code. Rationale: control and diagnosability — a failure points at one
  defect rather than at fixture-specific baggage such as `approach1`'s `bibliography: "refs.bib"`.
  Reachability note: `tests/test_preview_smoke_gate.py` already downloads and compiles four `@preview`
  packages in CI, so Typst Universe access is established; `charged-ieee` is a 5th package.

- **D-10: CONF-03's config→output matrix is scoped to the configs this phase actually touches.**
  `typst_package`, `typst_template`, `typst_template_function`, `typst_authors` (and the
  absence-of-`typst_output_dir` assertion) get assertions that setting the value **changes the
  emitted output**, not merely that it is registered. A full inventory of every registered `typst_*`
  goes to the backlog (see Deferred Ideas). Rationale: keeps v0.6.2 a bug-fix release; a full audit
  would very likely unearth more dead configs and balloon the phase.

- **D-11: Replace the modern-cv example rather than repair it.**
  Delete the `docs/source/examples/advanced.rst:59` modern-cv example and consolidate on the
  `charged-ieee` example that the D-09 fixture actually compiles. Rationale: establishes the
  invariant *every package example in the docs is backed by a test*. Rejected: repairing modern-cv
  (would need a second package download and a second gate), and keeping it with an "unverified" note
  (deliberately preserves the docs-vs-reality divergence this phase exists to eliminate).

- **D-12: `examples/charged-ieee/approach1` is in scope — fix it and pin its build with a test.**
  The bundled sample is confirmed broken today: `approach1` (package-alone + `typst_authors` +
  `typst_template_function` dict) trips BUG-A, BUG-B, BUG-C and BUG-E all at once. Update it to match
  post-fix behavior and add a test that its build passes. `approach2` (template route) works today
  and needs no functional change. Division of labour: the D-09 fixture isolates individual defects;
  the sample test guarantees the shipped example is not broken. Rationale: leaving a broken bundled
  sample would contradict the `[0.6.2]` release note that says the package path works.

### CONF-01 — `typst_output_dir` removal (pre-decided, carried forward, NOT re-opened)

- **D-13: `typst_output_dir` is deleted from every surface with no deprecation period.**
  Owner decision 2026-07-21, unchanged. Sphinx silently ignores unregistered `conf.py` variables
  (verified 2026-07-21 under `sphinx-build -W --keep-going`), so removal is behaviorally invisible,
  and the config is structurally unimplementable as documented — `outdir` comes from the
  `sphinx-build` CLI argument. Surfaces enumerated in ROADMAP SC#1. Per D-07, `typst_author_params`
  is removed alongside it under the same rationale.

<specifics>
## Specific Ideas

- The owner explicitly wants `typst_template_function` to be authoritative over auto-derived
  metadata (D-08), and explicitly accepts that under `typst_template` + `typst_template_function`
  the function defined in the user's own template file is the one that executes (D-04).
- `examples/charged-ieee/approach1/conf.py` is the canonical illustration of what the package path is
  *supposed* to look like — package alone, `typst_authors` dicts, `typst_template_function` dict with
  `params` (`abstract`, `index-terms`, `bibliography`, `paper-size`). Post-fix behavior should make
  that exact file work.
- Phase 23's `[0.6.2]` CHANGELOG entry needs a `### Removed` section covering **both**
  `typst_output_dir` and `typst_author_params`, plus a fix note for the `typst_package` path repair.
  ROADMAP line 376 currently names only `typst_output_dir` — Phase 23 must be told about the second
  removal.

</specifics>
