## Summary

This phase is a **verified-root-cause bugfix + config-removal** phase, not a new-technology phase — there
is no new library to adopt. The research task was to re-derive every code coordinate CONTEXT.md cites
against the live tree, and to trace `TemplateEngine.render()`'s actual data flow closely enough to catch
implementation traps that a literal reading of CONTEXT.md's decisions would miss.

All CONTEXT.md line-number citations were re-verified and are **accurate** against the current tree
(`writer.py:218`, `builder.py:533-535`, `template_engine.py:186-191`, `:301-351`, `:310-317`,
`:336-338`, `:423`, `__init__.py:57-58,60`, `tests/test_config_other_options.py:141-179`,
`tests/test_documentation_configuration.py:46`, `tests/test_template_engine.py:717`). One important
**correction** was found by executing the code, not just reading it: D-05's literal fix site
(`map_parameters()`'s "fill if missing" block, lines 186-191) is **not** where `title`/`authors`/`date`
actually get unconditionally injected. That happens earlier, in the unconditional "Apply mapping"
for-loop (lines 174-183), because `self.parameter_mapping` defaults to `DEFAULT_PARAMETER_MAPPING`
(`project→title`, `author→authors`, `release→date`) **regardless of `typst_package`**, and
`sphinx_metadata` (from `writer.py`) always contains `project`/`author`/`release`. A live repro proves
this: calling `map_parameters()` on a package-only `TemplateEngine` today returns
`{'title': 'My Paper', 'authors': ('John Doe',), 'date': ''}` — the exact `date: ""` shape the original
bug repro observed — **before the code ever reaches line 186**. Gating only lines 186-191 as literally
described would leave BUG-B's user-visible symptom (`unexpected argument: date`) unfixed. The correct
fix site is the ctor's `self.parameter_mapping` default (or an equivalent package-aware guard in
`map_parameters()` before the loop runs) — see Common Pitfalls #1 for the full trace and a proposed fix
shape.

A second implementation trap was found for D-07 (wiring `typst_authors`): `_format_authors_with_details()`
returns a **pre-formatted Typst source string** (its own hand-rolled dict-tuple serializer), but
`render()`'s parameter loop unconditionally re-runs every value through `_format_typst_value()`, which
would re-quote and escape that string as a Typst string literal — producing `authors: "(  (    name:
..."` instead of an actual tuple of dicts. `_format_typst_value()` already has fully general
list/dict recursion that produces the identical Typst shape from a native Python
`list[dict]` — see Common Pitfalls #2 for the recommended non-invasive wiring.

Both `charged-ieee:0.1.4`'s `ieee()` signature (BUG-B/C's proving ground) and the GATE-01 fixture shape
(D-09) were verified directly: the package is already downloaded in this environment
(`~/.cache/typst/packages/preview/charged-ieee/0.1.4/lib.typ`), `ieee()` takes
`title`/`authors`/`abstract`/`index-terms`/`paper-size`/`bibliography`/`figure-supplement`/`body` and
has **no `date` parameter** (confirms BUG-B exactly), and its `authors` array elements read
`name`/`department`/`organization`/`location`/`email` (confirms D-07's dict shape, with one addition —
`location` — worth offering in docs/examples but not required).

**Primary recommendation:** Implement D-01 through D-13 exactly as CONTEXT.md specifies, with two
load-bearing corrections: (1) fix BUG-B at the `parameter_mapping` default / apply-mapping-loop level,
not solely at the "fill if missing" fallback CONTEXT.md cites; (2) wire `typst_authors` as a native
Python `list[dict]` assigned to `params["authors"]` (reusing `_format_typst_value()`'s existing
recursion) rather than by stuffing `_format_authors_with_details()`'s pre-formatted string through the
generic per-key formatter.

### Pattern 1: GATE-01 real-compile fixture (established convention to copy for D-09)
**What:** A minimal Sphinx project under `tests/fixtures/<name>/`, built via
`sys.executable -m sphinx -b typst <fixture> <tmp_build>`, then the emitted `.typ` is fed to a real
`typst.compile()` call — no mocking, no string-matching on typst-py's error format for the *success*
path (string-matching the *failure* path pre-fix, in the module docstring only, is the established
"confirmed both directions" convention).
**When to use:** Any regression that only manifests when Typst actually parses/executes the emitted
markup (as opposed to a translator-output shape assertion).
**Example (from `tests/test_preview_smoke_gate.py`, reused verbatim for D-09):**
```python

    [sys.executable, "-m", "sphinx", "-b", "typst", str(fixture_dir), str(temp_build_dir)],
    capture_output=True, text=True,
```

## Common Pitfalls

### Pitfall 1: D-05's cited fix site (map_parameters lines 186-191) does not actually stop BUG-B
**What goes wrong:** If the planner implements D-05 literally as written — gate the "fill if missing"
fallback block (`template_engine.py:186-191`) behind `if not self.typst_package:` — the fixture will
still observe `unexpected argument: date` (or `title`/`authors`, depending on what the target package
function accepts), because `title`/`authors`/`date` are **already set** in `params` by the time
execution reaches line 186.
**Why it happens:** `self.parameter_mapping` (set in `__init__`, `template_engine.py:72-74`) defaults
to `DEFAULT_PARAMETER_MAPPING = {"project": "title", "author": "authors", "release": "date"}`
**unconditionally** — it has no awareness of `typst_package` at all. `map_parameters()`'s "Apply
mapping" for-loop (`template_engine.py:174-183`) then walks that mapping and sets `params[template_key]
= sphinx_metadata[sphinx_key]` for every key present in `sphinx_metadata` — and `sphinx_metadata`
(built in `writer.py:181-186`) **always** contains `project`/`author`/`release`/`copyright`,
unconditionally, regardless of any config. So `params["title"]`, `params["authors"]`, and
`params["date"]` are all populated by line 183, well before the "fill if missing" fallback at line 186
ever runs (and that fallback is in fact a no-op in this scenario, since the keys are already present).
Verified live:
```python
# Executed against the current tree, no modifications:
engine = TemplateEngine(
    typst_package="@preview/charged-ieee:0.1.4",
    typst_template_function={"name": "ieee", "params": {"abstract": "x"}},
)
sphinx_metadata = {"project": "My Paper", "author": "John Doe", "release": "", "copyright": "2025"}
engine.map_parameters(sphinx_metadata)
# => {'title': 'My Paper', 'authors': ('John Doe',), 'date': ''}
```
The `date: ""` value matches the original bug repro's observed `#show: ieee.with(..., date: "")` output
exactly — proving the mapping loop, not the fallback, is the actual unconditional-injection source.
**How to avoid:** Make `self.parameter_mapping`'s default package-aware. The minimal, most local fix:
in `__init__`, when `parameter_mapping` is not explicitly supplied *and* `typst_package` is set, default
`self.parameter_mapping` to `{}` instead of `DEFAULT_PARAMETER_MAPPING.copy()`. This preserves D-05's
stated intent exactly ("Only values explicitly mapped through `typst_template_mapping` ... are passed")
while actually stopping the injection at its real source. The existing "fill if missing" fallback
(186-191) then becomes correctly inert for the package-only case (nothing to gate — `params` stays
`{}` unless the user supplies an explicit `typst_template_mapping`), so D-05 does **not** need a second,
separate guard there; only the ctor-level default needs the `typst_package` check. The template path
(`typst_package` unset) is completely unaffected, since its `self.parameter_mapping` still defaults to
`DEFAULT_PARAMETER_MAPPING` exactly as today — satisfying "the existing template path keeps its
current behavior untouched."
**Warning signs:** A GATE-01 fixture asserting "no `date:` key in emitted `.typ`" for a package-only
build would still fail after an literal 186-191-only edit; that failure is the tripwire.

### Pitfall 2: Wiring `typst_authors` by calling `_format_authors_with_details()` double-formats the value
**What goes wrong:** `_format_authors_with_details()` (`template_engine.py:423-466`) does not return a
Python value — it returns an **already Typst-source-formatted string**, e.g.
`'(\n  (\n    name: "John Doe",\n    department: "Computer Science",\n  ),\n)'`. `render()`'s
parameter-output loop (`template_engine.py:341-343`) unconditionally re-runs *every* value through
`self._format_typst_value(value)`. Since that pre-formatted text is a Python `str`,
`_format_typst_value` treats it as a plain string and quote-escapes it
(`template_engine.py:368-371`), producing `authors: "(\n  (\n    name: \"John Doe\", ...` — a Typst
STRING LITERAL containing the text of a tuple, not an actual tuple. `charged-ieee`'s `ieee()` then
calls `authors.map(...)` (confirmed in the cached `lib.typ:185-216`) on what is now a Typst `str`, not
an `array` — `.map()` is not defined on `str` in Typst, so this is a **fatal compile error**, exactly
the class of regression CONF-02 exists to prevent.
**Why it happens:** `_format_authors_with_details()` was written as a self-contained legacy formatter
(pre-dating the more general `_format_typst_value()`) and was never actually wired into `render()`'s
pipeline (that is BUG-C itself — it currently has zero callers). Naively "just calling it," as
CONTEXT.md's code_context section suggests ("`_format_authors_with_details()` ... D-07 only needs to
call it"), skips over the fact that its OUTPUT TYPE (pre-formatted string) is incompatible with the
generic per-key formatter every other param already goes through.
**How to avoid:** `_format_typst_value()` already has fully general recursion for `list`/`tuple` of
`dict` (`template_engine.py:374-381`) — feeding it a **native Python `list[dict]`** produces the
identical Typst shape without any new formatting code. Recommended shape: in `map_parameters()`, when
`self.typst_authors` is truthy, build
`params["authors"] = [{"name": name, **details} for name, details in self.typst_authors.items()]`
(this is exactly the merge logic already inside `_format_authors_with_details()`'s `typst_authors`
branch, lines 441-444 — just stopping one level earlier, before that method's manual string-building
begins) and let the existing generic formatter in `render()` handle serialization untouched. This also
resolves Claude's Discretion item "whether the legacy branch is deleted in the same commit or an
adjacent one" in favor of: strip the `typst_author_params` branch (D-07's deletion target) and either
(a) delete `_format_authors_with_details()` entirely once its only remaining logic is a 3-line dict
comprehension inlined directly into `map_parameters()`, or (b) keep a slimmed
`_format_authors_with_details()` that *returns the native `list[dict]`* (rename/repurpose, drop the
string-building) if a dedicated helper is still wanted for testability. Verified: this recursion path
is already exercised by nothing regressive — `_format_typst_value` has no special-casing for author-
shaped dicts, it is fully generic.
**Warning signs:** If a GATE-01 fixture's pre-fix-vs-post-fix diff on the emitted `.typ` shows
`authors: "..."` (a single quoted string) rather than `authors: ((name: "...", ...),)` (a tuple of
dicts), the wiring used the string-returning path by mistake.

### Pitfall 3: D-02's "hoist" must be a move, not a copy — verified no duplication risk exists today
**What goes wrong:** If the essential-imports block (`template_engine.py:310-317`) is duplicated
(added unconditionally at the top of `render()` *and* left inside `if template_file:`), every
already-working template-path build would emit `#import "@preview/codly:1.3.0": *` (and the other
three) **twice** in the same master `.typ` file. Re-importing the same module twice into the same
Typst scope with `*` (wildcard) re-imports is not necessarily fatal in Typst but is unnecessary bloat
and a code-smell regression a reviewer would flag.
**Why it happens / why it's actually safe to fix:** Traced the full call graph — `render()` is called
from exactly one production call site (`writer.py:218`), and that block currently has exactly ONE
emission site, gated by `if template_file:` (true for every master today, since `writer.py` computes
`template_file` unconditionally pre-fix). There is no second source of these four imports **within
`render()`'s own output** — the only other place they appear is inside the *separately compiled*
`_template.typ` file (when `_write_template_file()` writes `base.typ`'s content, which itself imports
the same four packages at its own top level, `templates/base.typ:8-19`). That is a **different Typst
module scope** (imported via `#import "_template.typ": project`, not inlined) and has always coexisted
harmlessly with the master's own copy — Typst's `#show` rules are scoped to the file they appear in, so
`_template.typ`'s own `#show: codly-init.with()` never leaks into the master's scope, which is exactly
why the master needs its own copy today. **This cross-file relationship is pre-existing, unaffected by
D-02, and not a duplication risk** — D-02 only changes whether the master's *own* copy is
conditional or unconditional, not whether a second file also happens to import the same packages into
its own separate scope.
**How to avoid:** Implement by literally relocating the `output_parts.append(...)` calls for the four
imports + `codly-init`/`codly()` lines to run before the `if template_file:` conditional, and delete
them from inside that conditional. Do not add a second copy anywhere.
**Warning signs:** `grep -c '@preview/codly:1.3.0' <emitted .typ>` should stay `1` for the master file
itself, both before and after this change, for every existing template-path fixture.

### Pitfall 4: Phantom, unregistered config values already exist in `docs/source/user_guide/configuration.rst`
**What goes wrong (informational, NOT in this phase's scope per D-10):** `docs/source/user_guide/
configuration.rst` documents `typst_use_codly`, `typst_papersize`, `typst_fontsize`,
`typst_code_line_numbers`, and `typst_author` (singular) — **none of these are registered anywhere in
`typsphinx/__init__.py`** (`grep -c` for each returns 0). Writing any of them in a user's `conf.py`
does nothing (Sphinx silently accepts unknown config vars) and produces no warning. This is the exact
same escape class CONF-01/02 close for `typst_output_dir`/`typst_package`, just for a different set of
names, and is direct evidence supporting D-10's deferred "full config→output audit" backlog item.
**Why it happens:** Pre-existing doc drift, unrelated to this phase's BUG-A..F catalogue.
**How to avoid / recommendation:** Do not fix in this phase (D-10 explicitly scopes CONF-03's matrix to
the 5 configs this phase touches). File or update the existing "full config audit" backlog item to
name these four phantom configs explicitly, so the next config-audit phase does not have to
rediscover them from scratch.
**Warning signs:** none for this phase — flagged for hand-off only.

### Correct D-07 wiring shape (native `list[dict]`, no double-formatting)
```python
# Source: verified against typsphinx/template_engine.py:172-193 (map_parameters) and
# the existing merge logic already in _format_authors_with_details() (:441-444)
def map_parameters(self, sphinx_metadata: Dict[str, Any]) -> Dict[str, Any]:
    params = {}
    for sphinx_key, template_key in self.parameter_mapping.items():
        if sphinx_key in sphinx_metadata:
            value = sphinx_metadata[sphinx_key]
            if sphinx_key == "author" or template_key in ("authors", "doc_authors"):
                value = self._convert_to_authors_tuple(value)
            params[template_key] = value

    if "title" not in params:
        params["title"] = ""
    if "authors" not in params:
        params["authors"] = ()
    if "date" not in params:
        params["date"] = None

    # D-07: typst_authors (explicit config) overrides whatever "authors"
    # already resolved to -- a native Python list[dict] that
    # _format_typst_value()'s existing recursion serializes correctly.
    if self.typst_authors:
        params["authors"] = [
            {"name": name, **details} for name, details in self.typst_authors.items()
        ]

    return params
```
Verified this recursion produces the correct Typst shape by tracing `_format_typst_value()`
(`template_engine.py:374-381` for the list branch, `:378-381` for the dict branch) — a
`list[dict]` recurses to `(  (name: "...", department: "...",),  )`, matching what
`ieee()`'s `authors.map(author => ...)` (cached `lib.typ:192-210`) expects.

## Security Domain

`security_enforcement` is enabled in `.planning/config.json`, but this project is a local Sphinx
extension / CLI build tool with no network service, no authentication surface, and no user-facing HTTP
input — the classic OWASP-ASVS web categories (V2 Authentication, V3 Session Management, V4 Access
Control) do not apply, matching the precedent set by every prior v0.6.2 phase's RESEARCH.md (none of
which included a Security Domain section, since none introduced a web-facing surface). The threat model
here is narrower and specific to this phase's actual attack surface: `conf.py` is already fully-trusted,
arbitrary-code-execution Python (a Sphinx project's `conf.py` can do anything a Python script can do)
before typsphinx ever runs — so the relevant question is not "can a user inject something malicious via
config" (they already have full code execution if they control `conf.py`) but "does typsphinx's
handling of config values introduce a NEW class of risk beyond what `conf.py` already grants."

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | No | No auth surface — local build tool |
| V3 Session Management | No | No sessions — single-process CLI build |
| V4 Access Control | No | No multi-user access boundary |
| V5 Input Validation | Marginal-yes | `typst_package`/`typst_template_function`/`typst_authors` are Python-typed config values already validated by Sphinx's own `add_config_value(..., [type, ...])` type-checking (`__init__.py`); this phase adds no new user-facing input path (the D-09 fixture is a fixed, hardcoded test config, not runtime user input) |
| V6 Cryptography | No | No cryptographic operations in this phase |
| V12 Files and Resources | Marginal-yes | `typst_package` is a string handed to `typst.compile()`, which resolves and downloads a Typst Universe package by name — this is the SAME trust boundary every existing `@preview` import in this codebase already crosses (four packages already downloaded automatically today); this phase adds a 5th consumer of an already-cached, already-pinned-by-version package string, not a new download-arbitrary-package code path |

### Known Threat Patterns for this stack

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Malicious `typst_package` string pointing at an attacker-controlled or unpinned package | Tampering / Supply chain | Already mitigated by the existing convention (this phase does not change it): every `@preview` package reference in this codebase is version-pinned literally (`:0.1.4`, `:1.3.0`, etc.) in source, never resolved from a floating/latest tag; `typst_package` is a value the DEVELOPER of the Sphinx project sets in their own trusted `conf.py`, not an external/network input |
| `typst_authors`/`typst_template_function["params"]` values containing Typst syntax-breaking characters | Tampering (content injection into generated `.typ`) | Already mitigated, unaffected by this phase: `_format_typst_value()` escapes quotes/backslashes for every string value (`template_engine.py:368-371`) before embedding it in the emitted `.typ`; this phase reuses that existing escaping path for the newly-wired `typst_authors` values (Common Pitfall #2's recommended fix routes through this exact function), rather than bypassing it |
| Removing `typst_output_dir`/`typst_author_params` breaking a downstream consumer silently | Availability (breaking change) | Addressed structurally by D-13's own reasoning (Sphinx silently ignores unknown `conf.py` vars — no exception, no crash for users who still set them) plus this phase's `### Removed` CHANGELOG staging for Phase 23; not a code-level mitigation, a process one |

No new threats are introduced by this phase; the two marginal-applicability categories above describe
existing, already-accepted trust boundaries (developer-authored `conf.py`, version-pinned Typst Universe
package references) that this phase's changes are consistent with, not expansions of.
