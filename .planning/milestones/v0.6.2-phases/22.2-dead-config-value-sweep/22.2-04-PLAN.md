<objective>
Route the Typst Universe package path correctly: a package configured alone produces a self-contained
master with no shared-template reference (D-01), and a package configured together with a custom
template produces one warning with the custom template winning (D-03), while the supported
template-plus-function combination is left untouched (D-04) and Typst's own error text is never
matched on (D-06).

Purpose: CONF-02's routing half — BUG-A, the fatal defect. Today the writer unconditionally emits an
import of a shared template file while the builder unconditionally refuses to write that file
whenever a package is configured, so a package-alone project cannot compile at all.

Output: a writer and builder whose routing decisions agree, plus a new test module pinning all three
routing cases (package alone, both set, template alone).
</objective>

<assumption_delta_decision>
**Noun now primary:** the *routing decision* between the package path and the custom-template path.

Before this phase, `typst_package` and `typst_template` were both silently accepted and their
combination produced a broken document. After D-03, the combination is explicitly unsupported and
`typst_template` is primary — "Custom Template Wrapping" is the authoritative route whenever a
custom template is present, and the package setting is a strictly weaker alternative that only takes
effect in its absence. In the same movement, D-08 (plan `22.2-03`) inverts parameter precedence so
explicit configuration is authoritative over auto-derived Sphinx metadata; the two changes share one
principle — *what the developer wrote explicitly wins, and anything ignored is named out loud*.

**Decision: `promote`.** `typst_template` is promoted to the primary routing noun; `typst_package`
remains supported alone, not alongside. Rationale: the custom-template route is the only one that
already worked, it is the route both the bundled sample's comment and the published documentation
recommend for wrapping a package, and it can express everything the package route can. Promoting it
requires no new configuration surface in a phase whose purpose is removing configuration surface.

**What would force a later re-visit:** if a future phase adds a mechanism for the package path to
receive user-authored Typst preamble (making the two routes composable rather than exclusive), the
warning becomes wrong and the decision returns to `add-alongside`.
</assumption_delta_decision>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer `conf.py` → build routing decision | Two mutually exclusive config values reach a routing branch that decides which files are written and referenced. A silent resolution here is an integrity problem: the developer's stated intent is discarded without notice. |
| emitted `.typ` → Typst compiler | The generated document names files (the shared template) and packages that must exist at compile time; a routing disagreement produces a reference to a file that is never written. |
