<objective>
Repair the bundled `approach2` sample so it actually demonstrates what it claims: a custom Typst
template that wraps the pinned `charged-ieee` Typst Universe package.

Purpose: CONF-02. This is an **owner-ruled scope addition dated 2026-07-22**, correcting a premise in
D-12. D-12 states that approach2 "works today and needs no functional change"; verified during
planning, that is empirically false. Its configured template path resolves against the Sphinx source
directory (`approach2/source/`) while the template file ships at `approach2/_templates/`, so
typsphinx logs a not-found warning and silently falls back to the bundled default template. The
build exits 0, a shared template file is written, and the package is never loaded — a silent
fallback keeping a dead demonstration green is precisely the defect class this phase exists to
eliminate.

Two further defects were found in the same wrapper while verifying the fix, both of which keep the
sample from compiling even once the path is correct.

Output: an approach2 sample whose emitted shared template is its own wrapper and whose build
produces a real PDF.
</objective>

command:

2. Run **every** subsequent command through `uv run` (e.g. `uv run python -m sphinx …`, `uv run pytest …`).

Without this, the sample builds against the UNCHANGED main-tree package. See `CLAUDE.md`

## Owner ruling and D-12 premise correction (read before Task 1)

`22.2-CONTEXT.md` D-12 reads: *"`approach2` (template route) works today and needs no functional
change."* Verified by building it during planning — **this premise is false**. The build succeeds
and emits a shared template file, but that file is the bundled default template, not the sample's
wrapper; the pinned `charged-ieee` package is never imported. Two warnings are logged (a custom
template not found, and a template directory not found) and then execution continues.

The owner ruled on 2026-07-22 to **fold the fix into this phase and pin it with a test**, on the
grounds that shipping v0.6.2 with a release note saying the package path works, while the bundled
sample demonstrating the D-03-promoted route never loads the package, would be self-contradicting.
This plan is that ruling — it is **not** planner scope creep, and a later reader should not read it
as such.

## Verified-during-planning repair (known-achievable, not a guess)

Applying all three fixes below and building with the PDF-producing builder yields: `build
succeeded.` with **zero warnings**, a 68 778-byte PDF, and an emitted shared template containing
three references to the pinned package and none of the bundled default template's marker text.
