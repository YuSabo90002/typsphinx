# Phase 22.2 — Security

> Per-phase security contract: threat register, accepted risks, and audit trail.

---

## Trust Boundaries

| Boundary | Description | Data Crossing |
|----------|-------------|---------------|
| developer `conf.py` → typsphinx config read | `conf.py` is arbitrary Python already executed by Sphinx; fully trusted before typsphinx runs. Removing or adding a registration cannot expand this boundary. | Config values (project metadata, paths, package strings) |
| developer `conf.py` config values → emitted `.typ` source | Config strings (author details, template-function parameters) are interpolated into generated Typst source. The one genuine content-injection boundary in this phase. | Author names/emails/affiliations, template parameter values |
| repository docs & bundled examples → user's own project | Docs and samples are copy-paste sources; a doc naming a nonexistent config or an uncompilable sample propagates a broken configuration into user projects. | Documented config names, sample `conf.py` + template files |
| developer `conf.py` → build routing decision | `typst_package` and `typst_template` reach a routing branch deciding which files are written and referenced. Silent resolution discards stated intent. | Routing config pair |
| configured template path → filesystem resolution | The configured path is joined onto the Sphinx source dir; an unresolvable path warns and silently substitutes the bundled default. | Template file path |
| CI / developer machine → Typst Universe registry | The new real-compile gates resolve and (cold cache) download a version-pinned `@preview` package during compile. | Pinned package archive |
| emitted `.typ` → Typst compiler | The generated document names files and packages that must exist at compile time. | Generated Typst source |
